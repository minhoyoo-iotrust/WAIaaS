# #315 — 병렬 UAT 실행 시 세션 한도 초과로 테스트 진행 불가

- **유형:** BUG
- **심각도:** MEDIUM
- **상태:** OPEN
- **마일스톤:** —

## 현상

Agent UAT를 백그라운드 에이전트로 병렬 실행하면, 각 에이전트가 독립 세션을 생성하여 `max_sessions_per_wallet: 5` 한도에 빠르게 도달한다. 이후 새 세션 생성 시 `SESSION_LIMIT_EXCEEDED` 에러가 발생하고, 기존 세션 토큰도 `INVALID_TOKEN`으로 무효화되어 테스트 진행이 불가해진다.

### 재현 절차

1. UAT admin 카테고리 실행 (세션 1 생성)
2. UAT testnet 카테고리를 백그라운드 에이전트로 실행 (세션 2~3 생성)
3. 메인 에이전트에서 추가 세션 생성 시도 → `SESSION_LIMIT_EXCEEDED`
4. 기존 세션 토큰으로 API 호출 → `INVALID_TOKEN` (무효화됨)

### 에러 메시지

```json
{
  "code": "SESSION_LIMIT_EXCEEDED",
  "message": "Wallet 019c6fb6-2b2d-72a8-8515-235255be884d has reached session limit (max: 5)"
}
```

## 원인 분석

### 근본원인 후보

| 후보 | 가능성 | 설명 |
|------|--------|------|
| **A. UAT 세션 미정리** | **높음** | 백그라운드 에이전트가 시나리오 실행 중 세션을 여러 개 생성하지만, 종료 시 세션을 revoke하지 않아 잔여 세션 누적 |
| **B. 세션 한도 정책 부적합** | 중간 | `max_sessions_per_wallet: 5`가 병렬 테스트 시나리오에 비해 너무 낮음 |
| **C. 세션 공유 메커니즘 부재** | 중간 | 동일 마스터 인증으로 생성된 세션을 재사용하는 수단이 없어, 매번 새 세션 생성 |

### 핵심 문제

1. **세션 라이프사이클 미관리**: UAT 에이전트가 세션 생성만 하고 cleanup을 하지 않음
2. **기존 세션 재사용 불가**: 새 에이전트 프로세스에서 기존 활성 세션의 토큰을 조회하는 API가 없음 (세션 목록은 조회 가능하나 토큰 재발급 없이는 사용 불가)

## 해결 방안

### Phase 1: 관측 가능성 확보

1. **세션 생성/소멸 로깅 강화**: UAT 실행 전후 활성 세션 수를 로깅
2. **세션 한도 도달 시 경고**: 한도의 80% (4/5) 도달 시 warn 로그

### Phase 2: 근본원인 확정

- UAT 프로토콜에서 세션 생성 횟수 추적
- 백그라운드 에이전트의 세션 cleanup 동작 확인
- `INVALID_TOKEN` 발생 시점과 세션 한도 초과 시점의 상관관계 확인

### Phase 3: 원인별 수정

| 확정 원인 | 수정 방안 |
|-----------|----------|
| A. UAT 세션 미정리 | Agent UAT 프로토콜에 Phase 0 (세션 재사용) 및 Phase 7.5 (세션 revoke) 단계 추가 |
| B. 세션 한도 부적합 | UAT 실행 시 임시로 `max_sessions_per_wallet` 상향 후 원복하는 절차 추가 |
| C. 세션 공유 부재 | `GET /v1/sessions` 목록에서 활성 세션을 찾고 `POST /v1/sessions/{id}/reissue`로 토큰 재발급하여 재사용 |

## 테스트 항목

- [ ] 병렬 UAT 에이전트 2개 동시 실행 시 세션 한도 미초과 확인
- [ ] UAT 종료 후 잔여 세션이 정리되는지 확인
- [ ] 기존 활성 세션 재사용 시 `INVALID_TOKEN` 미발생 확인
- [ ] `max_sessions_per_wallet` 변경 후 원복 시 기존 세션 영향 없음 확인
