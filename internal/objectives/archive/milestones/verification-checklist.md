# HUMAN 검증 체크리스트

모든 마일스톤의 `[HUMAN]` 검증 항목을 한 곳에서 추적한다. 자동화 불가능한 UI/UX, 외부 API 실 호출, 문서 품질 등 사람이 직접 확인해야 하는 항목만 포함한다.

---

## 상태 정의

| 상태 | 의미 | 다음 액션 |
|------|------|----------|
| `⬜ PENDING` | 아직 검증 시점이 아님 (마일스톤 미도달) | 해당 마일스톤 구현 후 검증 |
| `🔍 READY` | 구현 완료, 검증 가능한 상태 | 검증 수행 |
| `✅ PASSED` | 검증 완료, 기준 충족 | 완료 |
| `⏭️ DEFERRED` | 건너뜀 (사유 필수 기록) | v2.0 릴리스 전 재검토 |
| `❌ FAILED` | 검증 실패, 수정 필요 | 수정 후 재검증 |

## 검증 기록 절차

### 1. 마일스톤 구현 완료 시

해당 마일스톤의 HUMAN 항목 상태를 `⬜ PENDING` → `🔍 READY`로 변경한다.

```
예: v1.3 구현 완료
v1.3-H01 | ... | 🔍 READY | | 구현 완료, 검증 가능
```

### 2. 검증 수행 시

항목을 직접 확인하고 결과를 기록한다.

**PASSED인 경우:**
```
v1.3-H01 | ... | ✅ PASSED | 2026-03-15 | MCP 6도구 정상 호출 확인
```

**FAILED인 경우:**
```
v1.6-H02 | ... | ❌ FAILED | 2026-04-10 | 에이전트 상태 카드 색상 미구현. Issue #42 생성
```

**DEFERRED인 경우:**
```
v1.5-H01 | ... | ⏭️ DEFERRED | 2026-04-05 | Jupiter Devnet API 일시 중단. v2.0 전 재검증 예정
```

### 3. 마일스톤 완료 시 (/gsd:complete-milestone)

마일스톤 아카이브 전 확인:

1. 해당 마일스톤의 HUMAN 항목 중 `🔍 READY` 또는 `⬜ PENDING`이 남아있지 않은지 확인
2. `❌ FAILED` 항목이 있으면 수정 후 재검증하거나, 사유와 함께 `⏭️ DEFERRED` 전환
3. 요약 테이블 숫자 업데이트
4. 아카이브 노트에 HUMAN 검증 상태 요약 포함

### 4. 건너뛰고 넘어갈 때

반드시 다음 두 가지를 기록한다:

1. **상태**: `⏭️ DEFERRED`
2. **비고**: 건너뛴 사유 + 재검증 예정 시점

```
v1.6-H01 | ... | ⏭️ DEFERRED | 2026-04-20 | 디자인 확정 전이라 UX 판단 유보. v1.7 완료 후 재검증
```

건너뛸 수 없는 항목:
- 보안 관련 (Kill Switch UI 등 v1.6-H05는 기능 정합성이 자동화되므로 UX만 DEFER 가능)
- v2.0-H03 (설계 부채 판단) — 릴리스 직전 필수

### 5. v2.0 릴리스 전 최종 점검

1. 이 파일의 요약 테이블에서 `FAILED` = 0, `PENDING/READY` = 0 확인
2. `DEFERRED` 항목 전수 재검토 — 릴리스에 영향 없는지 판단
3. 릴리스 게이트 체크리스트 전항목 체크
