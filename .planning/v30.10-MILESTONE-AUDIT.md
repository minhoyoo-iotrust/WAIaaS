## 6. Defects Found

### DEFECT-01 (MEDIUM): Admin UI Settings Key Mismatch

**Affected Requirements:** ADM-02, ADM-04
**Severity:** Medium -- Admin UI settings changes for ERC-8128 have no effect on daemon behavior

**Details:**

The registered setting keys in `packages/daemon/src/infrastructure/settings/setting-keys.ts` are:
- `erc8128.default_ttl_sec`
- `erc8128.default_nonce`
- `erc8128.default_algorithm`
- `erc8128.default_rate_limit_rpm`

The Admin UI system page (`packages/admin/src/pages/system.tsx`) writes to:
- `erc8128.default_ttl` (should be `erc8128.default_ttl_sec`)
- `erc8128.include_nonce` (should be `erc8128.default_nonce`)
- `erc8128.algorithm` (should be `erc8128.default_algorithm`)
- `erc8128.rate_limit_per_minute` (should be `erc8128.default_rate_limit_rpm`)

The daemon route reads from the registered keys (e.g., `settingsService.get('erc8128.default_ttl_sec')`). Admin UI writes to different keys, so user settings changes have no effect.

**Fix:** Update Admin UI system.tsx field names to match registered setting keys, OR rename setting keys to match Admin UI names.

### DEFECT-02 (LOW): MCP/SDK Verify Pass Signature as Top-Level Body Fields

**Affected Requirements:** API-04, API-06
**Severity:** Low -- MCP/SDK verify still works if user includes Signature-Input/Signature in headers dict

**Details:**

The MCP `erc8128_verify_signature` tool takes `signature_input` and `signature` as separate args, then sends them as top-level body fields (`signatureInput`, `signature`) to `POST /v1/erc8128/verify`. The REST API Zod schema at `erc8128.ts:124-129` only accepts `{ url, method, headers, body }`, so these top-level fields are silently stripped.

The core verifier (`verifier.ts:39-51`) extracts Signature-Input and Signature from the `headers` dict via case-insensitive lookup. If the user also includes them in the `headers` arg, verification works. But the separate params are misleading and redundant.

Same issue in SDK `verifyHttpSignature()` which sends `signatureInput`, `signature`, `contentDigest` as top-level params.

**Impact:** The verify flow works ONLY if the user includes Signature-Input and Signature in the `headers` record. The dedicated params are silently discarded.

**Fix:** Either (a) merge the separate params into the headers dict in MCP/SDK before sending, or (b) update the REST API verify schema to accept these as top-level fields and merge them into headers before calling verifyHttpSignature.
