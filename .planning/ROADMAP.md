## Phase Details

### Phase 317: Foundation
**Goal**: ERC-8004 통합의 모든 후속 작업이 의존하는 DB 스키마, 코어 타입, 설정 키, 알림 이벤트가 준비된 상태
**Depends on**: Nothing (first phase of v30.8)
**Requirements**: PKG-03, DB-01, DB-02, DB-03, DB-04, NOTIF-01
**Success Criteria** (what must be TRUE):
  1. DB v39 마이그레이션이 실행되면 agent_identities, reputation_cache 테이블이 생성되고, pending_approvals에 approval_type 컬럼이 추가되며, policies CHECK에 REPUTATION_THRESHOLD가 포함된다
  2. Admin Settings에서 `actions.erc8004_*` 9개 키를 조회/수정할 수 있고, feature gate(erc8004_agent_enabled)가 기본 false로 동작한다
  3. @waiaas/core에 5개 NotificationEventType(AGENT_REGISTERED 등)이 추가되어 EventBus에서 emit 가능하다
  4. PolicyType enum에 REPUTATION_THRESHOLD가 18번째 타입으로 추가된다
**Plans**: 2 plans

Plans:
- [x] 317-01: DB v39 마이그레이션 (agent_identities + reputation_cache + approval_type + policies 재생성) -- completed 2026-03-04
- [x] 317-02: 코어 Enum 확장 + Admin Settings 9키 + 알림 이벤트 5종 -- completed 2026-03-04

### Phase 318: ActionProvider + Registry Client
**Goal**: AI 에이전트가 Erc8004ActionProvider를 통해 온체인 Identity/Reputation/Validation 레지스트리에 쓰기 작업(등록, URI 설정, 메타데이터, 피드백, 검증 요청)을 수행할 수 있는 상태
**Depends on**: Phase 317
**Requirements**: PKG-01, PKG-02, IDEN-01, IDEN-04, IDEN-05, VALD-01
**Success Criteria** (what must be TRUE):
  1. Erc8004ActionProvider가 registerBuiltInProviders에 등록되고, actions.erc8004_agent_enabled 설정으로 활성화/비활성화된다
  2. register_agent 액션이 APPROVAL 티어로 Identity Registry에 NFT 민팅 트랜잭션을 생성한다
  3. set_agent_uri, set_metadata 액션이 각각 DELAY, NOTIFY 티어로 올바른 ContractCallRequest를 생성한다
  4. request_validation 액션이 feature gate(validation_registry_address 비어 있으면 비활성)로 보호되며, DELAY 티어로 동작한다
  5. 8개 write 액션 모두 resolve()에서 올바른 ABI 인코딩과 레지스트리 주소를 사용한다
**Plans**: 2 plans

Plans:
- [x] 318-01: Erc8004RegistryClient (viem 래퍼, ABI const, 멀티체인 주소 상수) -- completed 2026-03-04
- [x] 318-02: Erc8004ActionProvider (8 write 액션, IActionProvider 구현, registerBuiltInProviders 등록) -- completed 2026-03-04

### Phase 319: Read-Only Routes + Registration File
**Goal**: 에이전트 정보, 평판, 검증 상태를 REST API로 조회할 수 있고, 등록 파일이 자동 생성되어 공개 엔드포인트로 제공되며, connect-info에 ERC-8004 상태가 포함되는 상태
**Depends on**: Phase 318
**Requirements**: IDEN-06, IDEN-07, IDEN-08, REPU-03, VALD-02, API-01
**Success Criteria** (what must be TRUE):
  1. GET /v1/erc8004/agent/:agentId가 에이전트의 ID, wallet, URI, metadata를 반환한다
  2. GET /v1/erc8004/agent/:agentId/reputation이 평판 summary(count, score, decimals)를 tag 필터와 함께 반환한다
  3. GET /v1/erc8004/registration-file/:walletId가 WAIaaS MCP/REST 엔드포인트가 포함된 등록 파일 JSON을 자동 생성하여 반환한다
  4. GET /v1/connect-info 응답에 erc8004 필드(agentId, registryAddress, chainId, status 등)가 등록된 지갑에 대해 포함된다
  5. GET /v1/erc8004/validation/:requestHash가 검증 상태를 반환한다
**Plans**: 2 plans

Plans:
- [ ] 319-01-PLAN.md -- ERC-8004 read-only REST API 4 GET 엔드포인트 + 등록 파일 자동 생성 + 테스트
- [ ] 319-02-PLAN.md -- connect-info erc8004 확장 + 테스트

### Phase 320: Reputation Policy Engine + Cache
**Goal**: REPUTATION_THRESHOLD 정책이 Stage 3에서 상대방 에이전트의 온체인 평판을 평가하여 보안 티어를 자동 조정하고, 평판 데이터가 인메모리 캐시 + DB 폴백으로 효율적으로 제공되는 상태
**Depends on**: Phase 319
**Requirements**: REPU-01, REPU-02, REPU-04, REPU-05, REPU-06, REPU-07
**Success Criteria** (what must be TRUE):
  1. REPUTATION_THRESHOLD 정책이 Stage 3 파이프라인의 position 6(APPROVED_SPENDERS 다음, SPENDING_LIMIT 이전)에서 평가된다
  2. 상대방 평판이 min_score 미만이면 below_threshold_tier(기본 APPROVAL)로 에스컬레이션되고, 기존 티어보다 높은 경우에만 적용된다(maxTier 로직)
  3. 평판 데이터가 없거나 RPC 실패 시 unrated_tier(기본 APPROVAL)가 적용된다
  4. ReputationCacheService가 인메모리 캐시(TTL 300s) + DB 폴백 + RPC 3s 타임아웃으로 동작한다
  5. give_feedback, revoke_feedback 액션 실행 후 관련 캐시가 무효화된다
**Plans**: 2 plans

Plans:
- [ ] 320-01: ReputationCacheService (인메모리 Map + DB reputation_cache + RPC 폴백)
- [ ] 320-02: REPUTATION_THRESHOLD 정책 평가기 (DatabasePolicyEngine Stage 3, maxTier 로직)

### Phase 321: EIP-712 Approval + Wallet Linking
**Goal**: Owner가 EIP-712 typed data 서명으로 agentWallet을 링크/언링크할 수 있고, ApprovalWorkflow가 SIWE와 EIP-712 두 가지 승인 타입을 라우팅하는 상태
**Depends on**: Phase 320
**Requirements**: IDEN-02, IDEN-03
**Success Criteria** (what must be TRUE):
  1. set_agent_wallet 액션이 AgentWalletSet 4-field EIP-712 typehash(domain: ERC8004IdentityRegistry v1, owner 포함)로 서명 요청을 생성한다
  2. WalletConnect eth_signTypedData_v4를 통해 Owner가 EIP-712 서명을 제공할 수 있다
  3. unset_agent_wallet 액션이 APPROVAL 티어로 동작하며 agentWallet 연결을 해제한다
  4. EIP-712 승인이 필요한 경우 ApprovalChannelRouter가 WC/Admin UI 채널로만 라우팅하고, ntfy/Telegram에는 Admin UI 링크가 포함된 알림만 전송한다
**Plans**: 2 plans

Plans:
- [ ] 321-01: EIP-712 typed data 승인 흐름 + ApprovalWorkflow approval_type 라우팅
- [ ] 321-02: set_agent_wallet + unset_agent_wallet 액션 완성

  2. Admin UI에서 EVM 지갑 선택 + 이름/설명 입력으로 에이전트 등록을 시작하고, 등록 파일 JSON 트리 뷰어 + URL 복사가 가능하다
