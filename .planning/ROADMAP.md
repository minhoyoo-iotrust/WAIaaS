## Phase Details

### Phase 338: Foundation
**Goal**: Smart Account가 프로바이더 없이도 생성/관리되며, UserOp API의 Zod 스키마/에러 코드/DB 테이블이 준비된 상태
**Depends on**: Nothing (first phase of v31.2)
**Requirements**: SCHM-01, SCHM-02, SCHM-03, SCHM-04, SCHM-05, ERR-01, ERR-02, ERR-03, ERR-04, ERR-05, DB-01, DB-02, PROV-01, PROV-02, PROV-03, PROV-04, PROV-05
**Success Criteria** (what must be TRUE):
  1. Smart Account 지갑을 프로바이더 없이 생성할 수 있고, Lite 모드로 인식된다
  2. Lite 모드 지갑에서 POST /transactions/send 호출 시 CHAIN_ERROR와 userop API 안내를 받는다
  3. PUT /wallets/:id/provider로 프로바이더 추가 시 Full 모드로 전환되어 기존 AA 기능이 동작한다
  4. UserOp v0.7 Zod 스키마(Build Request/Response, Sign Request/Response, UserOperation)가 정의되어 있고 TypeScript 타입이 추론된다
  5. DB v45 마이그레이션으로 userop_builds 테이블이 생성되며, 스키마 스냅샷 테스트가 통과한다
**Plans**: 2 plans

Plans:
- [x] 338-01: Provider Lite/Full 모드 분기 + 지갑 생성 검증 제거
- [x] 338-02: Zod 스키마 + 에러 코드 + DB v45 마이그레이션

### Phase 339: UserOp Build API
**Goal**: 에이전트/플랫폼이 Smart Account 지갑의 트랜잭션 요청을 unsigned UserOp으로 변환하여 받을 수 있는 상태
**Depends on**: Phase 338
**Requirements**: BUILD-01, BUILD-02, BUILD-03, BUILD-04, BUILD-05, BUILD-06, BUILD-07, BUILD-08, BUILD-09, BUILD-10, BUILD-11, DATA-01, DATA-02, DATA-03, DATA-04
**Success Criteria** (what must be TRUE):
  1. POST /v1/wallets/:id/userop/build에 TransactionRequest를 보내면 sender, nonce, callData, entryPoint, buildId가 포함된 unsigned UserOp을 받는다
  2. 미배포 Smart Account의 경우 factory/factoryData가 응답에 자동 포함되고, deployed 상태가 DB에 갱신된다
  3. gas/paymaster 필드가 응답에 포함되지 않는다 (플랫폼이 채워야 할 영역)
  4. Solana 지갑에서 호출 시 ACTION_VALIDATION_FAILED 에러가 반환된다
  5. build 데이터가 TTL(10분)로 관리되며, 만료 레코드는 WorkerScheduler가 주기적으로 정리한다
**Plans**: 2 plans

Plans:
- [x] 339-01: UserOp Build endpoint + SmartAccount callData encoding
- [x] 339-02: Build data cleanup worker (TTL enforcement)

### Phase 340: UserOp Sign API
**Goal**: 플랫폼이 가스/페이마스터 필드를 채운 UserOp을 WAIaaS에 제출하면 서명된 UserOp을 받을 수 있으며, 모든 활동이 감사 로그와 알림으로 추적되는 상태
**Depends on**: Phase 339
**Requirements**: SIGN-01, SIGN-02, SIGN-03, SIGN-04, SIGN-05, SIGN-06, SIGN-07, SIGN-08, SIGN-09, SIGN-10, CONN-01, CONN-02, NTFY-01, NTFY-02, NTFY-03, NTFY-04
**Success Criteria** (what must be TRUE):
  1. POST /v1/wallets/:id/userop/sign에 buildId + 완성된 UserOp을 보내면 signature가 추가된 signedUserOperation과 txId를 받는다
  2. callData가 build 시점과 다르면 CALLDATA_MISMATCH, sender가 지갑 주소와 다르면 SENDER_MISMATCH 에러가 반환된다
  3. 만료/미존재/이미 사용된 buildId에 대해 각각 EXPIRED_BUILD/BUILD_NOT_FOUND/BUILD_ALREADY_USED 에러가 반환된다
  4. GET /v1/connect-info 응답에 Smart Account 지갑 존재 시 userop capability가 포함된다
  5. build/sign 시 USEROP_BUILD/USEROP_SIGNED 감사 로그가 기록되고, TX_REQUESTED/TX_SUBMITTED 알림이 발생한다
**Plans**: 2 plans

Plans:
- [ ] 340-01-PLAN.md -- UserOp Sign endpoint + callData verification + policy + audit + notifications
- [ ] 340-02-PLAN.md -- connect-info userop capability + prompt guidance

### Phase 341: Interface Integration
**Goal**: Admin UI에서 Lite/Full 모드를 시각적으로 확인하고, MCP/SDK/스킬 파일을 통해 에이전트가 UserOp API를 사용할 수 있는 상태
**Depends on**: Phase 340
**Requirements**: ADMIN-01, ADMIN-02, ADMIN-03, ADMIN-04, INTF-01, INTF-02, INTF-03, SKILL-01, SKILL-02, SKILL-03
**Success Criteria** (what must be TRUE):
  1. Admin UI 지갑 생성 폼에서 None (Lite mode) 옵션을 선택할 수 있고, API Key/URL 필드가 숨겨진다
  2. 지갑 상세/목록에서 Lite/Full 모드가 배지로 구분되어 표시된다
  3. MCP build_userop/sign_userop 도구로 UserOp 구성/서명이 가능하다
  4. SDK buildUserOp()/signUserOp() 메서드가 동작한다
  5. transactions/wallet/admin 스킬 파일에 UserOp API와 Lite/Full 모드가 문서화되어 있다
