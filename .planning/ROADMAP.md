## Phase Details

### Phase 380: ResolvedAction 타입 시스템
**Goal**: ActionProvider가 on-chain 트랜잭션뿐 아니라 off-chain 서명 액션도 표현할 수 있는 타입 기반이 완성된다
**Depends on**: Nothing (first phase)
**Requirements**: TYPE-01, TYPE-02, TYPE-03, TYPE-04, TYPE-05, TYPE-06, SIGN-01, SIGN-02
**Success Criteria** (what must be TRUE):
  1. ResolvedAction Zod union이 contractCall/signedData/signedHttp 3종을 kind 필드로 분기하는 설계가 존재한다
  2. SignedDataAction 스키마가 signingScheme, payload, venue, operation, credentialRef 등 필수/선택 필드를 명세한다
  3. SignedHttpAction 스키마가 기존 SignHttpMessageParams를 통합하면서 venue/operation을 추가한다
  4. 기존 13개 ActionProvider가 kind 없이 반환해도 registry가 contractCall로 정규화하는 전략이 문서화된다
  5. ISignerCapability 인터페이스와 SigningSchemeEnum(7종)이 정의되어 서명 능력의 통합 추상화 기반이 존재한다
**Plans:** 2/2 plans complete
Plans:
- [ ] 380-01-PLAN.md — ResolvedAction 3종 Zod 스키마 초안 + 정규화 전략 + 하위 호환 분석
- [ ] 380-02-PLAN.md — ISignerCapability 인터페이스 + SigningSchemeEnum(7종) 설계

### Phase 381: CredentialVault 인프라
**Goal**: AI 에이전트가 per-wallet CEX API 키 등 자격증명을 안전하게 저장/조회/관리할 수 있는 설계가 완성된다
**Depends on**: Phase 380
**Requirements**: CRED-01, CRED-02, CRED-03, CRED-04, CRED-05, CRED-06, CRED-07, CRED-08
**Success Criteria** (what must be TRUE):
  1. ICredentialVault CRUD 인터페이스(create/get/list/delete/rotate)와 5종 credential 타입이 설계된다
  2. per-wallet -> 글로벌 fallback 스코프 모델과 credentialRef 간접 참조(UUID/{walletId}:{name})가 명세된다
  3. wallet_credentials DB 스키마(AES-256-GCM 암호화, HKDF 도메인 분리)가 설계되고 re-encrypt/backup 통합 경로가 문서화된다
  4. credential lifecycle(생성/로테이션/만료/삭제)과 인증 모델(sessionAuth+masterAuth)이 설계된다
  5. Admin UI Credentials 탭 UX(목록/등록/삭제/로테이션, 원문 비노출)가 설계된다
**Plans:** 2/2 plans complete
Plans:
- [x] 381-01-PLAN.md — ICredentialVault 인터페이스 + DB 스키마 + 암호화 + 스코프 + 라이프사이클 + 인증 모델 설계
- [x] 381-02-PLAN.md — Admin UI Credentials 탭 UX + MCP 도구 + SDK 메서드 설계

### Phase 382: Signer Capabilities
**Goal**: 기존 4종 서명 모듈을 ISignerCapability 어댑터로 래핑하고 HMAC/RSA-PSS/signBytes 신규 능력을 추가하는 설계가 완성된다
**Depends on**: Phase 380
**Requirements**: SIGN-03, SIGN-04, SIGN-05, SIGN-06
**Success Criteria** (what must be TRUE):
  1. 기존 4종 signer(Eip712, PersonalSign, Erc8128, Transaction)의 ISignerCapability 어댑터 위임 설계가 존재한다
  2. HmacSignerCapability + RsaPssSignerCapability의 node:crypto 기반 설계가 존재한다
  3. signBytes() capability(Ed25519/ECDSA arbitrary bytes)의 설계가 존재한다
  4. SignerCapabilityRegistry(signingScheme -> ISignerCapability 자동 매핑)의 설계가 존재한다
**Plans:** 2/2 plans complete
Plans:
- [x] 382-01-PLAN.md — 기존 4종 어댑터 + signBytes 2종 상세 설계
- [x] 382-02-PLAN.md — HMAC/RSA-PSS capability + SignerCapabilityRegistry 설계

### Phase 383: 파이프라인 라우팅
**Goal**: ResolvedAction kind에 따라 contractCall/signedData/signedHttp 파이프라인으로 분기하고 off-chain 액션을 DB에 기록하는 설계가 완성된다
**Depends on**: Phase 380, Phase 381, Phase 382
**Requirements**: PIPE-01, PIPE-02, PIPE-03, PIPE-04, PIPE-05
**Success Criteria** (what must be TRUE):
  1. ActionProviderRegistry resolve() 결과의 kind별 3-way 파이프라인 라우팅 설계가 존재한다
  2. off-chain action DB 기록 설계(transactions 테이블 action_kind 컬럼 + nullable txHash)가 존재한다
  3. 정책 평가 시점(resolve 후 서명 전)이 signedData/signedHttp 경로에서도 보장되는 설계가 존재한다
  4. REST API/MCP 도구/SDK 메서드의 off-chain action 처리 + credential 관리 인터페이스 설계가 존재한다
**Plans**: TBD

### Phase 384: 정책 + 추적 확장
**Goal**: off-chain 액션에 대한 venue 기반 정책 평가와 비동기 상태 추적이 설계된다
**Depends on**: Phase 383
**Requirements**: PLCY-01, PLCY-02, PLCY-03, PLCY-04, TRCK-01, TRCK-02, TRCK-03, TRCK-04
**Success Criteria** (what must be TRUE):
  1. TransactionParam/ActionPolicyParam이 venue/actionCategory/notionalUsd/leverage 등으로 확장되는 설계가 존재한다
  2. VENUE_WHITELIST 정책(CONTRACT_WHITELIST 패턴 활용)과 ACTION_CATEGORY_LIMIT 정책(SPENDING_LIMIT 패턴 확장)이 설계된다
  3. AsyncTrackingResult.state가 PARTIALLY_FILLED/FILLED/CANCELED/SETTLED/EXPIRED로 확장되는 설계가 존재한다
  4. 상태 저장 위치(transactions 테이블 external_action_status vs 별도 테이블)가 결정되고 AsyncPollingService 쿼리 확장이 설계된다
