# Plan 306-02: CLI 커맨드 + config.toml 키 + 설계 문서 갱신

## 목표

`waiaas backup` / `waiaas restore` CLI 인터페이스를 안전 장치와 함께 설계하고, config.toml `[backup]` 섹션 키를 평탄 키 원칙으로 정의하며, 영향받는 설계 문서(24, 26, 28, 54)의 갱신 지점을 명시한다.

## 컨텍스트

### CLI 구조 (packages/cli/src/index.ts)

현재 CLI는 `commander` 패키지 기반이며 다음 커맨드를 제공한다:

```
waiaas init|start|stop|status|quickset|quickstart|update|set-master
waiaas wallet create|info
waiaas session prompt
waiaas owner connect|disconnect|status
waiaas mcp setup
waiaas notification setup
```

모든 커맨드는 `--data-dir <path>` 글로벌 옵션을 지원하며, 데이터 디렉토리 해석은 `resolveDataDir()`을 사용한다.

### config.toml 현재 구조

12개 섹션 (daemon, keystore, database, rpc, notifications, security, walletconnect, x402, display, telegram, incoming, actions), `KNOWN_SECTIONS` 상수로 관리. 평탄 키만 허용 (중첩 금지, `detectNestedSections()`).

환경변수 오버라이드: `WAIAAS_{SECTION}_{KEY}` 패턴 (`applyEnvOverrides()`).

## 1. CLI 커맨드 설계 (BKUP-03)

### 1.1 커맨드 개요

| 커맨드 | 설명 | 데몬 상태 | 인증 |
|--------|------|----------|------|
| `waiaas backup` | 암호화 백업 생성 | 실행 중이어야 함 (VACUUM INTO 사용) | 마스터 비밀번호 |
| `waiaas restore` | 암호화 백업에서 복원 | 정지 상태여야 함 | 마스터 비밀번호 |
| `waiaas backup list` | 백업 목록 조회 | 무관 | 불필요 |
| `waiaas backup inspect` | 백업 상세 조회 | 무관 | 불필요 |

### 1.2 waiaas backup

#### 명령어

```bash
waiaas backup [options]
```

#### 옵션

| 옵션 | 타입 | 기본값 | 설명 |
|------|------|--------|------|
| `--data-dir <path>` | string | `~/.waiaas` | 데이터 디렉토리 |
| `--output <path>` | string | config.toml `backup_dir` 또는 `{dataDir}/backups/` | 출력 디렉토리 |
| `--password <pw>` | string | (프롬프트) | 마스터 비밀번호 |

#### 플로우

```
waiaas backup
    |
    v
[1] 데몬 상태 확인
    -- PID 파일 또는 HTTP 헬스 체크로 확인
    -- 실행 중이 아니면:
       "Error: Daemon is not running. Start the daemon first: waiaas start"
       "Backup requires a running daemon for consistent database snapshots."
       exit(1)
    |
    v
[2] 마스터 비밀번호 획득
    -- --password 옵션 → 사용
    -- WAIAAS_MASTER_PASSWORD 환경변수 → 사용
    -- 없으면 → stdin 프롬프트 (마스킹)
    |
    v
[3] 마스터 비밀번호 검증
    -- daemon API에 인증 요청 (GET /health with masterAuth)
    -- 실패 시 "Error: Invalid master password" exit(1)
    |
    v
[4] 백업 생성 (EncryptedBackupService.createBackup 호출)
    -- daemon 측 REST API 엔드포인트를 통해 실행
    -- 또는 CLI가 직접 DB 접속 + VACUUM INTO 실행
    |
    v
[5] 결과 표시
    -- "Backup created: /path/to/waiaas-backup-20260303-143022.waiaas-backup"
    -- "Size: 10.5 MB | Files: 4 (DB + config + 2 keystores)"
    -- "Backups retained: 5/7"
```

**설계 결정 -- 데몬 API vs CLI 직접 실행:**

CLI가 데몬에 REST API를 호출하여 백업을 수행하는 방식을 채택한다.

| 방식 | 장점 | 단점 |
|------|------|------|
| **REST API** (채택) | VACUUM INTO가 DB 커넥션 필요, 데몬이 이미 보유 | 엔드포인트 추가 필요 |
| CLI 직접 DB 접속 | 단순 | SQLite 단일 연결 제한 충돌 가능, flock 경합 |

REST API 엔드포인트:
```
POST /v1/admin/backup
Authorization: X-Master-Password: <hash>

Response 200:
{
  "path": "/home/user/.waiaas/backups/waiaas-backup-20260303-143022.waiaas-backup",
  "size": 10500000,
  "created_at": "2026-03-03T14:30:22.000Z",
  "files": 4
}
```

## 3. 자동 백업 스케줄러 설계

### 3.1 개요

config.toml의 `backup.interval > 0`이면 데몬이 주기적으로 암호화 백업을 자동 생성한다.

### 3.2 통합 위치

BackgroundWorkers (packages/daemon/src/lifecycle/workers.ts)에 `BackupWorker`를 추가한다.

```
시작 순서 (doc 28):
  Config -> DB -> KeyStore -> Adapters -> Server -> Workers -> PID/Ready
                                                     ^
                                              BackupWorker 등록
종료 순서 (doc 28):
  Signal -> Server Stop -> ... -> Workers Stop -> ...
                                    ^
                              BackupWorker 정리
```

### 3.3 BackupWorker 설계

```typescript
class BackupWorker {
  private timer: NodeJS.Timeout | null = null;

  constructor(
    private readonly backupService: EncryptedBackupService,
    private readonly masterPassword: string,  // 데몬 시작 시 메모리에 보유
    private readonly intervalSec: number,
    private readonly logger: Logger,
  ) {}

  start(): void {
    if (this.intervalSec <= 0) return;  // 비활성

    // 첫 실행은 interval 후 (시작 직후 즉시 실행하지 않음)
    this.timer = setInterval(() => {
      this.runBackup().catch((err) => {
        this.logger.error('Auto-backup failed', { error: err.message });
      });
    }, this.intervalSec * 1000);

    this.logger.info(`Auto-backup scheduled every ${this.intervalSec}s`);
  }

  stop(): void {
    if (this.timer) {
      clearInterval(this.timer);
      this.timer = null;
    }
  }

  private async runBackup(): Promise<void> {
    const path = await this.backupService.createBackup(this.masterPassword);
    this.logger.info('Auto-backup created', { path });
  }
}
```

### 3.4 마스터 비밀번호 접근

자동 백업은 마스터 비밀번호가 필요하다. 데몬 시작 시 마스터 비밀번호는 이미 Argon2id 검증을 통과하여 메모리에 존재한다 (DaemonLifecycle Step 3: Keystore unlock).

- DaemonLifecycle이 마스터 비밀번호를 BackupWorker에 전달
- JavaScript 문자열은 불변이므로 제로화 불가 (기존 키스토어 unlock과 동일 한계)
- 데몬 종료 시 프로세스 메모리 전체가 해제됨

### 3.5 실패 처리

| 시나리오 | 처리 |
|---------|------|
| VACUUM INTO 실패 | 로그 error, 다음 주기에 재시도 |
| 디스크 공간 부족 | 에러 로그 + 이전 백업은 보존 |
| 파일 권한 에러 | 에러 로그 |
| 연속 실패 | 로그만 기록 (AutoStop 트리거 안함 -- 백업 실패는 데몬 기능 장애 아님) |

### 5.2 doc 26 (키스토어)

**변경 위치:** 신규 섹션 또는 기존 섹션 말미

**추가 내용:**

백업 시 키스토어 파일 포함 범위:
- `{dataDir}/keystore/*.json` 전체 -- 월렛별 키스토어 파일 모두 포함

## 7. 설계 결정 로그

| # | 결정 | 근거 | 대안 |
|---|------|------|------|
| D-01 | backup은 데몬 API, restore는 CLI 직접 | backup: VACUUM INTO가 DB 커넥션 필요 / restore: 데몬 미실행 상태 | 양쪽 모두 CLI 직접 -- DB 커넥션 경합 |
| D-02 | `backup` 서브커맨드 그룹 | list, inspect, create를 논리적으로 그룹화 | 각각 독립 커맨드 -- 관리 어려움 |
| D-03 | `waiaas backup` = `backup create` 기본 | 가장 빈번한 작업을 최소 타이핑으로 | 기본 커맨드 없음 -- 항상 `backup create` 입력 |
| D-04 | `restore` 최상위 커맨드 | 복원은 backup 서브그룹보다 독립적 의미가 강함 | `backup restore` -- "backup의 하위" 인식 |
| D-05 | `[backup]` 섹션 3개 평탄 키 | 최소 키로 필요 충분, 복잡도 최소 | 더 많은 키 -- YAGNI 위반 |
| D-06 | interval 기본 0 (비활성) | 개인 데몬에서 자동 백업은 선택적, 수동 백업이 기본 | 기본 활성화 -- 디스크 사용량 증가 |
| D-07 | retention_count 기본 7 | 일주일분 일일 백업 보존 (자동 백업 활성화 시) | 5 (기존 BackupService) -- 암호화 백업은 더 오래 보존 |

## 성공 기준

1. `waiaas backup` / `waiaas restore` CLI 인터페이스가 안전 장치(확인 프롬프트, 데몬 정지 확인, 기존 데이터 보존)와 함께 설계된다 (BKUP-03)
2. config.toml `[backup]` 섹션 키(dir, interval, retention_count)가 평탄 키 원칙을 준수하여 정의된다 (BKUP-04)
