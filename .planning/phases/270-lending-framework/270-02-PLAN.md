**Section 16: REST API 명세**

16.1 GET /v1/wallets/:id/health-factor 엔드포인트:
- Auth: sessionAuth (JWT, wallet must be in session's wallet list)
- Path params: id (wallet UUID)
- Query params: provider? (string, filter by specific provider name e.g., 'aave_v3')
- Success response (200): HealthFactorResponseSchema
- Error responses:
  - 404: wallet not found
  - 403: wallet not in session
  - 200 with status='NO_POSITIONS': wallet has no lending positions

16.2 HealthFactorResponseSchema (Zod):
- walletId (string)
- aggregated: object with:
  - healthFactor (number | null, null when no debt) — worst health factor across all providers
  - totalCollateralUsd (number)
  - totalDebtUsd (number)
  - status (enum: SAFE/WARNING/DANGER/CRITICAL/NO_POSITIONS)
- providers: array of per-provider objects:
  - provider (string), healthFactor (number), totalCollateralUsd (number), totalDebtUsd (number)
  - currentLtv (number), maxLtv (number)
  - positions: array of { asset (CAIP-19), positionType (SUPPLY/BORROW), amount (string), amountUsd (number), apy (number|null) }
- lastSyncedAt (number, unix timestamp seconds)
- OpenAPI: .openapi('HealthFactorResponse')

16.3 Phase 268 positions API와의 관계:
- The existing GET /v1/wallets/:id/positions (section 7) returns ALL position categories
- For lending-specific queries: use `?category=LENDING` query parameter (already designed in Phase 268)
- The health-factor endpoint is complementary: /positions gives raw positions, /health-factor gives computed risk metrics
- No duplication — /health-factor reads from defi_positions table (same source as /positions) but computes aggregated health factor
- Document the recommendation for AI agents: call /health-factor first to check safety, then /positions?category=LENDING for detail

16.4 OpenAPIHono createRoute 정의:
- Route path: /v1/wallets/{id}/health-factor
- Method: GET
- Tag: 'DeFi Positions'
- Request: { params: WalletIdParamSchema, query: HealthFactorQuerySchema }
- Responses: { 200: HealthFactorResponseSchema, 404: ErrorSchema, 403: ErrorSchema }
- File path: `packages/daemon/src/api/routes/defi-positions.ts` (same file as positions route from Phase 268)

16.5 설계 결정 기록:
- DEC-LEND-12: Single aggregated /health-factor endpoint (not per-provider) — simpler for AI agents (one call), per-provider data included in response body
- DEC-LEND-13: healthFactor field is null when totalDebtUsd = 0 — health factor is undefined with no debt (not infinity)
- DEC-LEND-14: lastSyncedAt from defi_positions.updated_at — shows data staleness; AI agent can request PositionTracker sync if too old

**Section 17: 프로토콜 인터페이스 매핑**

17.1 Aave V3 매핑 (EVM, IPool ABI → ILendingProvider):

Mapping table:
| ILendingProvider method | Aave V3 ABI function | Notes |
|---|---|---|
| resolve('supply', ...) | IPool.supply(asset, amount, onBehalfOf, referralCode=0) | Requires prior ERC-20 approve of Pool contract. resolve() returns [approveReq, supplyReq] when allowance insufficient |
| resolve('borrow', ...) | IPool.borrow(asset, amount, interestRateMode=2, referralCode=0, onBehalfOf) | interestRateMode: 2=variable (default), 1=stable (deprecated on most markets) |
| resolve('repay', ...) | IPool.repay(asset, amount, interestRateMode, onBehalfOf) | amount=type(uint256).max for full repay. Requires prior ERC-20 approve |
| resolve('withdraw', ...) | IPool.withdraw(asset, amount, to) | amount=type(uint256).max for full withdraw |
| getPosition(walletId) | Pool + aToken/debtToken balanceOf queries | Per-asset: aToken balance = supply position, variableDebtToken balance = borrow position |
| getHealthFactor(walletId) | IPool.getUserAccountData(user) | Returns 6-tuple: totalCollateralBase, totalDebtBase, availableBorrowsBase, currentLiquidationThreshold, ltv, healthFactor. Values in base currency (ETH, 8 decimals) |
| getMarkets(chain) | Pool.getReservesList() + getReserveData(asset) | Reserve data includes ltv, liquidationThreshold, apy (from ray-based rates) |

Contract addresses: Aave V3 Pool on Ethereum mainnet = 0x87870Bca3F3fD6335C3F4ce8392D69350B4fA4E2. Multi-chain: Pool address varies per network (config-based lookup table).

Position model: Global account — single getUserAccountData() returns aggregate health factor. Per-asset positions derived from aToken/debtToken balances.

Implementation milestone: m29-02

17.2 Kamino 매핑 (Solana, klend-sdk → ILendingProvider):

Mapping table:
| ILendingProvider method | Kamino SDK method | Notes |
|---|---|---|
| resolve('supply', ...) | KaminoAction.buildDepositTxns(market, amount, mint, wallet, obligation?) | Returns { setupIxs, lendingIxs, cleanupIxs }. Convert to SolanaTransactionRequest |
| resolve('borrow', ...) | KaminoAction.buildBorrowTxns(market, amount, mint, wallet, obligation) | Requires existing obligation (from prior supply) |
| resolve('repay', ...) | KaminoAction.buildRepayTxns(market, amount, mint, wallet, obligation) | amount can be MAX_U64 for full repay |
| resolve('withdraw', ...) | KaminoAction.buildWithdrawTxns(market, amount, mint, wallet, obligation) | amount can be MAX_U64 for full withdraw |
| getPosition(walletId) | market.getObligationByWallet(publicKey) | obligation.deposits[] + obligation.borrows[] → expand each entry to LendingPositionSummary |
| getHealthFactor(walletId) | obligation.stats (borrowLimit, userTotalDeposit, userTotalBorrow) | healthFactor = borrowLimit / userTotalBorrow (Infinity if no borrow) |
| getMarkets(chain) | market.loadReserves() + iterate reserves | Each reserve has ltv, liquidationThreshold, supply/borrow APY |

Position model: Obligation-based — a single obligation PDA per user per market can hold multiple deposits and borrows. Expand one obligation into N position rows (one per asset per type SUPPLY/BORROW).

SDK compatibility flag: klend-sdk may depend on @solana/web3.js 1.x (legacy). If so, Kamino provider needs an adapter layer to convert between web3.js PublicKey and @solana/kit 6.x Address types. Flag for implementation milestone m29-04.

Market address: Main market = `7u3HeHxYDLhnCoErrtycNokbQYbWGzLs6JSDqGAv5PfF` (JLEND program). Configurable per deployment.

Implementation milestone: m29-04

17.3 Morpho Blue 매핑 (EVM, IMorpho ABI → ILendingProvider):

Mapping table:
| ILendingProvider method | Morpho ABI function | Notes |
|---|---|---|
| resolve('supply', ...) | IMorpho.supplyCollateral(marketParams, assets, onBehalf, data='0x') | Requires ERC-20 approve of Morpho contract. MarketParams struct identifies the market |
| resolve('borrow', ...) | IMorpho.borrow(marketParams, assets, shares=0, onBehalf, receiver) | assets (amount) or shares (share-based), use assets for human-readable amounts |
| resolve('repay', ...) | IMorpho.repay(marketParams, assets, shares=0, onBehalf, data='0x') | assets or shares — use assets for precise repay, shares=type(uint256).max for full repay |
| resolve('withdraw', ...) | IMorpho.withdrawCollateral(marketParams, assets, onBehalf, receiver) | No partial withdraw by shares; specify exact asset amount |
| getPosition(walletId) | IMorpho.position(marketId, user) | Returns (supplyShares, borrowShares, collateral). Per-market — must query each known market |
| getHealthFactor(walletId) | Computed: (collateral * oraclePrice / 1e36 * LLTV / 1e18) / borrowedAmount | Per-market computation. Aggregate: worst health factor across all markets with positions |
| getMarkets(chain) | Config-based curated list (no on-chain registry) | Each market identified by keccak256(abi.encode(MarketParams)). Config stores known marketId → MarketParams mapping |

Position model: Per-market isolated — NO global getUserAccountData(). Each market has independent collateral/debt/LTV. Aggregated health factor = min(per-market health factors).

MarketParams struct: { loanToken, collateralToken, oracle, irm, lltv }. Market ID = keccak256(abi.encode(marketParams)).

Contract address: Morpho Blue on Ethereum mainnet = 0xBBBBBbbBBb9cc5e90e3b3Af64bdAF62C37EEFFCb. Immutable singleton.

Implementation milestone: m29-10

17.4 프로토콜 간 차이점 정리:

Comparison table:
| Aspect | Aave V3 | Kamino | Morpho Blue |
|---|---|---|---|
| Chain | EVM (multi) | Solana | EVM (Ethereum) |
| Position model | Global account | Obligation-based | Per-market isolated |
| Health factor | Single getUserAccountData() | borrowLimit / totalBorrow | Per-market computation, aggregate via min() |
| Market discovery | On-chain (getReservesList) | On-chain (reserves) | Config-based curated list |
| Multi-step | approve + action | Single TX (instructions) | approve + action |
| Interest rate | Variable (default), Stable (deprecated) | Variable only | Variable only (IRM-based) |
| Withdraw target | Specified address | Wallet (implicit) | Specified address |
| SDK dependency | viem ABI encoding | @kamino-finance/klend-sdk | viem ABI encoding |
| Solana/EVM adapter | EvmAdapter | SolanaAdapter | EvmAdapter |

17.5 설계 결정 기록:
- DEC-LEND-15: Morpho getMarkets() uses config-based curated list (not on-chain discovery) — Morpho Blue has no market registry; self-hosted model means operator curates which markets to support
- DEC-LEND-16: Kamino resolve() converts klend-sdk instructions to SolanaTransactionRequest (not ContractCallRequest) — Solana transactions use different request type, handled by SolanaAdapter in pipeline
- DEC-LEND-17: Aave V3 interest rate mode defaults to 2 (variable) in input schema with explicit note about stable rate deprecation — prevents STABLE_BORROWING_NOT_ENABLED errors
- DEC-LEND-18: Morpho health factor aggregation uses min() of per-market health factors — conservative approach surfaces worst-case risk to AI agent and monitoring
