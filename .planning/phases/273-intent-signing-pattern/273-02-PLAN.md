**Section 25.1: IChainAdapter signTypedData 확장**

Design the IChainAdapter extension for EIP-712 typed data signing:

```typescript
// TypedDataParams (input to signTypedData)
interface TypedDataParams {
  domain: {
    name: string;
    version: string;
    chainId: number;
    verifyingContract: `0x${string}`;
  };
  types: Record<string, Array<{ name: string; type: string }>>;
  primaryType: string;
  message: Record<string, unknown>;
}

// TypedDataSignature (output from signTypedData)
interface TypedDataSignature {
  signature: `0x${string}`;  // 65-byte signature (r[32] + s[32] + v[1])
  signer: `0x${string}`;     // recovered signer address for verification
}

// IChainAdapter extension (+1 method, total 23 methods)
signTypedData(params: TypedDataParams, privateKey: Uint8Array): Promise<TypedDataSignature>;
```

Implementation notes for EvmAdapter:
- Delegates to viem: `privateKeyToAccount(hexKey).signTypedData({ domain, types, primaryType, message })`
- Same key handling pattern as existing `signExternalTransaction()`: Uint8Array -> hex conversion, try/finally for key release
- Returns recovered signer address for verification: `recoverTypedDataAddress({ domain, types, primaryType, message, signature })`

SolanaAdapter:
- `signTypedData()` throws `ChainError('NOT_SUPPORTED', 'EIP-712 typed data signing is EVM-only')`
- Solana intent protocols (if any) would use a different signing mechanism, out of scope for v29

Relationship to existing sign-only pipeline:
- sign-only: `signExternalTransaction(txHash, privateKey)` — signs raw transaction bytes
- signTypedData: signs EIP-712 structured data — completely different code path
- Both share key management lifecycle (decrypt -> use -> release) but differ in cryptographic operation

**Section 25.3: IntentOrderTracker (IAsyncStatusTracker 구현)**

Design IntentOrderTracker following the established IAsyncStatusTracker pattern (LiFi bridge, Lido withdrawal, Jito epoch):

```typescript
class IntentOrderTracker implements IAsyncStatusTracker {
  name = 'intent-order';
  pollIntervalMs = 10_000;    // 10 seconds
  maxAttempts = 180;           // 30 minutes total
  timeoutTransition = 'TIMEOUT' as const;

  async checkStatus(metadata: BridgeMetadata): Promise<AsyncStatusResult> {
    const { statusApiUrl, orderUid, protocol } = metadata;

    // GET {statusApiUrl}/{orderUid}
    const response = await fetch(`${statusApiUrl}/${orderUid}`);
    const order = await response.json();

    // Status mapping (CoW Protocol -> WAIaaS)
    switch (order.status) {
      case 'presignaturePending':
      case 'open':
        return { state: 'PENDING' };
      case 'fulfilled':
        return {
          state: 'COMPLETED',
          details: {
            executedSellAmount: order.executedSellAmount,
            executedBuyAmount: order.executedBuyAmount,
            solver: order.solver,
          },
        };
      case 'cancelled':
      case 'expired':
        return {
          state: 'FAILED',
          details: { reason: order.status },
        };
      default:
        return { state: 'PENDING' }; // Unknown status, keep polling
    }
  }
}
```

Registration: IntentOrderTracker is registered in AsyncPollingService during daemon startup (same as LiFiBridgeTracker, LidoWithdrawalTracker).

bridge_metadata JSON structure for intent orders:
```typescript
{
  tracker: 'intent-order',
  orderUid: string,           // Protocol-specific order ID
  statusApiUrl: string,       // From intentMetadata.statusApiUrl
  protocol: string,           // 'cow_protocol'
  notificationEvent: 'INTENT_ORDER_FILLED',
  enrolledAt: number,         // Date.now()
}
```

Notification on completion: INTENT_ORDER_FILLED event dispatched via EventBus when order transitions to 'fulfilled'. Include executedSellAmount, executedBuyAmount in notification payload for AI agent consumption.

**Section 25.4: ContractCallRequest 파이프라인과의 분기점**

Define the exact bifurcation point in the existing codebase:

Location: `packages/daemon/src/api/routes/actions.ts` — the actions route handler.

Current flow:
```
POST /v1/actions/:action → registry.executeResolve() → ContractCallRequest[] → 6-stage pipeline
```

Extended flow:
```
POST /v1/actions/:action → registry.executeResolve() → (ContractCallRequest | SignableOrder)[]
                                                              ↓
                                                    type === 'INTENT'?
                                                    ├── YES → Intent pipeline (10-step)
                                                    └── NO  → 6-stage pipeline (unchanged)
```

Key constraint: A single executeResolve() call should return EITHER all ContractCallRequest OR all SignableOrder results, not mixed. This is because intent protocols (CoW) return a single order, and DeFi protocols return one or more contract calls. Mixed results would require complex transaction atomicity guarantees.

If mixed results are returned (error case):
- Log warning
- Reject the entire batch with INVALID_ACTION_RESULT error

Document that the 6-stage pipeline code path is completely unchanged — intent support is additive, not modifying.

**Section 26.1: 4중 바인딩 (chainId + verifyingContract + nonce + deadline)**

Document the 4-layer binding that prevents signature misuse:

1. **chainId binding**: EIP-712 domain.chainId is included in the domain separator hash. A signature for chainId=1 (mainnet) is invalid on chainId=11155111 (sepolia). Server-side validation: `domain.chainId === wallet.chainId`. Prevents cross-chain replay.

2. **verifyingContract binding**: EIP-712 domain.verifyingContract specifies the settlement contract. A signature for CoW Protocol's 0x9008D19f... is invalid on any other contract. Server-side validation: verifyingContract must be in INTENT_VERIFYING_CONTRACT_WHITELIST. Prevents phishing contract attacks.

3. **nonce binding**: Protocol-specific nonce (e.g., CoW Protocol uses order UID derived from order hash + owner + validTo, not a sequential nonce). The nonce is protocol-managed, not wallet-managed. For CoW Protocol: the orderUid is `keccak256(orderDigest, owner, validTo)` — unique per order. Server-side tracking: store orderUid in bridge_metadata for duplicate detection.

4. **deadline binding (validTo)**: Unix timestamp after which the signature is invalid. Server-side validation: `validTo - Math.floor(Date.now() / 1000) <= MAX_DEADLINE_SECONDS` (default 300 = 5 minutes). Prevents long-lived signatures that could be exploited if leaked.

Visualize the 4-layer protection:
```
┌─ Layer 1: chainId ──────── Prevents cross-chain replay ──────────┐
│  ┌─ Layer 2: verifyingContract ── Prevents phishing contract ───┐ │
│  │  ┌─ Layer 3: nonce/orderUid ──── Prevents order replay ────┐ │ │
│  │  │  ┌─ Layer 4: validTo ──────── Limits signature lifetime ┐│ │ │
│  │  │  │                                                      ││ │ │
│  │  │  │            EIP-712 Signed Order                      ││ │ │
```

**Section 26.2: 서버 사이드 검증 규칙**

Define the IntentSecurityValidator with ordered validation rules (fail-fast):

```typescript
class IntentSecurityValidator {
  // Rule 1: Deadline validation
  validateDeadline(validTo: number): void {
    const now = Math.floor(Date.now() / 1000);
    const remaining = validTo - now;
    if (remaining <= 0) throw new ChainError('INTENT_EXPIRED', 'Order deadline has already passed');
    if (remaining > MAX_DEADLINE_SECONDS) throw new ChainError('INTENT_DEADLINE_TOO_LONG',
      `Deadline ${remaining}s exceeds maximum ${MAX_DEADLINE_SECONDS}s`);
  }

  // Rule 2: Chain ID validation
  validateChainId(domainChainId: number, walletChainId: number): void {
    if (domainChainId !== walletChainId) throw new ChainError('INTENT_CHAIN_MISMATCH',
      `Domain chainId ${domainChainId} does not match wallet chainId ${walletChainId}`);
  }

  // Rule 3: Verifying contract whitelist (default-deny)
  validateVerifyingContract(verifyingContract: string, whitelist: string[]): void {
    if (!whitelist.length) throw new ChainError('INTENT_NO_WHITELIST',
      'INTENT_VERIFYING_CONTRACT_WHITELIST not configured — all intent signing denied');
    const normalized = verifyingContract.toLowerCase();
    if (!whitelist.some(addr => addr.toLowerCase() === normalized)) {
      throw new ChainError('INTENT_CONTRACT_NOT_WHITELISTED',
        `verifyingContract ${verifyingContract} not in whitelist`);
    }
  }

  // Rule 4: Duplicate order detection
  async validateNoDuplicate(orderHash: string): Promise<void> {
    const existing = await db.select().from(transactions)
      .where(and(
        eq(transactions.type, 'INTENT'),
        sql`json_extract(bridge_metadata, '$.orderUid') = ${orderHash}`
      )).get();
    if (existing) throw new ChainError('INTENT_DUPLICATE_ORDER',
      `Order ${orderHash} already submitted`);
  }
}
```

Validation execution order (pipeline step 3):
1. validateDeadline — reject expired or too-long-lived orders first (cheapest check)
2. validateChainId — reject cross-chain attempts
3. validateVerifyingContract — reject unwhitelisted contracts (default-deny)
4. validateNoDuplicate — reject replay attempts (DB query, most expensive)

**INTENT_VERIFYING_CONTRACT_WHITELIST policy:**
- Follows default-deny pattern identical to CONTRACT_WHITELIST
- Stored in Admin Settings as `intent_verifying_contract_whitelist`
- Format: JSON array of 0x-prefixed addresses (case-insensitive comparison)
- If empty or not configured: ALL intent signing is denied
- Configurable via Admin UI and CLI
- Per-chain: whitelist entries implicitly scoped to the wallet's chain (chainId in domain)

**MAX_DEADLINE_SECONDS configuration:**
- Default: 300 (5 minutes)
- Stored in Admin Settings as `intent_max_deadline_seconds`
- Hot-reloadable via SettingsService
- Range: 60-3600 (1 minute to 1 hour, validated)

**Section 26.3: 공격 벡터 분석 + 완화**

Document known attack vectors and mitigations:

| Attack Vector | Description | Mitigation | Validation Rule |
|---------------|-------------|------------|-----------------|
| Cross-chain replay | Signature valid on mainnet replayed on testnet | chainId in EIP-712 domain separator | Rule 2: chainId must match wallet |
| Phishing contract | Malicious provider sets verifyingContract to attacker contract | INTENT_VERIFYING_CONTRACT_WHITELIST (default-deny) | Rule 3: whitelist check |
| Signature replay | Same signed order submitted multiple times | Protocol nonce (orderUid uniqueness) + DB duplicate check | Rule 4: duplicate detection |
| Long-lived signature | Leaked signature usable for days/weeks | MAX_DEADLINE_SECONDS (5min) | Rule 1: deadline validation |
| Amount manipulation | Provider inflates sellAmount after user approval | SignableOrder is Zod-validated before signing; signed message is immutable | Step 2: Zod validation |
| API endpoint hijack | Provider sets orderApiUrl to attacker server | INTENT_VERIFYING_CONTRACT_WHITELIST ensures only known protocols; orderApiUrl could optionally be validated against known protocol APIs | Rule 3 + optional URL whitelist |
| Key exposure via signTypedData | Private key remains in memory after signing | Same mitigation as signExternalTransaction: try/finally with releaseKey() | Step 8: key release in finally |

Residual risk assessment:
- **Solver front-running**: Not mitigable by wallet — inherent to solver model. CoW Protocol's batch auction design partially mitigates.
- **Price slippage**: Mitigated by buyAmount minimum in the signed order. AI agent must set reasonable slippage.
- **Protocol-level vulnerability**: Not in WAIaaS scope — settlement contract security is the protocol's responsibility.

**Section 26.4: 설계 결정**

Record design decisions:
- DEC-INTENT-15: IntentSecurityValidator is a dedicated class (not embedded in PolicyEngine) — security validation runs before policy evaluation
- DEC-INTENT-16: INTENT_VERIFYING_CONTRACT_WHITELIST is default-deny (consistent with CONTRACT_WHITELIST pattern)
- DEC-INTENT-17: MAX_DEADLINE_SECONDS=300 (5 minutes) — balances security (short-lived signatures) with usability (enough time for solver execution)
- DEC-INTENT-18: Validation order is fail-fast: deadline -> chainId -> whitelist -> duplicate (cheapest to most expensive)
- DEC-INTENT-19: Nonce management is protocol-side (CoW orderUid), not WAIaaS-side — WAIaaS only checks for duplicate submission
- DEC-INTENT-20: Key release follows existing signExternalTransaction pattern (try/finally with releaseKey())
- DEC-INTENT-21: Residual risks (solver front-running, price slippage) are documented but not WAIaaS's responsibility
