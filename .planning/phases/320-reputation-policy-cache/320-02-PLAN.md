    ```typescript
    private async evaluateReputationThreshold(
      resolved: PolicyRow[],
      transaction: TransactionParam,
    ): Promise<{ tier: PolicyTier; reason: string } | null>
    ```
    Logic:
    - Find REPUTATION_THRESHOLD policy in resolved
    - If not found, return null
    - Parse rules as ReputationThresholdRules
    - If check_counterparty is false, return null (skip)
    - If no reputationCacheService, apply unrated_tier (treat as unrated when service unavailable)
    - Lookup counterparty: use `transaction.toAddress` as the agentId for reputation lookup (ERC-8004 agentId can be derived from address, but for policy evaluation the toAddress IS the lookup key since it's the on-chain identity)
    - NOTE: For ERC-8004, the agentId is a uint256 NFT token ID, not an address. The toAddress is the contract being called. For the REPUTATION_THRESHOLD policy to work in practice, we need to resolve the agentId from the toAddress. Strategy: Check if toAddress matches any `agent_identities.wallet_id` in the local DB, and if so use the corresponding `chain_agent_id` as the agentId for the reputation lookup. If no match found, treat as unrated.
    - Actually simpler: the `toAddress` in TransactionParam is the counterparty wallet. We check `agent_identities` table for any row where the wallet's address matches `toAddress`. If found, get the `chain_agent_id` and use that for reputation lookup. If not found, treat as unrated (this is a non-registered agent).
    - Call `reputationCacheService.getReputation(agentId, rules.tag1, rules.tag2)`
    - If null (unrated): return `{ tier: rules.unrated_tier as PolicyTier, reason: 'Counterparty agent unrated (no reputation data)' }`
    - If score < min_score: return `{ tier: rules.below_threshold_tier as PolicyTier, reason: 'Counterparty reputation below threshold: score=${score}, min=${min_score}' }`
    - If score >= min_score: return null (pass, continue evaluation)

    d. In `evaluate()` method, add Step 4e.5 AFTER the APPROVED_SPENDERS block (step 4e, line ~316) and BEFORE APPROVE_AMOUNT_LIMIT (step 4f, line ~318):
    ```typescript
    // Step 4e.5: Evaluate REPUTATION_THRESHOLD (counterparty reputation check)
    const reputationResult = await this.evaluateReputationThreshold(resolved, transaction);
    if (reputationResult !== null) {
      // Reputation-based tier is an escalation, not a final result.
      // Store it and apply via maxTier at the end with SPENDING_LIMIT result.
      // But since reputation doesn't deny -- it only escalates tier -- we need
      // a different approach. Return the escalated tier as the minimum floor.
      // Actually: REPUTATION_THRESHOLD sets a tier floor. Continue evaluation
      // and the final tier = maxTier(reputation tier, spending tier).
      // For simplicity and consistency with APPROVE_TIER_OVERRIDE pattern:
      // We track the reputation tier as a "floor" and apply maxTier at the end.
    }
    ```

    IMPORTANT design choice: Unlike deny policies (WHITELIST, ALLOWED_TOKENS), REPUTATION_THRESHOLD does NOT deny transactions. It ESCALATES the tier. So it should NOT return early. Instead:
    - Store the reputation tier as a `reputationFloorTier` variable
    - At the end of evaluate(), before returning, apply `maxTier(result.tier, reputationFloorTier)` if set
    - This is the same pattern as how cumulative spending works (maxTier of per-tx tier and cumulative tier)

    Revised approach:
    - After step 4e (APPROVED_SPENDERS), call evaluateReputationThreshold
    - If result is non-null, store `reputationFloorTier = result.tier`
    - Continue all other evaluations as normal
    - At the final return point (after SPENDING_LIMIT), apply `maxTier(finalTier, reputationFloorTier)`
    - This ensures reputation only escalates, never downgrades

    - Since evaluateReputationThreshold is async (RPC call) but evaluateAndReserve is sync (BEGIN IMMEDIATE), we need to handle this. Two options:
      a. Pre-fetch reputation BEFORE entering the transaction (in stage3Policy)
      b. Make evaluateReputationThreshold work with pre-fetched data

    Best approach: Pre-fetch reputation in `stage3Policy()` (outside the IMMEDIATE txn) and pass it as a parameter to evaluateAndReserve. This avoids async inside sync transaction.
