두 개의 IAsyncStatusTracker 구현체를 생성한다.

**LidoWithdrawalTracker** (`packages/actions/src/providers/lido-staking/withdrawal-tracker.ts`):
- `implements IAsyncStatusTracker` (import from `../../common/async-status-tracker.js`)
- `name = 'lido-withdrawal'`
- `maxAttempts = 480` (480 x 30s = 4 hours active polling; Lido withdrawal takes 1-5 days, but active phase polls aggressively)
- `pollIntervalMs = 30_000` (30초)
- `timeoutTransition = 'TIMEOUT'` (terminal -- no two-phase monitoring for staking unlike bridges)
- `checkStatus(txId, metadata)`: Lido Withdrawal Queue는 on-chain 상태이므로, metadata에서 `requestId`와 `withdrawalQueueAddress`를 읽어 EVM RPC로 `isFinalized` 상태를 조회한다. 단, 외부 RPC 호출은 실제 구현이 복잡하므로 metadata-based 간이 방식을 사용: metadata.status가 이미 'claimable'이면 COMPLETED 반환, metadata.withdrawalRequestedAt 기반으로 예상 시간 5일 경과 시 TIMEOUT hint를 details에 포함하며 PENDING 유지. 이는 checkStatus가 아직 체인 조회 없이 메타데이터 기반으로 동작하는 v1 구현이다.
- 실제로는 metadata에 `tracker: 'lido-withdrawal'`가 기록되어야 AsyncPollingService가 이 tracker를 resolve한다.
- COMPLETED 시 details에 `{ completedAt: Date.now(), protocol: 'lido' }` 포함
- PENDING 시 details에 `{ estimatedDaysRemaining: number, protocol: 'lido' }` 포함

**JitoEpochTracker** (`packages/actions/src/providers/jito-staking/epoch-tracker.ts`):
- `implements IAsyncStatusTracker`
- `name = 'jito-epoch'`
- `maxAttempts = 240` (240 x 30s = 2 hours; Solana epoch boundary ~2-3일이지만 active polling은 2시간)
- `pollIntervalMs = 30_000` (30초)
- `timeoutTransition = 'TIMEOUT'` (terminal)
- `checkStatus(txId, metadata)`: metadata에서 `targetEpoch`를 읽고, 현재 에포크와 비교하여 경과 여부를 판단. metadata.status가 'deactivated'이면 COMPLETED. metadata.unstakeRequestedAt 기반으로 예상 시간(~3일) 경과 시 정보를 details에 포함. v1은 metadata 기반 추적.
- COMPLETED 시 details에 `{ completedAt: Date.now(), protocol: 'jito' }` 포함

**packages/actions/src/index.ts**:
- `export { LidoWithdrawalTracker } from './providers/lido-staking/withdrawal-tracker.js'`
- `export { JitoEpochTracker } from './providers/jito-staking/epoch-tracker.js'`

**알림 이벤트 매핑**: AsyncPollingService의 callbacks.emitNotification은 이미 daemon.ts에서 NotificationService.notify()로 연결되어 있다 (Step 4c-10). 따라서 tracker가 COMPLETED를 반환하면 processResult에서 emitNotification이 호출되고, 이벤트 타입이 'BRIDGE_COMPLETED'로 발행된다.

스테이킹 전용 이벤트를 위해 AsyncPollingService.processResult가 tracker.name을 기반으로 이벤트를 분기하도록 해야 하지만, 현재 구조에서 이벤트 타입은 'BRIDGE_COMPLETED'/'BRIDGE_FAILED'/'BRIDGE_TIMEOUT' 하드코딩이다. 이를 수정하면 기존 bridge 동작에 영향을 줄 수 있으므로, v1에서는 tracker의 checkStatus details에 `{ notificationEvent: 'STAKING_UNSTAKE_COMPLETED' }` 키를 포함시키고, processResult에서 details.notificationEvent가 있으면 그 값을 사용하도록 AsyncPollingService를 최소 수정한다:

`packages/daemon/src/services/async-polling-service.ts`의 processResult에서 COMPLETED 케이스:
```typescript
// Determine notification event type
const eventType = (updatedMetadata.notificationEvent as string) ?? (isRefunded ? 'BRIDGE_REFUNDED' : 'BRIDGE_COMPLETED');
```

FAILED 케이스:
```typescript
const eventType = (updatedMetadata.notificationEvent as string) ?? 'BRIDGE_FAILED';
```

handleTimeout에서 TIMEOUT 케이스:
```typescript
const eventType = (metadata.notificationEvent as string) ?? 'BRIDGE_TIMEOUT';
```

LidoWithdrawalTracker와 JitoEpochTracker의 checkStatus에서 COMPLETED 시 `details.notificationEvent = 'STAKING_UNSTAKE_COMPLETED'`, metadata에는 tracker 등록 시 `notificationEvent: 'STAKING_UNSTAKE_TIMEOUT'`도 포함 (timeout 시 사용).

이렇게 하면 bridge 기존 동작은 notificationEvent 미설정이므로 기존 BRIDGE_* 이벤트를 그대로 사용하고, staking tracker는 STAKING_UNSTAKE_COMPLETED/TIMEOUT을 사용한다.

**수정 대상 파일**: daemon.ts (tracker 등록), async-polling-service.ts (이벤트 타입 동적 결정)
