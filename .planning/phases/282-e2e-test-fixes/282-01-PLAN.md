# Plan 282-01: Default Wallet/Network Removal E2E Verification Tests

## Goal
Create a comprehensive E2E test file that verifies all 8 behavioral changes from the v29.3 "default wallet/default network removal" milestone. These tests use the full Hono app (`createApp()`) with real middleware/routes to verify end-to-end behavior.

## Context

Phases 279-281 removed the default wallet/default network concept from:
- DB schema (migration v27: dropped `is_default`, `default_network` columns)
- Core resolution (resolveWalletId 2-priority, resolveNetwork getSingleNetwork)
- JWT (no `wlt` claim)
- API endpoints (3 routes deleted, response fields removed)
- SDK/MCP/Admin UI (external interfaces cleaned)

This plan creates integration tests that exercise the full HTTP stack to verify each behavioral change end-to-end.

5. **Test group 4: JWT payload (E2E-06)**

   ```typescript
   describe('JWT payload (E2E-06)', () => {
     it('newly issued JWT has no wlt claim', async () => {
       const wallet = await createWallet('solana', 'testnet', 'sol-jwt');
       const { token } = await createSession([wallet.id]);
       // Decode the JWT payload (wai_sess_ prefix + base64url JWT)
       const jwtPart = token.replace('wai_sess_', '');
       const parts = jwtPart.split('.');
       expect(parts.length).toBe(3);
       const payloadJson = JSON.parse(
         Buffer.from(parts[1]!, 'base64url').toString('utf-8'),
       );
       // Should have sub, iat, exp but NOT wlt
       expect(payloadJson).toHaveProperty('sub');
       expect(payloadJson).toHaveProperty('iat');
       expect(payloadJson).toHaveProperty('exp');
       expect(payloadJson).not.toHaveProperty('wlt');
     });
   });
   ```

7. **Test group 6: MCP multi-wallet (E2E-08)**

   For MCP testing, we verify the behavior through the REST API that MCP calls under the hood, since MCP tools ultimately call the same API endpoints. The key behavior is that multi-wallet sessions must specify wallet_id.

   ```typescript
   describe('MCP multi-wallet wallet_id requirement (E2E-08)', () => {
     it('multi-wallet session transaction without walletId -> WALLET_ID_REQUIRED', async () => {
       const wallet1 = await createWallet('solana', 'testnet', 'mcp-sol-1');
       const wallet2 = await createWallet('solana', 'testnet', 'mcp-sol-2');
       const { token } = await createSession([wallet1.id, wallet2.id]);
       // POST /v1/wallet/send -- the API endpoint MCP uses for transfers
       const res = await app.request('/v1/wallet/send', {
         method: 'POST',
         headers: {
           Host: HOST,
           'Content-Type': 'application/json',
           ...authHeader(token),
         },
         body: JSON.stringify({
           to: '11111111111111111111111111111112',
           amount: '0.001',
           // walletId intentionally omitted
         }),
       });
       expect(res.status).toBe(400);
       const body = await json(res);
       expect(body.code).toBe('WALLET_ID_REQUIRED');
     });

     it('multi-wallet session with explicit walletId succeeds (or reaches pipeline)', async () => {
       const wallet1 = await createWallet('solana', 'testnet', 'mcp-sol-3');
       const wallet2 = await createWallet('solana', 'testnet', 'mcp-sol-4');
       const { token } = await createSession([wallet1.id, wallet2.id]);
       // POST /v1/wallet/send with explicit walletId
       const res = await app.request('/v1/wallet/send', {
         method: 'POST',
         headers: {
           Host: HOST,
           'Content-Type': 'application/json',
           ...authHeader(token),
         },
         body: JSON.stringify({
           to: '11111111111111111111111111111112',
           amount: '0.001',
           walletId: wallet1.id,
         }),
       });
       // Should not be 400/WALLET_ID_REQUIRED -- may be 200 (submitted) or 4xx pipeline error
       expect(res.status).not.toBe(400);
       // OR if it reaches pipeline and there's no policy setup, it may return a different status
       // The key assertion: it's NOT WALLET_ID_REQUIRED
       if (res.status === 400) {
         const body = await json(res);
         expect(body.code).not.toBe('WALLET_ID_REQUIRED');
       }
     });
   });
   ```

   **Important notes for the executor:**
   - The MCP tools call REST API endpoints, so testing the REST API with multi-wallet session tokens effectively validates MCP behavior.
   - The balance endpoint (`GET /v1/wallet/balance`) and send endpoint (`POST /v1/wallet/send`) both call `resolveWalletId()` which enforces the WALLET_ID_REQUIRED rule.
   - If the send endpoint reaches the pipeline but fails for other reasons (no policy, adapter mock issues), that's acceptable -- the key verification is that WALLET_ID_REQUIRED is correctly thrown when walletId is missing, and NOT thrown when walletId is present.
