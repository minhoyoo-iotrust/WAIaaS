## 1. 개요

데몬의 운영 통계를 단일 JSON 엔드포인트 `GET /v1/admin/stats`로 제공하기 위한 데이터 모델을 설계한다. AdminStatsResponseSchema (Zod), 인메모리 카운터 인터페이스 (IMetricsCounter), DB 집계 쿼리, 1분 TTL 캐시를 정의한다.

### 1.1 설계 원칙

1. **외부 의존성 없음** -- Prometheus/Grafana 없이 JSON 응답만으로 충분
2. **Zod SSoT** -- 응답 스키마를 Zod로 정의, TS 타입 자동 도출
3. **인메모리 + DB 혼합** -- 실시간 카운터는 메모리, 이력 집계는 DB
4. **1분 TTL 캐시** -- 빈번한 호출 시 DB 부하 방지

## 3. 인메모리 카운터 인터페이스 (STAT-02)

### 3.1 IMetricsCounter 인터페이스

```typescript
// packages/core/src/metrics/metrics-counter.ts

/**
 * 인메모리 메트릭 카운터 인터페이스.
 * 데몬 시작 시 초기화, 재시작 시 리셋 (영속 불필요).
 */
export interface IMetricsCounter {
  /** 단일 카운터 증가 */
  increment(key: string, labels?: Record<string, string>): void;

  /** 레이턴시 기록 (히스토그램 대체) */
  recordLatency(key: string, durationMs: number, labels?: Record<string, string>): void;

  /** 카운터 값 조회 */
  getCount(key: string, labels?: Record<string, string>): number;

  /** 평균 레이턴시 조회 */
  getAvgLatency(key: string, labels?: Record<string, string>): number;

  /** 전체 메트릭 스냅샷 (stats 엔드포인트용) */
  snapshot(): MetricsSnapshot;

  /** 리셋 (테스트용) */
  reset(): void;
}

export interface MetricsSnapshot {
  counters: Record<string, number>;
  latencies: Record<string, { count: number; totalMs: number; avgMs: number }>;
}
```

### 3.3 카운터 연동 지점 (Integration Points)

| 카운터 키 | 라벨 | 연동 지점 | 설명 |
|----------|------|----------|------|
| `rpc.calls` | `network` | `RpcPool.getUrl()` 호출 후 성공 시 | RPC 호출 횟수 |
| `rpc.errors` | `network` | `RpcPool.reportFailure()` 호출 시 | RPC 에러 횟수 |
| `rpc.latency` | `network` | adapter 메서드 래퍼 (try/finally) | RPC 응답 시간 |
| `autostop.triggered` | `rule` | `AutoStopService.suspendWallet()` 시 | AutoStop 발동 |
| `tx.submitted` | `chain` | Stage 5d 완료 시 | TX 제출 수 (인메모리 보조) |
| `tx.failed` | `chain` | Pipeline 실패 시 | TX 실패 수 (인메모리 보조) |

### 3.4 RPC 카운터 연동 방법

RpcPool에 onEvent 콜백이 이미 존재한다. 추가 연동 방식:

```typescript
// daemon.ts (데몬 초기화)
const metricsCounter = new InMemoryMetricsCounter();

const rpcPool = RpcPool.createWithDefaults({
  onEvent: (event) => {
    // 기존 이벤트 핸들링...

    // 메트릭 카운터 연동
    if (event.type === 'RPC_HEALTH_DEGRADED') {
      metricsCounter.increment('rpc.errors', { network: event.network });
    }
  },
});
```

RPC 호출 레이턴시는 ChainAdapter 래퍼 수준에서 측정:

```typescript
// 어댑터 메서드 호출 래퍼 (설계 패턴)
async function withRpcMetrics<T>(
  counter: IMetricsCounter,
  network: string,
  fn: () => Promise<T>,
): Promise<T> {
  const start = Date.now();
  try {
    const result = await fn();
    counter.increment('rpc.calls', { network });
    counter.recordLatency('rpc.latency', Date.now() - start, { network });
    return result;
  } catch (e) {
    counter.increment('rpc.errors', { network });
    counter.recordLatency('rpc.latency', Date.now() - start, { network });
    throw e;
  }
}
```

## 4. DB 집계 쿼리 (STAT-03)

### 4.1 transactions 집계

```sql
-- 상태별 카운트 (전체)
SELECT status, COUNT(*) as cnt FROM transactions GROUP BY status;

-- 타입별 카운트 (전체)
SELECT type, COUNT(*) as cnt FROM transactions GROUP BY type;

-- 최근 24시간 통계
SELECT
  COUNT(*) as count,
  COALESCE(SUM(amount_usd), 0) as volume_usd,
  CAST(SUM(CASE WHEN status = 'CONFIRMED' THEN 1 ELSE 0 END) AS REAL)
    / NULLIF(COUNT(*), 0) as success_rate,
  AVG(CASE
    WHEN executed_at IS NOT NULL AND created_at IS NOT NULL
    THEN executed_at - created_at
    ELSE NULL
  END) as avg_duration_sec
FROM transactions
WHERE created_at >= ?;  -- now - 86400

-- 최근 7일 통계
SELECT
  COUNT(*) as count,
  COALESCE(SUM(amount_usd), 0) as volume_usd
FROM transactions
WHERE created_at >= ?;  -- now - 604800
```

### 4.2 sessions 집계

```sql
-- 활성 세션 수 (revokedAt IS NULL AND expiresAt > now)
SELECT COUNT(*) as active
FROM sessions
WHERE revoked_at IS NULL AND expires_at > ?;  -- now (timestamp)

-- 전체 세션 수
SELECT COUNT(*) as total FROM sessions;

-- 24시간 내 폐기 세션 수
SELECT COUNT(*) as revoked_24h
FROM sessions
```

## 5. 1분 TTL 캐시 (STAT-03)

### 5.1 설계

```typescript
// packages/daemon/src/services/admin-stats-service.ts

export class AdminStatsService {
  private cache: AdminStatsResponse | null = null;
  private cacheTimestamp = 0;
  private readonly ttlMs = 60_000; // 1분

  constructor(private deps: {
    sqlite: Database;
    rpcPool: RpcPool;
    autoStopService: AutoStopService;
    metricsCounter: IMetricsCounter;
    settingsService: SettingsService;
  }) {}

  async getStats(): Promise<AdminStatsResponse> {
    const now = Date.now();
    if (this.cache && (now - this.cacheTimestamp) < this.ttlMs) {
      return this.cache;
    }

    const stats = this.buildStats();
    this.cache = stats;
    this.cacheTimestamp = now;
    return stats;
  }

  /** 캐시 무효화 (테스트/강제 갱신용) */
  invalidateCache(): void {
    this.cache = null;
    this.cacheTimestamp = 0;
  }

  private buildStats(): AdminStatsResponse {
    // 1. DB 집계 (동기 -- better-sqlite3 드라이버)
    const txStats = this.queryTransactionStats();
    const sessionStats = this.querySessionStats();
    const walletStats = this.queryWalletStats();
    const notifStats = this.queryNotificationStats();

    // 2. 인메모리 카운터
    const rpcStats = this.buildRpcStats();
    const autoStopStats = this.buildAutoStopStats();

    // 3. 시스템 정보
    const systemStats = this.buildSystemStats();

    return {
      transactions: txStats,
      sessions: sessionStats,
      wallets: walletStats,
      rpc: rpcStats,
      autostop: autoStopStats,
      notifications: notifStats,
      system: systemStats,
    };
  }
}
```

### 5.2 캐시 설계 근거

| 항목 | 결정 | 근거 |
|------|------|------|
| TTL | 1분 | Admin 대시보드 자동 새로고침 주기 (30초~60초) 대비 충분 |
| 캐시 단위 | 전체 응답 1건 | 카테고리별 분리 시 복잡도 대비 이점 미미 |
| 무효화 | TTL 만료 + 수동 `invalidateCache()` | 이벤트 기반 무효화는 과도한 복잡도 |
| 동시 접근 | Node.js 단일 스레드 -- 동시성 문제 없음 | better-sqlite3 동기 쿼리 |

### 5.3 DB 쿼리 실행 전략

```
AdminStatsService.getStats()
  ├── 캐시 유효? → 캐시 반환 (0ms)
  └── 캐시 만료
      ├── DB 집계 4건 (transactions, sessions, wallets, notification_logs)
      │   └── better-sqlite3 동기 실행, 각 ~1ms (인덱스 활용)
      ├── 인메모리 카운터 스냅샷 (~0ms)
      ├── RpcPool.getStatus() per network (~0ms)
      ├── AutoStopService.getStatus() (~0ms)
      └── 시스템 정보 수집 (~1ms, fs.statSync for DB size)
      → 총 ~5ms 예상
```

```typescript
function buildRpcPoolStatus(rpcPool: RpcPool): {
  totalEndpoints: number;
  availableEndpoints: number;
  cooldownEndpoints: number;
} {
  let total = 0, available = 0, cooldown = 0;
  for (const network of rpcPool.getNetworks()) {
    const statuses = rpcPool.getStatus(network);
    for (const s of statuses) {
      total++;
      if (s.status === 'available') available++;
      else cooldown++;
    }
  }
  return { totalEndpoints: total, availableEndpoints: available, cooldownEndpoints: cooldown };
}
```
