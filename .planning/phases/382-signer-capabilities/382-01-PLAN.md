From signer-capability-interface.md:
```typescript
export interface ISignerCapability {
  readonly scheme: SigningScheme;
  canSign(params: SigningParams): boolean;
  sign(params: SigningParams): Promise<SigningResult>;
}

export type SigningScheme = 'eip712' | 'personal' | 'hmac-sha256' | 'rsa-pss' | 'ecdsa-secp256k1' | 'ed25519' | 'erc8128';

export interface SigningResult {
  signature: string | Uint8Array;
  metadata?: Record<string, unknown>;
}

export class SigningError extends Error {
  constructor(message: string, scheme: SigningScheme, code: SigningErrorCode, cause?: Error);
}

export type SigningErrorCode = 'INVALID_KEY' | 'INVALID_PARAMS' | 'CREDENTIAL_MISSING' | 'SIGNING_FAILED' | 'KEY_NOT_SUPPORTED';
```

설계 문서를 작성한다. 구현 코드가 아닌 설계 명세만 산출하며, 각 capability별로 다음을 포함한다:

**1. Eip712SignerCapability 어댑터 설계:**
- scheme: 'eip712'
- 래핑 대상: sign-message.ts의 signTypedData 경로 (viem signTypedData 사용)
- canSign() 검사: params.scheme === 'eip712', domain/types/value 존재 확인
- sign() 위임: viem의 signTypedData({ privateKey, domain, types, primaryType, message: value })
- metadata 반환: { v, r, s } 분리값
- 에러 매핑: viem 에러 -> SigningError('SIGNING_FAILED')
- 기존 sign-message.ts 코드와의 관계: 직접 import 없음, 같은 viem 함수를 독립적으로 호출

**2. PersonalSignCapability 어댑터 설계:**
- scheme: 'personal'
- 래핑 대상: sign-message.ts의 signPersonalMessage 경로
- canSign(): params.scheme === 'personal', message 존재 확인
- sign(): viem의 signMessage({ privateKey, message })
- 기존 경로와 분리 보장

**3. Erc8128SignerCapability 어댑터 설계:**
- scheme: 'erc8128'
- 래핑 대상: core/erc8128/http-message-signer.ts
- canSign(): params.scheme === 'erc8128', method/url/headers 존재 확인
- sign(): HttpMessageSigner.sign() 위임 (기존 모듈 재사용 — 이 경우만 예외적으로 기존 모듈 import 허용, 이유: ERC-8128 RFC 9421 구현이 복잡하여 중복 불가)
- metadata 반환: { signatureInput, signedHeaders }
- import 허용 근거 문서화

**4. TransactionSignerCapability 참조용 설계:**
- scheme: 해당 없음 (SigningSchemeEnum에 transaction은 없음)
- 설계 목적: 기존 6-stage pipeline의 IChainAdapter.signTransaction()을 ISignerCapability로 표현하는 것이 가능함을 보여주되, registry에 등록하지 않음
- 향후 활용 시나리오 명시 (off-chain에서 트랜잭션 서명만 필요한 경우)
- 설계만 존재, 구현 우선순위 낮음

**5. EcdsaSignBytesCapability 설계 (signBytes 계열):**
- scheme: 'ecdsa-secp256k1'
- 구현 기반: viem의 sign() 또는 @noble/secp256k1 직접 사용
- canSign(): params.scheme === 'ecdsa-secp256k1', data(hex string) 존재 확인
- sign(): data를 hex decode -> hash(keccak256) -> secp256k1 sign -> hex signature 반환
- hash 전략: 입력이 이미 해시인 경우와 raw data인 경우 구분 (hashData 옵션 boolean, 기본값 true)
- 용도: 크로스체인 메시지, 커스텀 프로토콜
- 보안 고려: raw bytes signing은 위험성이 높으므로 정책 평가에서 별도 riskLevel 부여 권장 (Phase 384에서 설계)

**6. Ed25519SignBytesCapability 설계 (signBytes 계열):**
- scheme: 'ed25519'
- 구현 기반: @solana/kit의 signBytes() 또는 tweetnacl.sign.detached()
- canSign(): params.scheme === 'ed25519', data(hex string) 존재, privateKey(Uint8Array 64 bytes) 확인
- sign(): data를 hex decode -> ed25519 sign -> hex signature 반환
- Ed25519는 내부적으로 해시하므로 외부 해시 불필요
- 키 형태: Solana keypair 64 bytes (secret 32 + public 32)
- 용도: Solana native off-chain signing, 커스텀 프로토콜
