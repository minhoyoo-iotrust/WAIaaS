# Plan 305-02: Cursor Pagination + REST API 엔드포인트 스펙 + 설계 문서 갱신

## 목표

audit_log의 id AUTOINCREMENT 기반 cursor pagination 설계를 상세화하고, `GET /v1/audit-logs` 엔드포인트의 완전한 OpenAPI 스펙(요청/응답/에러 코드)을 정의한다. 영향받는 설계 문서 4개(25, 29, 32, 37)의 갱신 내용을 명시한다.

## 컨텍스트

### 기존 Pagination 패턴

코드베이스에 3가지 pagination 패턴이 공존한다:

| 엔드포인트 | 방식 | cursor 타입 | 정렬 | limit |
|-----------|------|------------|------|-------|
| `GET /v1/transactions` | cursor (UUID) | `string.uuid` | id DESC | 기본 20, 최대 100 |
| `GET /v1/wallet/incoming` | cursor (composite) | base64url({d,i}) | detectedAt DESC, id DESC | 기본 20, 최대 100 |
| `GET /v1/admin/notifications/log` | offset | page + pageSize | createdAt DESC | 기본 20, 최대 100 |
| `GET /v1/admin/transactions` | cursor (UUID) | `string.uuid` | id DESC | 기본 20, 최대 100 |

**선택: cursor 방식 (id 기반)**

- offset 방식(notification/log)은 대량 데이터에서 OFFSET 성능 저하 우려. audit_log는 빠르게 성장하는 테이블 (일 ~3,000건 추정)
- audit_log.id는 AUTOINCREMENT로 단조 증가 보장 → cursor 구현이 가장 단순
- 기존 transactions cursor(UUID)와 달리 정수 cursor → 비교 연산 효율적

## 설계 내용

### 1. Cursor Pagination 상세 설계 (AUDIT-02)

#### 1.1 개요

```
클라이언트                           서버
───────                           ─────
GET /v1/audit-logs?limit=50
                         ──→     SELECT * FROM audit_log
                                  ORDER BY id DESC
                                  LIMIT 51  (limit + 1)
                         ←──     { data: [...50건], nextCursor: 42, hasMore: true }

GET /v1/audit-logs?limit=50&cursor=42
                         ──→     SELECT * FROM audit_log
                                  WHERE id < 42
                                  ORDER BY id DESC
                                  LIMIT 51
                         ←──     { data: [...50건], nextCursor: 10, hasMore: true }

GET /v1/audit-logs?limit=50&cursor=10
                         ──→     SELECT * FROM audit_log
                                  WHERE id < 10
                                  ORDER BY id DESC
                                  LIMIT 51
                         ←──     { data: [...9건], nextCursor: null, hasMore: false }
```

#### 1.2 Cursor 동작 규칙

| 규칙 | 내용 |
|------|------|
| **정렬** | `ORDER BY id DESC` (최신 우선). id AUTOINCREMENT가 삽입 순서를 보장하므로 timestamp 정렬과 실질적으로 동일 |
| **cursor 의미** | `cursor=N`이면 `WHERE id < N`. 즉 cursor 값 미포함, 그 아래부터 |
| **hasMore 판정** | `LIMIT (limit + 1)` 쿼리 후 결과가 limit + 1개이면 hasMore=true, 마지막 행은 버림 |
| **nextCursor 계산** | hasMore=true일 때 반환 데이터의 마지막 항목의 id |
| **빈 결과** | data=[], nextCursor=null, hasMore=false |
| **cursor 없이 첫 호출** | cursor 파라미터 생략 → WHERE 조건 없음 (최신부터) |
| **기본 limit** | 50 |
| **최대 limit** | 200 |
| **최소 limit** | 1 |

#### 1.3 필터 + Cursor 조합

필터와 cursor를 동시에 사용하는 경우, 모든 필터 조건에 AND로 cursor 조건을 추가한다.

```sql
-- 예: wallet_id + event_type + from/to + cursor
SELECT * FROM audit_log
WHERE wallet_id = ?
  AND event_type = ?
  AND timestamp >= ?
  AND timestamp <= ?
  AND id < ?           -- cursor
ORDER BY id DESC
LIMIT 51;              -- limit + 1
```

**인덱스 활용:**
- `WHERE wallet_id = ? AND id < ?` → `idx_audit_log_wallet_id` + PRIMARY KEY (id)
- `WHERE event_type = ? AND id < ?` → `idx_audit_log_event_type` + PRIMARY KEY
- `WHERE wallet_id = ? AND timestamp BETWEEN ? AND ?` → `idx_audit_log_wallet_timestamp`
- `WHERE tx_id = ?` → `idx_audit_log_tx_id` (신규, 305-01에서 정의)

SQLite의 쿼리 플래너가 가장 선택적인 인덱스를 자동 선택한다.

#### 1.4 구현 의사 코드

```typescript
// packages/daemon/src/api/routes/audit-logs.ts (신규)

async function handleAuditLogQuery(query: AuditLogQuery, sqlite: SQLiteDatabase) {
  const limit = query.limit ?? 50;

  // WHERE 조건 동적 빌드
  const conditions: string[] = [];
  const params: (string | number)[] = [];

  if (query.wallet_id) {
    conditions.push('wallet_id = ?');
    params.push(query.wallet_id);
  }
  if (query.event_type) {
    conditions.push('event_type = ?');
    params.push(query.event_type);
  }
  if (query.severity) {
    conditions.push('severity = ?');
    params.push(query.severity);
  }
  if (query.from !== undefined) {
    conditions.push('timestamp >= ?');
    params.push(query.from);
  }
  if (query.to !== undefined) {
    conditions.push('timestamp <= ?');
    params.push(query.to);
  }
  if (query.tx_id) {
    conditions.push('tx_id = ?');
    params.push(query.tx_id);
  }
  if (query.cursor !== undefined) {
    conditions.push('id < ?');
    params.push(query.cursor);
  }

  const whereClause = conditions.length > 0
    ? `WHERE ${conditions.join(' AND ')}`
    : '';

  // 쿼리 실행 (limit + 1로 hasMore 판정)
  const sql = `SELECT * FROM audit_log ${whereClause} ORDER BY id DESC LIMIT ?`;
  params.push(limit + 1);

  const rows = sqlite.prepare(sql).all(...params) as AuditLogRow[];

  const hasMore = rows.length > limit;
  const data = hasMore ? rows.slice(0, limit) : rows;
  const nextCursor = hasMore && data.length > 0
    ? data[data.length - 1]!.id
    : null;

  return { data, nextCursor, hasMore };
}
```

**raw SQL 선택 근거:** Drizzle ORM으로도 구현 가능하지만, 동적 WHERE 빌드가 필요한 경우 raw SQL이 더 직관적. 기존 admin/incoming 라우트에서도 복잡한 필터 조합 시 raw SQL 사용 패턴이 있음.

#### 1.5 total 필드 (선택적)

전체 건수를 반환하는 `total` 필드는 별도 COUNT 쿼리가 필요하다.

```sql
SELECT COUNT(*) as count FROM audit_log ${whereClause};
```

**설계 결정:** total은 **기본적으로 포함하지 않음**. 쿼리 파라미터 `include_total=true`가 있을 때만 COUNT 실행. 이유:
- audit_log는 빠르게 성장 (일 ~3,000건, 연 ~1M건)
- 필터 없는 COUNT(*)는 전체 테이블 스캔
- 관리자가 전체 건수가 필요한 경우는 드묾

```typescript
// AuditLogQuerySchema에 추가
include_total: z.coerce.boolean().default(false).optional().openapi({
  description: '응답에 total 필드를 포함할지 여부 (기본 false, COUNT 쿼리 추가 비용)',
}),
```

### 2. GET /v1/audit-logs 엔드포인트 스펙 (AUDIT-04)

#### 2.1 OpenAPI 정의

```typescript
// packages/daemon/src/api/routes/audit-logs.ts

const auditLogsRoute = createRoute({
  method: 'get',
  path: '/audit-logs',
  tags: ['Admin'],
  summary: 'Query audit logs',
  description:
    'Paginated query of security audit logs with filters. ' +
    'Requires masterAuth (X-Master-Password header). ' +
    'Results are ordered by id DESC (newest first) with cursor-based pagination.',
  request: {
    query: AuditLogQuerySchema,
  },
  responses: {
    200: {
      description: 'Paginated audit log list',
      content: { 'application/json': { schema: AuditLogResponseSchema } },
    },
    400: {
      description: 'Invalid query parameters',
      content: {
        'application/json': {
          schema: z.object({
            code: z.string(),
            message: z.string(),
          }),
        },
      },
    },
    401: {
      description: 'Invalid or missing master password',
      content: {
        'application/json': {
          schema: z.object({
            code: z.literal('INVALID_MASTER_PASSWORD'),
            message: z.string(),
          }),
        },
      },
    },
  },
});
```

#### 2.2 인증

| 항목 | 값 |
|------|-----|
| **미들웨어** | `masterAuth` |
| **헤더** | `X-Master-Password: <plaintext password>` |
| **근거** | 감사 로그는 모든 보안 이벤트를 포함. 세션 토큰으로 조회하면 에이전트가 자신의 활동 기록에 접근 가능 → 보안 위험 |

#### 2.3 에러 코드

| 코드 | HTTP | 조건 | 설명 |
|------|------|------|------|
| `INVALID_MASTER_PASSWORD` | 401 | X-Master-Password 누락 또는 불일치 | 기존 에러 코드 재사용 |
| `VALIDATION_ERROR` | 400 | Zod 검증 실패 | 기존 openApiValidationHook에 의해 자동 처리 |

**INVALID_CURSOR 에러 불필요 결정:**

원래 ROADMAP에서 `INVALID_CURSOR` 에러 코드를 계획했으나, cursor가 단순 정수(id)이므로 Zod의 `z.coerce.number().int().positive()` 검증으로 충분하다. 유효하지만 존재하지 않는 cursor 값 (예: 99999999)은 빈 결과를 반환하면 되므로 별도 에러 불필요.

### 3. 설계 문서 갱신 내용

#### 3.1 설계 문서 25 (sqlite-schema.md) 갱신

**섹션 2.5 audit_log 테이블:**

1. **이벤트 타입 목록 테이블 교체:**
   - 기존 23개 → 20개로 재정의
   - 각 이벤트에 `발생 서비스`, `details 스키마` 컬럼 추가
   - 용어 변경 반영 (agent→wallet): `AGENT_CREATED` → `WALLET_CREATED`, `AGENT_SUSPENDED` → `WALLET_SUSPENDED`

2. **인덱스 추가:**
   ```sql
   CREATE INDEX idx_audit_log_tx_id ON audit_log(tx_id);
   ```
   기존 5개 → 6개 인덱스. `tx_id` 필터 지원.

3. **마이그레이션 노트 추가:**
   - 스키마 변경 없음 (event_type은 TEXT, CHECK 제약 없음 → 새 이벤트 추가에 DDL 변경 불필요)
   - 인덱스 추가는 `CREATE INDEX IF NOT EXISTS`로 무중단 적용 가능
   - 마이그레이션 버전: v34 (현재 v33)

4. **크기 추정 업데이트:**
   - 기존: "tx당 ~3건" → 이벤트 20개로 확대 시에도 모든 이벤트가 매 TX마다 발생하지 않음
   - 예상 변경: "tx당 ~4건" (TX_SUBMITTED + TX_CONFIRMED 추가). 연간 크기 ~400MB → ~440MB 소폭 증가

#### 3.2 설계 문서 29 (api-framework-design.md) 갱신

**cursor pagination 유틸리티 섹션 추가:**

```markdown
### X.X Cursor Pagination 유틸리티

코드베이스에 3가지 cursor pagination 패턴이 존재한다. 공통 유틸리티를 정의한다:

1. **UUID cursor** (transactions): `WHERE id < cursor` (UUID v7 lexicographic)
2. **Composite cursor** (incoming): base64url({detectedAt, id})
3. **Integer cursor** (audit-logs): `WHERE id < cursor` (AUTOINCREMENT)

향후 신규 paginated 엔드포인트는 리소스의 PK 타입에 따라 적절한 패턴을 선택한다.

| PK 타입 | cursor 방식 | 예시 |
|---------|------------|------|
| UUID v7 | UUID 문자열 cursor | transactions, sessions |
| AUTOINCREMENT | 정수 cursor | audit_log |
| 복합 (timestamp + id) | base64url JSON cursor | incoming_transactions |
```

| T-07 | masterAuth 보호 | X-Master-Password 없이 요청 시 401 |

### 5. Admin UI 연동 가이드

audit-logs API는 Admin UI에서도 소비될 수 있다. 구현 시 참고할 UI 연동 설계:

#### 5.1 Admin UI 감사 로그 페이지

```
/admin/audit-logs
├── 필터 바: [이벤트 타입 드롭다운] [심각도 드롭다운] [기간 선택기] [월렛 선택기]
├── 로그 테이블: id | 시각 | 이벤트 | 심각도 | 액터 | 월렛 | 상세
└── 페이지네이션: [이전] [다음] (cursor 기반)
```

#### 5.2 심각도별 시각 표시

| severity | 배지 색상 | 아이콘 |
|---------|----------|--------|
| info | 회색/파랑 | 정보 |
| warning | 노랑/주황 | 경고 |
| critical | 빨강 | 위험 |

#### 5.3 details 렌더링

details 필드는 이벤트 타입에 따라 다른 구조를 가진다. Admin UI에서는:
- 기본: JSON.stringify로 표시
- 향상: eventType별 커스텀 렌더러 (예: TX_CONFIRMED → txHash 링크)

## 검증 기준

1. cursor pagination이 id AUTOINCREMENT 기반으로 설계되어 기본 50건, 최대 200건 제한이 명시된다
2. cursor + 필터 조합의 SQL 쿼리 패턴이 기존 인덱스를 활용하도록 설계된다
3. GET /v1/audit-logs 엔드포인트의 요청/응답/에러 코드가 OpenAPI 수준으로 스펙화된다
4. masterAuth 인증이 명시되고, sessionAuth 접근이 거부되는 보안 모델이 설계된다
5. 설계 문서 4개(25, 29, 32, 37)의 갱신 내용이 구체적으로 명시된다
