**20.1: MaturityMonitor ↔ IYieldProvider 연동 설계**
Phase 269 섹션 10.2의 MaturityMonitor가 IYieldProvider를 통해 yield 포지션을 모니터링하는 전체 데이터 흐름을 명세한다:

- **데이터 흐름:**
  1. PositionTracker가 PendleProvider.getPositions() 호출 (IPositionProvider 인터페이스)
  2. defi_positions 테이블에 category='YIELD' 포지션 저장 (metadata JSON에 YieldMetadataSchema)
  3. MaturityMonitor.evaluate()가 defi_positions에서 category='YIELD', status='ACTIVE' 행 읽기
  4. metadata.maturity 에포크와 현재 시간 비교하여 severity 산정
  5. MATURITY_WARNING 알림 발생 (Phase 269 섹션 11 SSoT 체인)

- **MaturityMonitor가 IYieldProvider에 직접 의존하지 않는 이유:**
  - DEC-MON-03: monitors read from defi_positions cache, never make direct RPC calls
  - MaturityMonitor는 defi_positions.metadata.maturity 값만 필요 (DB에서 읽기)
  - IYieldProvider.getPosition()은 API 응답용, MaturityMonitor는 DB 캐시 사용

- **MaturityMonitor 트리거 조건 매핑 (Phase 269 섹션 10.2 확인):**
  - WARNING (7일 전): daysRemaining <= 7 && daysRemaining > 1
  - DANGER (1일 전): daysRemaining <= 1 && daysRemaining > 0
  - CRITICAL (만기 후 미상환): isExpired && status='ACTIVE'
  - 각 severity가 MaturityInfo.warningLevel과 1:1 매핑됨 (DEC-YIELD-06)

- **만기 후 자동 액션 (미래 확장 가능성만 기술):**
  - 현재 설계: MATURITY_WARNING 알림만 발생, 자동 상환(auto-redeem) 없음
  - 이유: 자동 상환은 높은 복잡도, out of scope (REQUIREMENTS.md)
  - 미래: 정책 기반 자동 상환 가능성은 열어둠 (MaturityMonitor → PolicyEngine → auto-action pipeline)

**20.2: positions 테이블 YIELD 카테고리 확장 검증**
Phase 268 섹션 5.3에서 이미 정의된 YIELD 스키마의 완전성을 검증하고, Pendle 구현에 필요한 추가 필드가 있는지 확인한다:

- **기존 YieldMetadataSchema (섹션 5.3) 검토:**
  - tokenType: 'PT' | 'YT' | 'LP' -- Pendle 3가지 토큰 유형 수용 확인
  - marketId: string -- Pendle market address 저장 충분
  - maturity: number (epoch seconds) -- MaturityMonitor 필수 필드 충족
  - apy: number | null -- 현재 APY 캐시 가능
  - entryPrice: number | null -- 진입 시점 가격 기록 가능

- **Pendle 특화 추가 필드 권장 사항:**
  - syAddress: string (optional) -- SY 토큰 주소 (라우팅 참조용)
  - underlyingAsset: string (optional) -- 기초 자산 CAIP-19 (human-readable context)
  - 이 필드들은 YieldMetadataSchema에 .optional()로 추가 가능 (기존 스키마 하위 호환)
  - 또는 PendleProvider가 metadata JSON에 자유롭게 추가 가능 (Zod passthrough)

- **결론:** Phase 268 YieldMetadataSchema는 핵심 요구사항을 충족함. Pendle 특화 필드는 구현 시(m29-06) metadata JSON에 추가하되, 프레임워크 스키마 변경은 불필요. DEC-YIELD-09로 기록.

- DEC-YIELD-10: MaturityMonitor는 IYieldProvider에 직접 의존하지 않음 (DB 캐시 기반, DEC-MON-03 준수)

**20.3: Pendle V2 프로토콜 매핑 (Router + Hosted SDK → IYieldProvider)**

Protocol mapping 테이블 형식 (Phase 270 섹션 17 동일):

| IYieldProvider 메서드 | Pendle Router 함수 | Pendle Hosted SDK | 비고 |
|---|---|---|---|
| resolve('buy_pt') | IPAllActionV3.swapExactTokenForPt(receiver, market, minPtOut, guessPtOut, input, limit) | GET /core/v2/sdk/{chainId}/convert tokensIn=[underlying] tokensOut=[PT] | Hosted SDK가 routing/optimization 처리 |
| resolve('buy_yt') | IPAllActionV3.swapExactTokenForYt(receiver, market, minYtOut, guessYtOut, input, limit) | GET /core/v2/sdk/{chainId}/convert tokensIn=[underlying] tokensOut=[YT] | YT 가격은 만기 접근 시 0에 수렴 |
| resolve('redeem_pt') | IPAllActionV3.redeemPyToToken(receiver, YT, netPyIn, output) | GET /core/v2/sdk/{chainId}/convert tokensIn=[PT] tokensOut=[underlying] | post-maturity: PT only, pre-maturity: PT+YT |
| resolve('add_liquidity') | IPAllActionV3.addLiquiditySingleToken(receiver, market, minLpOut, guessPtReceivedFromSy, input, limit) | GET /core/v2/sdk/{chainId}/convert tokensIn=[underlying] tokensOut=[LP] | SY 라우팅은 SDK가 자동 처리 |

| getPosition() | -- | GET /core/v1/{chainId}/dashboard/positions/database/{userAddress} | 또는 defi_positions DB 캐시 |
| getYieldForecast() | -- | GET /core/v2/{chainId}/markets/{marketAddress}/data | implied/underlying/fixed APY + PT/YT 가격 |
