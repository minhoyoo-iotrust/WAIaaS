설계 문서를 작성한다. 다음 섹션을 포함해야 한다:

**섹션 1: 개요** — 기존 `POST /v1/actions/:provider/:action` 엔드포인트가 off-chain action 결과도 반환하도록 확장하는 설계. 별도 엔드포인트를 추가하지 않고 기존 엔드포인트를 확장하는 이유 기술.

**섹션 2: REST API 응답 스키마 확장 (PIPE-04)** — 기존 action 실행 응답에 kind별 결과를 포함하는 설계
- 기존 응답: `{ transactionId, txHash, status }` (contractCall)
- 확장 응답 union:
  - contractCall: 기존 응답 그대로 (무변경)
  - signedData: `{ actionId, kind: 'signedData', venue, operation, signature, metadata?, tracking? }`
  - signedHttp: `{ actionId, kind: 'signedHttp', venue, operation, signature, signedHeaders, signatureInput, tracking? }`
- 응답 스키마 Zod 초안
- actionId는 transactions 테이블의 UUID (action_kind로 구분)
- off-chain 결과 조회: `GET /v1/transactions/:id`가 action_kind에 따라 다른 응답 포함

**섹션 3: off-chain action 조회 API** — 기존 transactions 조회 API에서 off-chain 레코드 반환
- `GET /v1/wallets/:walletId/transactions` 목록에 off-chain 레코드 포함
- 필터 파라미터 추가: `?actionKind=signedData|signedHttp|contractCall`
- `GET /v1/transactions/:id` 상세 응답에 venue, operation, external_id 포함
- 기존 on-chain 트랜잭션 응답과의 하위 호환

**섹션 4: MCP 도구 확장 (PIPE-05)**
- 기존 `action-execute` 도구가 off-chain 결과도 반환하도록 확장
  - 응답에 kind, venue, operation 필드 추가
  - 에이전트가 결과의 kind를 보고 on-chain/off-chain 구분 가능
- 신규 도구 `action-list-offchain`: off-chain action 이력 조회
  - 파라미터: walletId, venue?, limit?, offset?
  - 반환: off-chain action 목록 (actionId, kind, venue, operation, status, createdAt)
- 기존 credential MCP 도구(4개)는 Phase 381에서 설계 완료, 여기서는 참조만

**섹션 5: SDK 메서드 확장 (PIPE-05)**
- 기존 `executeAction(provider, action, params)` 반환 타입을 union으로 확장
  - `ActionResult = ContractCallResult | SignedDataResult | SignedHttpResult`
  - kind 필드로 구분
- 신규 메서드:
  - `listOffchainActions(walletId, options?)`: off-chain action 이력 조회
  - `getActionResult(actionId)`: 특정 action 결과 조회 (on-chain/off-chain 모두)
- 기존 credential SDK 메서드(4개)는 Phase 381에서 설계 완료, 여기서는 참조만
- TypeScript 타입 정의 초안

**섹션 6: connect-info 확장** — off-chain action 지원 여부 노출
- capabilities.externalActions: true
- capabilities.supportedVenues: string[] (등록된 ActionProvider의 venue 목록)
- capabilities.signing은 Phase 382에서 이미 설계됨 (참조)

**섹션 7: 인증 모델** — off-chain action 실행 시 인증 경로
- action 실행: sessionAuth (기존과 동일)
- credential 관리: masterAuth (Phase 381 설계 참조)
- 정리: 어떤 작업에 어떤 인증이 필요한지 매트릭스
