**1. TransactionParam 확장 설계 (PLCY-01)**
- 기존 TransactionParam 인터페이스에 off-chain action 전용 필드 추가:
  - `venue?: string` — 외부 서비스 식별자 (signedData/signedHttp에서 전달)
  - `actionCategory?: 'trade' | 'withdraw' | 'transfer' | 'sign' | 'deposit'` — Phase 380 policyContext에서 정의한 5종 카테고리
  - `notionalUsd?: number` — 추정 USD 가치 (정책 한도 평가용)
  - `leverage?: number` — 레버리지 배율 (perp 관련, 기존 perpLeverage와 통합 방향 문서화)
  - `expiry?: string` — 주문 만료 시각 (ISO 8601)
  - `hasWithdrawCapability?: boolean` — venue에서 자금 인출 가능 여부 (위험도 평가용)
- Phase 383 설계의 `toPolicyParam()` 함수가 이 확장 필드를 사용하는 흐름 문서화
- 기존 perpLeverage/perpSizeUsd와 leverage/notionalUsd 간 관계 정리: off-chain perp는 leverage/notionalUsd 사용, on-chain perp는 기존 perpLeverage/perpSizeUsd 유지 (하위 호환)

**2. VENUE_WHITELIST 정책 설계 (PLCY-02)**
- 기존 CONTRACT_WHITELIST 패턴을 그대로 활용하는 default-deny 화이트리스트:
  - DB policies 테이블의 `policy_type` 컬럼에 'VENUE_WHITELIST' 값 추가
  - `VenueWhitelistRules` 인터페이스: `{ venues: Array<{ id: string; name?: string }> }` — CONTRACT_WHITELIST의 `contracts` 패턴과 동일
  - venue id 매칭 규칙: 대소문자 무관 (toLowerCase 정규화)
  - 정책 평가 알고리즘: action.venue가 whitelist에 없으면 DENY (CONTRACT_WHITELIST 4c 단계와 동일 위치)
  - 평가 순서: 기존 정책 평가 체인에 4j 단계로 추가 (4i-c PERP_MAX_POSITION_USD 다음)
  - network scoping: VENUE_WHITELIST는 network 필드 무시 (venue는 네트워크 독립적)
  - contractCall에서는 venue가 null이므로 항상 통과 (VENUE_WHITELIST가 off-chain 전용)
- Zod 스키마 초안 (VenueWhitelistRulesSchema)
- Admin UI 정책 등록 UX: 기존 CONTRACT_WHITELIST 폼 패턴 재사용 (venues 목록, id+name)
- Admin Settings 키: `policy.venue_whitelist_enabled` (boolean, default false)

**3. ACTION_CATEGORY_LIMIT 정책 설계 (PLCY-03)**
- SPENDING_LIMIT의 카테고리별 확장:
  - DB policies 테이블의 `policy_type` 컬럼에 'ACTION_CATEGORY_LIMIT' 값 추가
  - `ActionCategoryLimitRules` 인터페이스:
    ```
    {
      category: 'trade' | 'withdraw' | 'transfer' | 'sign' | 'deposit',
      daily_limit_usd?: number,     // 24시간 누적 USD 한도
      monthly_limit_usd?: number,   // 30일 누적 USD 한도
      per_action_limit_usd?: number, // 단건 USD 한도
      tier_on_exceed: PolicyTier,   // 한도 초과 시 적용 tier (DELAY/APPROVAL)
    }
    ```
  - 누적 금액 계산: transactions 테이블에서 action_kind='signedData'|'signedHttp' AND venue 필터 AND actionCategory 필터로 기간 내 notionalUsd 합산
  - 평가 순서: 4k 단계 (VENUE_WHITELIST 4j 다음)
  - contractCall에서는 actionCategory가 null이므로 항상 통과
  - SPENDING_LIMIT과의 독립성: ACTION_CATEGORY_LIMIT는 off-chain 전용, SPENDING_LIMIT는 on-chain 전용 (중복 평가 없음)
- 누적 합산 쿼리 설계 (transactions 테이블 + 기간 필터 + metadata.notionalUsd 저장 위치 결정)
- notionalUsd 저장: transactions 테이블 metadata JSON 내 `notionalUsd` 필드 또는 별도 컬럼? -> metadata JSON 내에 저장 (스키마 변경 최소화)
- reservedAmount TOCTOU 방지: 기존 evaluateAndReserve() BEGIN IMMEDIATE 패턴 재사용
- Zod 스키마 초안 (ActionCategoryLimitRulesSchema)

**4. ActionDefinition riskLevel 확장 설계 (PLCY-04)**
- 기존 ActionDefinition 인터페이스에 off-chain action 위험도 필드 추가:
  - `riskLevel?: 'low' | 'medium' | 'high' | 'critical'` — 위험도 등급
  - `defaultTier?: PolicyTier` — 정책 미설정 시 기본 tier (INSTANT/NOTIFY/DELAY/APPROVAL)
  - `requiresVenueWhitelist?: boolean` — VENUE_WHITELIST 정책 필수 여부
  - `requiresOwnerApproval?: boolean` — Owner 승인 필수 여부
- riskLevel -> defaultTier 자동 매핑 규칙:
  - low -> INSTANT, medium -> NOTIFY, high -> DELAY, critical -> APPROVAL
- ActionProvider.getActions()에서 ActionDefinition 반환 시 riskLevel 포함
- 정책 미설정 시 defaultTier 적용 경로
