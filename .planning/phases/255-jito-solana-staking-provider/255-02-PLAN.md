Add a new entry in the `providers` array inside `registerBuiltInProviders()`:

```typescript
{
  key: 'jito_staking',
  enabledKey: 'actions.jito_staking_enabled',
  factory: () => {
    // Jito is mainnet-only -- no testnet pool exists.
    // Read admin override addresses; empty string falls back to mainnet defaults.
    const stakePoolOverride = settingsReader.get('actions.jito_staking_stake_pool_address');
    const jitosolMintOverride = settingsReader.get('actions.jito_staking_jitosol_mint');

    const addresses = getJitoAddresses('mainnet');

    const config: JitoStakingConfig = {
      enabled: true,
      stakePoolAddress: stakePoolOverride || addresses.stakePoolAddress,
      jitosolMint: jitosolMintOverride || addresses.jitosolMint,
      stakePoolProgram: addresses.stakePoolProgram,
    };
    return new JitoStakingActionProvider(config);
  },
},
```

Place the jito_staking entry AFTER the lido_staking entry in the providers array.

**Test cases (9 minimum):**

1. **registers jito_staking when enabled=true**
   - Create registry, enable jito_staking, call registerBuiltInProviders
   - Expect loaded contains 'jito_staking', registry.getProvider('jito_staking') defined
   - Expect registry.listActions('jito_staking') has length 2

2. **skips jito_staking when enabled=false**
   - Expect skipped contains 'jito_staking', loaded does not
   - registry.getProvider('jito_staking') undefined

3. **uses custom stake_pool_address when set via Admin Settings**
   - Provide custom address in settings reader
   - executeResolve('jito_staking/stake', ...) -> result[0].to equals custom address

4. **uses default mainnet address when override is empty**
   - Enable with empty overrides
   - executeResolve('jito_staking/stake', ...) -> result[0].to equals JITO_MAINNET_ADDRESSES.stakePoolAddress

5. **provider-trust auto-tags with actionProvider=jito_staking**
   - executeResolve('jito_staking/stake', ...) -> all results have actionProvider='jito_staking'

6. **stake returns ContractCallRequest with Solana fields**
   - executeResolve('jito_staking/stake', { amount: '1.0' }, ctx)
   - Expect results[0].type == 'CONTRACT_CALL'
   - Expect results[0].programId to be defined (SPL Stake Pool program)
   - Expect results[0].instructionData to be defined (base64)
   - Expect results[0].accounts to be defined and length > 0

7. **unstake returns ContractCallRequest with Solana fields**
   - executeResolve('jito_staking/unstake', { amount: '1.0' }, ctx)
   - Same field checks as stake
   - instructionData should differ from stake (different instruction index)

8. **stake instructionData encodes correct amount in lamports**
   - Stake with amount '2.0' -> decode base64 instructionData
   - First byte should be 14 (DepositSol index)
   - Bytes 1-8 as LE u64 should be 2000000000n (2 SOL = 2 * 10^9 lamports)

9. **INSUFFICIENT_BALANCE error propagates for SOL shortfall (JITO-04)**

   - The pipeline's Stage 3 balance check should throw ChainError with code 'INSUFFICIENT_BALANCE'
   - If the integration test setup does not go through the full pipeline (i.e., only tests registration + provider resolve), verify at minimum that: (a) the provider faithfully encodes the large amount in the ContractCallRequest's instructionData, and (b) document that the actual INSUFFICIENT_BALANCE enforcement happens in pipeline Stage 3 (SolanaAdapter.getBalance vs requested amount). The key point is that the amount is correctly passed through so the pipeline CAN enforce the balance check.
