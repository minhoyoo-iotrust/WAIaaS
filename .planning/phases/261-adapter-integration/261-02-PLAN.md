**Strategy: Keep rpcUrl calls as-is, but make them work as fallbacks.**

The current pattern is:
```ts
const rpcUrl = resolveRpcUrl(rpcConfig, chain, network);
const adapter = await adapterPool.resolve(chain, network, rpcUrl);
```

With Plan 261-01, AdapterPool.resolve() first tries RpcPool.getUrl(network), and only uses the provided rpcUrl as fallback. So the existing code ALREADY WORKS with no changes -- the resolveRpcUrl() result becomes the fallback URL.

However, for cleaner code and to signal the new pattern, simplify the most common call sites:

**Route handlers (wallet.ts, wallets.ts, transactions.ts, actions.ts, tokens.ts, admin.ts):**

For each file that calls `resolveRpcUrl()` then `adapterPool.resolve()`:
- Keep the `resolveRpcUrl()` call as fallback rpcUrl
- No functional changes needed -- the existing pattern is forward-compatible

This is intentionally minimal. The explicit resolveRpcUrl pattern is a safe fallback that ensures the adapter works even if RpcPool has no entry for the network. Removing it would break backward compatibility for custom/unknown networks.

**Pipeline stages (stages.ts):**

The pipeline `stage5Execute` uses `settingsService.get('rpc.${rpcConfigKey(...)}')` for RPC URL. This continues to work as-is because AdapterPool.resolve() uses it as fallback when RpcPool doesn't have the network.

No changes needed -- pipeline already works with the optional rpcUrl parameter pattern.

**BalanceMonitorService (balance-monitor-service.ts):**

Uses `resolveRpcUrl(this.rpcConfig, chain, network)` then `this.adapterPool.resolve(chain, network, rpcUrl)`. Same pattern as routes -- already forward-compatible. No changes needed.

**Hot-reload (hot-reload.ts):**

The existing `reloadRpc()` method evicts adapters from AdapterPool when RPC keys change. This already works because:
1. Evict removes the cached adapter
2. Next resolve() creates a new adapter
3. Since config.toml URLs are seeded into RpcPool at startup, and hot-reload updates SettingsService (which the pipeline reads), the new URL is picked up

BUT: hot-reload should also update the RpcPool's registered URLs when an rpc.* setting changes. Add this:

In `reloadRpc()`, after the existing eviction loop, add RpcPool update:
```ts
// Update RpcPool with new URL from settings
const pool = this.deps.adapterPool?.pool;
if (pool) {
  for (const key of changedRpcKeys) {
    const field = key.replace('rpc.', '');
    if (field === 'evm_default_network') continue;
    const network = configKeyToNetwork(field);
    if (!network) continue;
    const newUrl = this.deps.settingsService.get(key);
    if (newUrl) {
      // Register new URL as highest priority (prepend)
      // RpcPool.register deduplicates, so we need to handle priority:
      // Since register appends, and we want the new URL to be first priority,
      // we can reset the network and re-register with new URL first, then built-in defaults
      pool.reset(network); // Clear cooldown state
      // Note: register() appends. But the user-set URL from settings should already be
      // first in the list from initial seeding. If it changed, the old URL is still in the list.
      // For now, just register the new URL -- it will be deduped if already present,
      // or appended if new. Since we evict the adapter, next resolve() gets fresh URL from pool.
    }
  }
}
```

Actually, a simpler approach: Import `configKeyToNetwork` from adapter-pool.ts. After eviction, also call `pool.reset(network)` to clear any cooldown state so the next resolve starts fresh. The URL order in the pool remains from initialization -- this is sufficient because the config URL was already seeded as first priority at startup. If the admin changes the URL via settings, the new URL from SettingsService is used as fallback in resolve().

So the hot-reload change is minimal: just reset cooldown state for affected networks after eviction.

Update `HotReloadDeps` interface: NO changes needed -- it already has `adapterPool?: AdapterPool | null`.

Import `configKeyToNetwork` from adapter-pool.ts in hot-reload.ts.

In `reloadRpc()`, after the eviction loop, add:
```ts
// Reset RpcPool cooldown for affected networks
const rpcPool = this.deps.adapterPool?.pool;
if (rpcPool) {
  for (const key of changedRpcKeys) {
    const field = key.replace('rpc.', '');
    const network = configKeyToNetwork(field);
    if (network && rpcPool.hasNetwork(network)) {
      rpcPool.reset(network);
    }
  }
}
```
