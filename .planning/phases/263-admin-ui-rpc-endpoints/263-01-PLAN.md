## Backend: GET /admin/rpc-status endpoint

1. **OpenAPI schemas** in `openapi-schemas.ts`: Add `RpcStatusResponseSchema` -- a Zod schema for the response:
   ```ts
   // Response shape: { networks: Record<string, RpcEndpointStatus[]> }
   // where RpcEndpointStatus = { url: string, status: 'available'|'cooldown', failureCount: number, cooldownRemainingMs: number }
   ```

2. **Route definition** in `admin.ts`: Add `rpcStatusRoute` using createRoute:
   - Method: GET, Path: `/admin/rpc-status`, Tags: ['Admin']
   - Response 200 schema: `RpcStatusResponseSchema`

3. **Handler** in `admin.ts` createAdminRoutes:
   - Add `rpcPool?: RpcPool` to the deps type (import RpcPool from @waiaas/core/rpc)
   - In the handler: iterate `deps.rpcPool?.getNetworks()`, call `deps.rpcPool.getStatus(network)` for each, return `{ networks: { [network]: status[] } }`
   - If rpcPool is not available, return `{ networks: {} }`
   - This endpoint uses masterAuth (same as all /admin/* endpoints)

4. **Wire RpcPool** into admin deps: Find where createAdminRoutes is called in the daemon setup and pass the rpcPool instance. Check `packages/daemon/src/daemon.ts` or equivalent bootstrap file for the wiring point.

**How data flows on load:**
1. `GET /admin/settings` returns `rpc_pool` category with per-network JSON arrays (user URLs)
2. For each network, merge: user URLs (from settings) + built-in defaults (from const)
3. Mark each URL as `isBuiltin: boolean` based on presence in `BUILT_IN_RPC_URLS`
4. Display merged list in priority order

**How data flows on save:**
1. Collect the full ordered URL list per network (both user and built-in)
2. Filter out disabled built-in URLs from the save payload
3. Send only user-added URLs to `PUT /admin/settings` as `rpc_pool.{network}` JSON arrays
4. Or: send the complete intended list and let the server-side hot-reload merge handle it
   - Actually, per 262-01 decision: rpc_pool.* stores user URLs only. Built-in defaults are merged server-side.
   - So save: filter out built-in URLs from the ordered list, save remaining as JSON array
   - Reorder that includes built-in URLs: the overall priority is determined by the order of user URLs relative to built-in defaults (which are always appended at the end during hot-reload merge)
   - Simplification: User URLs appear at top (highest priority), built-in at bottom. User can reorder within user URLs. Built-in order is fixed.
