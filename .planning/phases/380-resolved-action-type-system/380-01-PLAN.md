설계 문서를 작성한다. 한국어로 작성. 다음 섹션을 포함:

**1. 개요**: Phase 380 목표 요약, 기존 IActionProvider.resolve() 반환 타입 한계

**2. ResolvedAction Zod discriminatedUnion 설계**:
- `kind` 필드로 분기하는 3종 union: `contractCall | signedData | signedHttp`
- Zod 스키마 초안 (TypeScript 코드 블록):
  - `SignedDataActionSchema`: kind: 'signedData', signingScheme(SigningSchemeEnum), payload(Record or Buffer-like), venue(string), operation(string), credentialRef?(string), tracking?(object with trackerName/metadata), policyContext?(object with actionCategory/notionalUsd/leverage/expiry/hasWithdrawCapability)
  - `SignedHttpActionSchema`: kind: 'signedHttp', method(string), url(string), headers(Record), body?(string), venue(string), operation(string), credentialRef?(string), signingScheme(erc8128 | hmac-sha256 | rsa-pss 등), coveredComponents?(string[]), preset?('minimal'|'standard'|'strict'), ttlSec?(number), nonce?(string|false). 기존 SignHttpMessageParams 필드를 통합하되 privateKey/chainId/address는 제외(ActionContext에서 주입)
  - `ContractCallRequestSchema` 확장: 기존 스키마에 `kind?: 'contractCall'` optional 필드 추가. z.literal('CONTRACT_CALL')인 type 필드와 별개로 kind 필드는 파이프라인 라우팅용
  - `ResolvedActionSchema`: z.discriminatedUnion('kind', [...]) — 정규화 후 사용. 정규화 전 raw 반환값은 별도 `RawResolvedAction` 타입으로 표현 (kind 없는 ContractCallRequest 허용)

**3. 정규화 전략**:
- ActionProviderRegistry에서 resolve() 결과를 정규화하는 함수 `normalizeResolvedAction()` 설계
- kind 필드가 없고 type === 'CONTRACT_CALL'이면 kind: 'contractCall' 추가
- kind 필드가 없고 __apiDirect === true이면 기존 ApiDirectResult 경로 유지 (ResolvedAction 밖)
- 정규화 후에는 반드시 kind가 존재하므로 discriminatedUnion 파싱 가능

**4. IActionProvider.resolve() 반환 타입 확장**:
- 기존: `ContractCallRequest | ContractCallRequest[] | ApiDirectResult`
- 확장: `ContractCallRequest | ContractCallRequest[] | ApiDirectResult | SignedDataAction | SignedHttpAction`
- 또는 더 깔끔하게: `ResolvedAction | ResolvedAction[] | ApiDirectResult` (ContractCallRequest는 ResolvedAction의 kind:'contractCall' 멤버)
- 하위 호환: 기존 provider는 ContractCallRequest를 반환 → normalizeResolvedAction()이 kind 추가

**5. 하위 호환 분석**:
- 기존 13개 ActionProvider 목록 (Jupiter, 0x, LI.FI, Lido, Jito, AaveV3, Kamino, Pendle, Drift, DCent, Hyperliquid, Across, Polymarket)
- 각 provider의 resolve() 반환 타입 현황 (ContractCallRequest 또는 ApiDirectResult)
- kind 없이 반환해도 registry 정규화로 기존 경로 100% 유지됨을 명시
- 주의사항: ContractCallRequest[]는 multi-step 시나리오, 각 요소에 kind 추가

**6. ApiDirectResult와의 관계**:
- ApiDirectResult는 ResolvedAction union에 포함하지 않음 (별도 판별)
- isApiDirectResult() 타입 가드로 먼저 분기 후, 나머지를 ResolvedAction으로 처리
- 향후 ApiDirectResult를 SignedDataAction으로 마이그레이션 가능성 언급

**7. Pitfall 방지 체크리스트**:
- [ ] kind 필드와 기존 type 필드 혼동 금지 (type은 트랜잭션 유형, kind는 파이프라인 라우팅)
- [ ] SignedDataAction.payload는 서명 방식에 따라 다른 구조 (EIP-712 → domain+types+value, HMAC → raw bytes)
- [ ] credentialRef는 optional — 글로벌 credential fallback 가능
- [ ] tracking은 optional — 동기 완료 액션은 tracking 불필요
- [ ] policyContext는 optional — 정책 평가에 필요한 추가 컨텍스트
