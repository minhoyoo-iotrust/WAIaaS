설계 문서를 작성한다. 한국어로 작성. 다음 섹션을 포함:

**1. 개요**: 기존 4종 서명 모듈이 분산되어 있는 문제, ISignerCapability로 통합하는 목표

**2. SigningSchemeEnum 설계**:
- Zod enum 초안: `z.enum(['eip712', 'personal', 'hmac-sha256', 'rsa-pss', 'ecdsa-secp256k1', 'ed25519', 'erc8128'])`
- 각 scheme 설명:
  - `eip712`: EIP-712 typed structured data (domain + types + value). CoW Protocol, Polymarket CLOB 등
  - `personal`: EIP-191 personal sign (arbitrary message)
  - `hmac-sha256`: HMAC-SHA256 (CEX API 인증, Binance/OKX 등)
  - `rsa-pss`: RSA-PSS (일부 금융 API, 레거시 시스템)
  - `ecdsa-secp256k1`: raw ECDSA secp256k1 (arbitrary bytes signing)
  - `ed25519`: Ed25519 (Solana native, 크로스체인 메시지 등)
  - `erc8128`: ERC-8128 signed HTTP request (RFC 9421 + EIP-191)
- 확장 가능성: 향후 `hmac-sha512`, `ecdsa-p256` 등 추가 가능

**3. ISignerCapability 인터페이스 설계**:
- TypeScript 인터페이스 초안:
  ```typescript
  interface ISignerCapability {
    readonly scheme: SigningScheme;
    canSign(params: SigningParams): boolean;
    sign(params: SigningParams): Promise<SigningResult>;
  }
  ```
- `SigningParams` 타입: scheme에 따라 다른 구조 (discriminated union or generic)
  - eip712: { domain, types, value, privateKey }
  - personal: { message, privateKey }
  - hmac-sha256: { data, secret } (credential에서 주입)
  - rsa-pss: { data, privateKey } (credential에서 주입)
  - ecdsa-secp256k1: { data, privateKey }
  - ed25519: { data, privateKey }
  - erc8128: { method, url, headers, body?, privateKey, chainId, address, ... }
- `SigningResult` 타입: { signature: string | Uint8Array, metadata?: Record<string, unknown> }
- `canSign()`: 해당 capability가 주어진 params를 서명할 수 있는지 검사 (키 타입 호환성 등)

**4. 기존 4종 signer ISignerCapability 어댑터 개요**:
- Phase 382에서 상세 설계하지만, 여기서는 매핑 관계만 정리:
  - `Eip712SignerCapability` (scheme: eip712) — sign-message.ts typedData 래핑
  - `PersonalSignCapability` (scheme: personal) — sign-message.ts personal 래핑
  - `Erc8128SignerCapability` (scheme: erc8128) — http-message-signer.ts 래핑
  - `TransactionSignerCapability` — IChainAdapter.signTransaction() 래핑. 단, contractCall은 기존 6-stage pipeline 그대로 사용하므로 ISignerCapability 경로 사용 안 함. 향후 확장용
- 신규 capability (Phase 382 상세 설계):
  - `HmacSignerCapability` (scheme: hmac-sha256) — node:crypto createHmac
  - `RsaPssSignerCapability` (scheme: rsa-pss) — node:crypto sign with RSA-PSS
  - `signBytes` capability (scheme: ecdsa-secp256k1 / ed25519) — arbitrary bytes signing

**5. SignerCapabilityRegistry 개요**:
- Phase 382에서 상세 설계하지만, 여기서는 인터페이스만 정리:
  - `register(capability: ISignerCapability): void`
  - `get(scheme: SigningScheme): ISignerCapability | undefined`
  - `resolve(action: SignedDataAction | SignedHttpAction): ISignerCapability` — action의 signingScheme으로 capability 조회
- 초기 등록: daemon 부팅 시 기존 4종 + 신규 2종 자동 등록

**6. 기존 파이프라인 무변경 원칙**:
- sign-message.ts / sign-only pipeline / ERC-8128 모듈의 기존 호출 경로는 절대 변경하지 않음
- ISignerCapability는 새 ActionProvider 경로(SignedDataAction/SignedHttpAction 처리)에서만 사용
- 기존 경로와 새 경로의 관계 다이어그램

**7. 설계 결정 사항**:
- SigningParams를 scheme별 discriminated union으로 할지, generic object로 할지 → discriminated union 권장 (타입 안전성)
- credential 주입 시점: resolve() 시점이 아니라 sign() 직전에 CredentialVault에서 주입
- 키 메모리 관리: sign() 완료 후 즉시 클리어 (기존 ActionContext.privateKey 패턴 동일)
