Across Protocol 공식 문서(https://docs.across.to)를 리서치하여 설계 문서 79의 전반부를 작성한다. 기존 프로젝트 리서치 파일(.planning/research/m31-06-across-*)을 주요 입력으로 활용하되, 필요 시 공식 문서에서 누락 정보를 보완한다.

설계 문서 79는 기존 design doc 77(DCent Swap Aggregator)의 형식을 따르되, 다음 섹션을 포함해야 한다:

**1. 개요 및 목표** (doc 79 헤더)
- `# 설계 문서 79: Across Protocol 크로스체인 브릿지`
- WAIaaS v31.6 마일스톤 설명, Intent 기반 브릿지 개요

**2. Across API 엔드포인트 분석** (DES-01)
- 5개 엔드포인트별 상세 사양:
  - GET /suggested-fees: inputToken, outputToken, originChainId, destinationChainId, amount, recipient, message 파라미터. 캐싱 금지 명시.
  - GET /limits: inputToken, outputToken, originChainId, destinationChainId 파라미터. minDeposit, maxDeposit, maxDepositInstant, maxDepositShortDelay 반환.
  - GET /available-routes: originChainId?, destinationChainId?, originToken?, destinationToken? 필터. 5분 캐시 가능.
  - GET /deposit/status: depositTxnRef(txHash) 파라미터. filled/pending/expired/refunded 4-state 반환.
  - GET /swap/approval: 통합 API (approvalTxns + swapTx calldata 반환). 캐싱 금지.
- 각 엔드포인트의 요청/응답 예시(JSON)
- 에러 응답 형식

**3. Zod 스키마 설계** (DES-01 일부)
- 각 API 응답에 대한 Zod 스키마 코드 블록:
  - AcrossSuggestedFeesResponseSchema
  - AcrossLimitsResponseSchema
  - AcrossAvailableRoutesResponseSchema
  - AcrossDepositStatusResponseSchema
  - AcrossSwapApprovalResponseSchema
- .passthrough() 사용하여 확장 필드 허용
- 기존 LI.FI schemas.ts 패턴 참조

**4. SpokePool depositV3 컨트랙트 인터페이스** (DES-02)
- depositV3() 12개 파라미터 상세 설명 (타입, 소스, 주의사항)
- Solidity function signature
- viem ABI fragment (encodeFunctionData 용)
- 체인별 SpokePool 주소 테이블 (Ethereum, Arbitrum, Optimism, Base, Polygon, Linea)
- 체인별 WETH 주소 테이블 (네이티브 토큰 브릿지 용)
- V3FundsDeposited 이벤트 ABI (depositId 추출 용)

**5. 수수료 모델 및 outputAmount 계산** (DES-03)
- totalRelayFee = lpFee + relayerCapitalFee + relayerGasFee 구조
- outputAmount = inputAmount - totalRelayFee.total 공식 확정
- pct 값 형식 (wei 기반 18자리 소수?) 정확한 변환 방법
- isAmountTooLow 플래그를 활용한 사전 검증
- Zod refine()으로 outputAmount <= inputAmount 런타임 검증

**6. 브릿지 상태 추적 방식** (DES-04)
- 기존 bridge_status/bridge_metadata 컬럼 재사용 확정
- bridge_metadata JSON 구조: { tracker, txHash, originChainId, destChainId, inputToken, outputToken, depositId?, fillTxnRef?, notificationEvent, enrolledAt }
- LI.FI 선례: BridgeStatusTracker 2-phase polling 패턴 재사용
- Across 최적화: Phase 1 폴링 간격 15초 (LI.FI 30초 대비 단축, Across fills 2-10초)
- status 값 매핑: filled->COMPLETED, pending->PENDING, expired->FAILED, refunded->COMPLETED(refunded:true)
- DB 마이그레이션 불필요 확정 및 근거

**7. fillDeadline / exclusivityDeadline 기본값 전략** (DES-05)
- fillDeadline: /suggested-fees API 반환값 우선 사용. API 미반환 시 quoteTimestamp + Admin Settings 기본값(21600초 = 6시간)
- exclusivityDeadline: /suggested-fees API 반환값 그대로 사용. exclusiveRelayer=0x0이면 exclusivityDeadline=0
- Admin Settings 오버라이드 키: actions.across_fill_deadline_buffer_sec
- quoteTimestamp 만료 방지: Stage 5 실행 직전 fresh quote 재조회 (late-bind 패턴)
- uint32 초 단위 통일 (`Math.floor(Date.now() / 1000)`)

**8. AcrossBridgeActionProvider 인터페이스 설계** (DES-06)
- IActionProvider 구현, 5 actions: quote, execute, status, routes, limits
- ActionProviderMetadata: name='across_bridge', chains=['ethereum'], mcpExpose=true
- 각 action의 InputSchema (Zod) 정의
- resolve() 메서드 분기 로직 (actionName별)
- execute action의 approve + depositV3 BATCH 플로우
- 네이티브 토큰 vs ERC-20 분기 로직
- AcrossConfig 타입 + 기본값 + 체인 매핑
- WETH 주소 매핑 테이블

**9. MCP / SDK 설계** (DES-06 일부)
- MCP 도구 4개: across-bridge-quote, across-bridge-execute, across-bridge-status, across-bridge-routes
- SDK 메서드 4개: acrossBridgeQuote, acrossBridgeExecute, acrossBridgeStatus, acrossBridgeRoutes
- 요청/응답 타입 정의
- connect-info capability: across_bridge: true

**10. Admin Settings 설계**
- 6개 키: across_enabled, across_api_base_url, across_integrator_id, across_fill_deadline_buffer_sec, across_default_slippage_pct, across_max_slippage_pct
- 각 키의 기본값, 타입, 설명
- Admin UI 표시 전략 (기존 DeFi Settings 패턴)

**11. 파이프라인 통합 전략**
- Quote -> Approve -> DepositV3 플로우의 6-stage 파이프라인 매핑
- Stage 3 정책 평가: inputAmount 기준
- Stage 4 Owner 승인: 출발/도착 체인, 금액, 수수료 표시
- Stage 5 실행: late-bind quote (fresh /suggested-fees 재조회)
- Post-execution: bridge_status enrollment
- CONTRACT_WHITELIST에 SpokePool 주소 등록 안내

**12. 에러 핸들링**
- insufficient liquidity -> INSUFFICIENT_LIQUIDITY ChainError
- unsupported route -> UNSUPPORTED_ROUTE ChainError
- quote expired -> QUOTE_EXPIRED ChainError (자동 재조회 시도)
- fillDeadline passed -> 자금 반환 대기 안내
- 각 에러의 ChainError 코드, 메시지, 사용자 표시 메시지

**13. 보안 고려사항**
- approve 금액 = inputAmount (최소 권한, MaxUint256 금지)
- Smart Account recipient는 WETH 수신 경고
- CONTRACT_WHITELIST에 SpokePool 주소 등록 필수
- fillDeadline 과거 시간 검증 (on-chain revert 방지)

**14. 설계 결정 요약**
- DS-XX 형식으로 번호 부여. 예:
  - DS-01: REST API 직접 호출 (SDK 사용 않음)
  - DS-02: bridge_status/bridge_metadata 기존 컬럼 재사용 (DB 마이그레이션 없음)
  - DS-03: /suggested-fees 캐싱 금지, /available-routes 5분 캐시 허용
  - DS-04: late-bind quote 패턴 (Stage 5 직전 fresh quote)
  - 등등
