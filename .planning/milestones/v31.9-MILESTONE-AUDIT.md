## 5. Detailed Findings

### FIND-01: Resolution Notification Callback Not Wired in Daemon (SETL-04 PARTIAL)

**Severity:** MEDIUM
**Location:** `packages/daemon/src/lifecycle/daemon.ts` lines 1264-1269
**Requirement affected:** SETL-04 (notification when market resolves)

The `createPolymarketInfrastructure()` factory accepts an optional 5th parameter `emitNotification`, but daemon.ts calls it with only 4 arguments:

```typescript
// daemon.ts:1264-1269
const pmInfra = createPolymarketInfrastructure(
  {},
  { apiKeys: apiKeyDbAdapter as never, orders: orderDbAdapter, positions: positionDbAdapter as never },
  encryptFn,
  decryptFn,
  // <-- 5th arg emitNotification is MISSING
);
```

The `ResolutionMonitor` constructor receives `undefined` for `emitNotification`, so `this.emitNotification` is never set. When `resolution-monitor.ts:101` checks `if (this.emitNotification)`, it always evaluates to false. Resolution detection works (marks resolved in DB), but no notification is ever emitted to the user.

**Impact:** SETL-04 states "User receives notification when market resolves (auto-redeem suggestion included)". The notification pathway exists structurally but is disconnected at the daemon wiring level. The ResolutionMonitor unit tests pass because they mock the callback, but the real daemon runtime never fires it.

**Fix:** Pass an `emitNotification` callback in daemon.ts that dispatches to the existing notification infrastructure (e.g., `walletNotificationChannel.emit()`).

### FIND-02: Python SDK Missing Polymarket Methods (INTG-03 PARTIAL)

**Severity:** LOW
**Location:** `python-sdk/` (no Polymarket code found)
**Requirement affected:** INTG-03 ("TypeScript/Python SDK methods added")

The TypeScript SDK (`packages/sdk/src/client.ts`) has 15 Polymarket methods (8 action + 7 query). The Python SDK (`python-sdk/`) has zero Polymarket references. INTG-03 claims both SDKs are covered, but only TypeScript is implemented.

This is a known pattern in the project -- the Python SDK typically lags behind TypeScript SDK additions. The requirement text says "TypeScript/Python SDK" but only TypeScript was actually delivered in Phase 373-02.

### FIND-03: CLOB-08 (FAK) and CLOB-09 (Batch) Not Implemented

**Severity:** LOW (acknowledged)
**Location:** Requirements marked as Pending in REQUIREMENTS.md
**Requirements affected:** CLOB-08, CLOB-09

No FAK order type handling or batch order endpoint exists in the codebase. The schemas support GTC, GTD, FOK order types but not FAK. No batch submission API is present in ClobClient.

These are correctly tracked as Pending in REQUIREMENTS.md. The ROADMAP Phase 371 success criteria mention "GTC/GTD/FOK/FAK orders" but the plan summaries only claim CLOB-01 through CLOB-07 (GTC, FOK, GTD) as completed.
