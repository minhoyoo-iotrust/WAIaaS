DCent Swap Backend API(`https://agent-swap.dcentwallet.com`)의 7개 엔드포인트를 실제 호출하여 응답 구조를 분석하고 문서화한다. `internal/design/77-dcent-swap-aggregator.md` 파일을 생성하여 다음 섹션을 작성한다.

**섹션 1: API 엔드포인트 분석** — 7개 엔드포인트 각각에 대해:
- HTTP 메서드 + URL
- 요청 파라미터 (body/query)
- 응답 JSON 구조 (실제 호출 결과 기반)
- 에러 응답 패턴

실제 API 호출로 확인할 항목:
1. `GET /api/swap/v3/get_supported_currencies` — 전체 응답 구조, currencies 배열 내 필드 (id, name, symbol, network, contractAddress, decimals 등)
2. `GET /api/swap/v3/is_supported_by_currency_id?currencyId=ETHEREUM` — 응답 구조
3. `POST /api/swap/v3/get_quotes` — body: `{ from: "ETHEREUM", to: "ERC20/0xa0b86991c6218b36c1d19d4a2e9eb0ce3606eb48", amount: "1000000000000000000", fromDecimals: 18, toDecimals: 6 }` → providers.bestOrder 구조, 각 프로바이더 객체 필드(rate, fee, minAmount, maxAmount, estimatedTime, providerType 등)
4. `POST /api/swap/v3/get_dex_swap_transaction_data` — 요청/응답 구조
5. `POST /api/swap/v3/get_evm_dex_approve_calldata` — 요청/응답 구조
6. `POST /api/swap/v3/create_exchange_transaction` — 요청/응답 구조
7. `POST /api/swap/v3/get_transactions_status` — 요청/응답 구조

API 호출 시 `curl` 사용. 에러 응답도 기록한다 (예: 지원하지 않는 페어, 금액 부족 등).

**섹션 2: 프로바이더 유형 분석** — Exchange(크로스체인)와 DEX(동일체인) 프로바이더 구분:
- Exchange 프로바이더 목록 (Changelly, ChangeNOW, Exolix 등) — payInAddress 방식
- DEX 프로바이더 목록 (OKX, 1inch, Uniswap, SushiSwap, LI.FI, Rubic, Squid Router 등) — txdata 방식
- 프로바이더별 지원 체인/토큰 매트릭스 (get_quotes 응답에서 추출)

**섹션 3: Multi-hop 라우팅 자체 지원 검증** — RSRCH-05 핵심:
- 직접 경로가 없는 토큰 페어로 `get_quotes` 호출 (예: 마이너 ERC20 → 마이너 ERC20)
- 응답에 경로 정보가 포함되는지 확인 (중간 토큰 경유 여부)
- 결론: DCent 자체 multi-hop 지원 여부 → Phase 345 범위 확정
  - 지원 시: Phase 345 스킵 또는 범위 축소
  - 미지원 시: WAIaaS 레이어에서 2-hop 구현 필요

**섹션 4: API 운영 특성** — 인증, Rate Limit, 견적 유효 시간:
- API Key 필요 여부 확인
- Rate limit 정책 (429 응답 테스트)
- fixedRate 견적 유효 시간
- 에러 코드 체계

**섹션 5: Currency ID 체계 분석** — RSRCH-02 기반:
- DCent Currency ID 포맷 분류: 네이티브(BITCOIN, ETHEREUM, SOLANA), ERC20(ERC20/0x...), 체인 토큰(TOKEN:chainId)
- get_supported_currencies 응답에서 ID 패턴 추출 및 분류
- WAIaaS CAIP-19 식별자와의 대응 관계 정리

주의: API 호출 시 timeout 10초 설정. 응답이 크면 주요 필드만 발췌하여 문서에 기록한다.

**섹션 6: CAIP-19 <-> DCent Currency ID 변환 설계** — RSRCH-02:
- 변환 규칙 매트릭스 (양방향):
  - `eip155:1/slip44:60` <-> `ETHEREUM` (네이티브)
  - `eip155:1/erc20:0xa0b8...` <-> `ERC20/0xa0b8...` (ERC20)
  - `eip155:137/erc20:0x...` <-> `POLYGON_ERC20/0x...` 또는 해당 DCent 포맷 (체인별 차이 확인)
  - `solana:5eykt4UsFv8P8NJdTREpY1vzqKqZKvdp/slip44:501` <-> `SOLANA` (Solana 네이티브)
  - SPL 토큰 등 추가 패턴
- 변환 함수 시그니처: `caip19ToDcentId(caip19: string): string`, `dcentIdToCaip19(dcentId: string, network?: string): string`
- get_supported_currencies 캐싱 전략: 24h TTL, 메모리 캐시, 데몬 시작 시 프리로드
- 알 수 없는 ID 처리 (fallback → get_supported_currencies 조회)

**섹션 7: DEX Swap 파이프라인 매핑 설계** — RSRCH-04:
- 플로우: get_quotes → 최적 DEX 프로바이더 선택 → get_evm_dex_approve_calldata (ERC20 sell 시) → get_dex_swap_transaction_data → BATCH [APPROVE, CONTRACT_CALL]
- BATCH 구성:
  - Element 1 (조건부): `{ type: 'APPROVE', to: tokenAddress, calldata: approveCalldata }` — ERC20 sell 시에만
  - Element 2: `{ type: 'CONTRACT_CALL', to: routerAddress, calldata: swapCalldata, value: nativeValue }`
- min/max 금액 검증 (견적 응답의 minAmount, maxAmount)
- slippage 처리 (사용자 지정 또는 기본값)
- 에러 매핑: `fail_empty_providers` → NO_ROUTE, `fail_smaller_than_min` → AMOUNT_TOO_SMALL 등

**섹션 8: Exchange 파이프라인 매핑 설계** — RSRCH-03:
- 플로우: get_quotes → Exchange 프로바이더 선택 → create_exchange_transaction → payInAddress + payInAmount → TRANSFER 파이프라인
- TRANSFER 구성: `{ type: 'TRANSFER', to: payInAddress, amount: payInAmount }`
- extraId(memo) 처리 (일부 체인에서 필요)
- 상태 추적: get_transactions_status 폴링 전략 (간격: 30초, 최대: 1시간)
- 완료/실패 알림: NotificationService 연동 (SWAP_COMPLETE, SWAP_FAILED 이벤트)

**섹션 9: DcentSwapActionProvider 인터페이스 설계** — RSRCH-06:
- 기존 패턴 참조: `ZeroExSwapActionProvider`, `LiFiActionProvider`
- metadata: name='dcent_swap', chains=['ethereum', 'polygon', ...], mcpExpose=true, requiresApiKey=false
- actions 정의 (4개):
  1. `get_quotes` — 견적 조회 (정보성, ContractCallRequest 미반환 → metadata에 담아 빈 배열 반환 또는 별도 query 메서드)
  2. `dex_swap` — DEX 스왑 실행 → ContractCallRequest[] 반환 (approve + swap)
  3. `exchange` — Exchange 실행 → ContractCallRequest 반환 (TRANSFER)
  4. `swap_status` — 상태 조회 (정보성)
- resolve() 분기 로직: actionName에 따라 DEX/Exchange 플로우 선택
- DcentSwapApiClient 클래스 설계 (HTTP 클라이언트, retry, timeout)

**섹션 10: MCP/SDK/정책/Admin 통합 설계** — RSRCH-06:
- MCP 도구 4개: `dcent_get_quotes`, `dcent_swap` (DEX), `dcent_exchange` (크로스체인), `dcent_swap_status`
- SDK 메서드 4개: `getDcentQuotes()`, `dcentSwap()`, `dcentExchange()`, `getDcentSwapStatus()`
- 정책 엔진 통합:
  - CONTRACT_WHITELIST: DEX 라우터 주소 자동 허용 또는 사용자 추가 필요
  - ALLOWED_TOKENS: from/to 토큰 정책 평가
  - SPENDING_LIMIT: 스왑 금액 USD 환산 평가
- Admin Settings 키: `dcent_swap.enabled`, `dcent_swap.api_url`, `dcent_swap.default_slippage_bps`, `dcent_swap.max_slippage_bps`
- connect-info capability: `dcent_swap: { enabled, supportedChains, providerCount }`
