<task id="286-01-T3">
**DB migration v29 + LATEST_SCHEMA_VERSION 업데이트**

1. `packages/daemon/src/infrastructure/database/migrate.ts` 수정:

   a. `LATEST_SCHEMA_VERSION`를 `29`로 업데이트

   b. DDL 문의 CHECK constraint는 이미 `NETWORK_TYPES` SSoT 배열 참조하므로 자동 반영

   c. 새 migration v29 추가:
   ```typescript
   MIGRATIONS.push({
     version: 29,
     description: 'Rename Solana network IDs to solana-{network} format (v29.5 #211)',
     managesOwnTransaction: true,
     up: (sqlite) => {
       sqlite.exec('BEGIN');
       try {
         // 1. transactions 테이블: 12-step recreation (CHECK constraint 포함)
         // network 컬럼 값 변환: mainnet→solana-mainnet, devnet→solana-devnet, testnet→solana-testnet
         // CHECK constraint가 NETWORK_TYPES SSoT를 사용하므로 테이블 재생성 필요

         // transactions_new 생성 (최신 DDL 그대로, CHECK 포함)
         // INSERT ... SELECT with CASE WHEN:
         //   WHEN chain = 'solana' AND network = 'mainnet' THEN 'solana-mainnet'
         //   WHEN chain = 'solana' AND network = 'devnet' THEN 'solana-devnet'
         //   WHEN chain = 'solana' AND network = 'testnet' THEN 'solana-testnet'
         //   ELSE network
         // DROP old, RENAME new, recreate indexes

         // 2. policies 테이블: 12-step recreation (CHECK constraint 포함)
         // 동일 CASE WHEN 패턴

         // 3. defi_positions 테이블: 12-step recreation (CHECK constraint 포함)
         // 동일 CASE WHEN 패턴

         // 4. incoming_transactions 테이블: network 컬럼에 CHECK 없으므로 UPDATE만
         sqlite.exec(`UPDATE incoming_transactions SET network = 'solana-mainnet' WHERE chain = 'solana' AND network = 'mainnet'`);
         sqlite.exec(`UPDATE incoming_transactions SET network = 'solana-devnet' WHERE chain = 'solana' AND network = 'devnet'`);
         sqlite.exec(`UPDATE incoming_transactions SET network = 'solana-testnet' WHERE chain = 'solana' AND network = 'testnet'`);

         // 5. incoming_tx_cursors 테이블: network 컬럼에 CHECK 없으므로 UPDATE만
         sqlite.exec(`UPDATE incoming_tx_cursors SET network = 'solana-mainnet' WHERE chain = 'solana' AND network = 'mainnet'`);
         sqlite.exec(`UPDATE incoming_tx_cursors SET network = 'solana-devnet' WHERE chain = 'solana' AND network = 'devnet'`);
         sqlite.exec(`UPDATE incoming_tx_cursors SET network = 'solana-testnet' WHERE chain = 'solana' AND network = 'testnet'`);

         // 6. token_registry 테이블: network 컬럼에 CHECK 없으므로 UPDATE만
         sqlite.exec(`UPDATE token_registry SET network = 'solana-mainnet' WHERE network = 'mainnet'`);
         sqlite.exec(`UPDATE token_registry SET network = 'solana-devnet' WHERE network = 'devnet'`);
         sqlite.exec(`UPDATE token_registry SET network = 'solana-testnet' WHERE network = 'testnet'`);

         sqlite.exec('COMMIT');
       } catch (err) {
         sqlite.exec('ROLLBACK');
         throw err;
       }
     },
   });
   ```

   **중요**: transactions, policies, defi_positions 테이블은 `network` 컬럼에 `CHECK (network IN (...))` constraint가 있어서 12-step table recreation이 필요하다. incoming_transactions, incoming_tx_cursors, token_registry는 CHECK constraint가 없어서 UPDATE만으로 충분하다.

   d. transactions 12-step recreation 시 현재 DDL에 맞게 모든 컬럼 포함 (bridge_status, bridge_metadata, amount_usd, reserved_amount_usd 포함). 기존 v7 migration의 transactions recreation을 참조.

   e. policies 12-step recreation 시 현재 DDL에 맞게 network 컬럼 포함.

   f. defi_positions 12-step recreation 시 현재 DDL에 맞게 모든 컬럼 포함.

</task>

## 검증 기준

1. `SOLANA_NETWORK_TYPES`가 `['solana-mainnet', 'solana-devnet', 'solana-testnet']`
2. `NETWORK_TYPES` 배열에 `'solana-mainnet'`, `'solana-devnet'`, `'solana-testnet'` 포함, `'mainnet'`, `'devnet'`, `'testnet'` 미포함
3. `BUILT_IN_RPC_DEFAULTS`의 키가 `'solana-mainnet'`, `'solana-devnet'`, `'solana-testnet'`
4. `CAIP2_TO_NETWORK` Solana 항목의 network 값이 `'solana-mainnet'` 등
5. explorer URL 매핑에서 `'solana-mainnet'` 키 사용
6. DB migration v29가 존재하고 `LATEST_SCHEMA_VERSION === 29`
7. migration v29가 `transactions`, `policies`, `defi_positions`는 12-step recreation, `incoming_transactions`, `incoming_tx_cursors`, `token_registry`는 UPDATE로 변환
