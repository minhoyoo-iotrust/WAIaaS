**1절: 개요 및 목표**
- 배경: Polymarket = Off-chain CLOB + On-chain CTF Settlement 하이브리드
- 목표: 5개 Phase(370-374) 범위와 전달물 명시
- Hyperliquid v31.4 선례와의 유사성 설명 (ApiDirectResult 패턴 재사용)
- Out of Scope: Proxy Wallet 생성(EOA signatureType=0 사용), WebSocket(v2 deferred), 다른 예측 시장 프로토콜

**2절: Polymarket 아키텍처 개요**
- 3개 서비스: CLOB API(clob.polymarket.com), Gamma API(gamma-api.polymarket.com), Data API(data-api.polymarket.com)
- Polygon 온체인 컨트랙트: CTF Exchange, Neg Risk CTF Exchange, Conditional Tokens(ERC-1155), USDC.e
- 데이터 플로우 다이어그램: Agent -> MCP/REST -> ActionProvider -> CLOB API / 6-stage Pipeline
- 컨트랙트 주소 테이블 (Polygon mainnet, 8개 주소)

**3절: CLOB 인증 플로우**
- L1 인증: EIP-712 ClobAuth 서명 -> POST /auth/api-key -> {apiKey, secret, passphrase} 취득
- L2 인증: HMAC-SHA256(timestamp + method + path + body, secret) -> 요청별 헤더
- API Key 라이프사이클: 지갑별 1세트, polymarket_api_keys 테이블 암호화 저장, lazy creation on first use
- CLOB REST 엔드포인트 참조 테이블 (12개 핵심 엔드포인트)

**4절: EIP-712 서명 아키텍처** (DSGN-02 핵심)
- 3개 EIP-712 도메인 상세:
  - Domain 1: ClobAuthDomain (name='ClobAuthDomain', version='1', chainId=137) — API 키 생성용
  - Domain 2: Polymarket CTF Exchange (name='Polymarket CTF Exchange', version='1', chainId=137, verifyingContract=0x4bFb...) — 바이너리 마켓 주문용
  - Domain 3: Polymarket CTF Exchange (동일 name, verifyingContract=0xC5d5... Neg Risk) — Neg Risk 마켓 주문용
- Order struct 12개 필드 상세 (salt, maker, signer, taker, tokenId, makerAmount, takerAmount, expiration, nonce, feeRateBps, side, signatureType)
- ClobAuth struct 4개 필드 상세
- signatureType: 0=EOA(WAIaaS 기본), 1=POLY_PROXY(미지원), 2=GNOSIS_SAFE(미지원)
- PolymarketSigner 클래스 설계: static signOrder(), static signClobAuth(), static buildHmacHeaders()
- viem signTypedData 직접 사용 (ethers.js 금지)

**5절: Hyperliquid EIP-712 비교 분석** (DSGN-03 핵심)
- 비교 매트릭스 작성: 도메인 구조, 서명 방식, 인증 체계, nonce 전략, API 클라이언트 패턴
- 공통점: ApiDirectResult 패턴, requiresSigningKey, 전용 DB 테이블, RateLimiter, MarketData 서비스
- 차이점: Polymarket은 직접 struct 서명(phantom agent 불필요), HMAC L2 인증(Hyperliquid는 불필요), 듀얼 프로바이더(Hyperliquid는 3개)
- 공유 추상화 범위 결정:
  - 공유 가능: ApiDirectResult Stage 5 분기, ActionProvider 등록 패턴, Admin Settings/connect-info 패턴
  - 공유 불가: 서명 로직(완전히 다른 체계), API 클라이언트(다른 인증), nonce 관리(Polymarket은 salt 기반)
- 결론: 코드 레벨 공유 추상화 없음, 패턴 레벨 재사용만 수행

**6절: Neg Risk (멀티 아웃컴 마켓)**
- Neg Risk 동작 원리: WrappedCollateral, NegRiskAdapter, 바이너리 마켓 분해
- 라우팅 로직: Gamma API neg_risk 플래그 기반 -> CTF Exchange vs Neg Risk CTF Exchange 분기
- USDC approve 대상: 바이너리=CTF Exchange, Neg Risk=Neg Risk CTF Exchange
- Redeem 경로: Neg Risk는 NegRiskAdapter 경유

기존 설계 문서(doc 79 Across Protocol)의 포맷과 수준을 따른다. 모든 코드 예제는 TypeScript + viem 2.x 기반이다.
