다음 구조로 async-tracking-extension-design.md를 작성한다 (설계 문서, 구현 코드 아님):

**1. AsyncTrackingResult.state 확장 설계 (TRCK-01)**
- 기존 4종 state: PENDING, COMPLETED, FAILED, TIMEOUT
- 신규 5종 state 추가:
  - `PARTIALLY_FILLED` — 부분 체결 (CLOB order 등). 폴링 계속
  - `FILLED` — 완전 체결. COMPLETED와 구분되는 이유: COMPLETED는 on-chain 완료, FILLED는 off-chain 주문 체결
  - `CANCELED` — 사용자/시스템 취소. 정상적인 종료
  - `SETTLED` — 정산 완료 (결제, 브릿지 정산 등). 자금 이동 확인됨
  - `EXPIRED` — 시간 만료 (주문 TTL 초과). TIMEOUT과 구분: TIMEOUT은 폴링 초과, EXPIRED는 비즈니스 만료
- state 카테고리 분류:
  - 폴링 계속: PENDING, PARTIALLY_FILLED (BRIDGE_MONITORING도)
  - 정상 종료: COMPLETED, FILLED, SETTLED, REFUNDED
  - 실패 종료: FAILED, CANCELED, EXPIRED, TIMEOUT
- state 확장이 기존 코드에 미치는 영향 분석:
  - AsyncPollingService.processResult(): state에 따른 bridge_status 업데이트 로직 확장 필요
  - 기존 BRIDGE_STATUS_VALUES에 신규 값 추가: PARTIALLY_FILLED, FILLED, CANCELED, SETTLED, EXPIRED
  - BridgeStatusEnum Zod 스키마도 연동 확장
- 하위 호환: 기존 tracker(LI.FI, Across, Lido, Jito, gas-condition)는 기존 4종 state만 사용, 신규 tracker만 확장 state 사용

**2. AsyncPollingService 쿼리 확장 설계 (TRCK-02)**
- 기존 쿼리 조건: `bridge_status IN ('PENDING', 'BRIDGE_MONITORING') OR status = 'GAS_WAITING'`
- 확장 쿼리 조건: 위 조건에 `OR bridge_status = 'PARTIALLY_FILLED'` 추가
- resolveTrackerName() 확장: off-chain action의 경우 `metadata.tracker` 또는 `metadata.trackerName` (Phase 380 SignedDataActionTracking.trackerName과 통합)
- off-chain action 대상 식별: `action_kind IN ('signedData', 'signedHttp') AND bridge_status IN ('PENDING', 'PARTIALLY_FILLED')` (BRIDGE_MONITORING은 on-chain 전용)
- 쿼리 성능: action_kind 인덱스(v56) + bridge_status 기존 인덱스 활용
- off-chain action의 첫 bridge_status 설정 시점:
  - Phase 383에서 action.tracking이 있으면 즉시 CONFIRMED 대신 PENDING으로 기록하는 설계 수정 제안
  - tracking이 없으면 CONFIRMED (비동기 추적 불필요, 즉시 완료)
  - tracking이 있으면 bridge_status='PENDING' + bridge_metadata에 trackerName/metadata 저장

**3. tracker 메타데이터 확장 설계 (TRCK-03)**
- 기존 bridge_metadata JSON 필드 구조: `{ tracker: string, lastPolledAt: number, pollCount: number, ...tracker-specific }`
- off-chain action용 메타데이터 확장:
  - `venue: string` — 외부 서비스 식별자 (검색/필터용)
  - `operation: string` — 액션 이름
  - `trackerName: string` — IAsyncStatusTracker 이름 (기존 `tracker`와 통합)
  - `externalId?: string` — 외부 서비스 반환 ID (주문 ID 등)
  - `notionalUsd?: number` — 추정 USD 가치 (정책 누적 계산용)
  - `actionCategory?: string` — 카테고리 (trade/withdraw 등)
  - `fillPercentage?: number` — 체결률 (PARTIALLY_FILLED 시 0~100)
  - `trackerSpecific: Record<string, unknown>` — tracker 구현체별 자유 필드
- metadata 전달 경로: SignedDataAction.tracking.metadata -> bridge_metadata JSON -> IAsyncStatusTracker.checkStatus() metadata 파라미터
- 기존 tracker와의 호환: 기존 tracker는 기존 metadata 구조 그대로 사용, 신규 필드 무시

**4. 상태 저장 위치 결정 (TRCK-04)**
- 두 가지 선택지 분석:
  - Option A: transactions 테이블 bridge_status/bridge_metadata 재사용
    - 장점: 기존 인프라(AsyncPollingService, 쿼리, 인덱스) 100% 재사용, 추가 테이블 불필요, 마이그레이션 최소
    - 단점: bridge_status 의미론적 혼용 (on-chain 브릿지 + off-chain order), 컬럼명 혼란
  - Option B: 별도 external_action_status 테이블
    - 장점: 깔끔한 분리, 의미론적 명확성
    - 단점: 별도 쿼리/인덱스/마이그레이션 필요, AsyncPollingService 이중 쿼리
- **결정: Option A (bridge_status/bridge_metadata 재사용)**
  - 근거: AsyncPollingService의 기존 인프라를 그대로 활용할 수 있어 구현 복잡도가 크게 낮음. 컬럼명 혼란은 코드 주석으로 해소. 향후 리네이밍(bridge_status -> tracking_status)은 별도 마일스톤에서 가능.
  - 대안 메모: 향후 tracking_status/tracking_metadata로 리네이밍 마이그레이션 고려 (BREAKING 변경이므로 major version에서)
- DB 마이그레이션 필요 여부:
  - bridge_status CHECK 제약에 신규 5종 값 추가 필요 (PARTIALLY_FILLED, FILLED, CANCELED, SETTLED, EXPIRED)
  - 마이그레이션 v57 (Phase 383 v56 다음)
  - CHECK 제약 변경 SQL 설계

**5. 기존 tracker 영향 분석**
- LI.FI BridgeStatusTracker: 영향 없음 (PENDING/COMPLETED/FAILED만 반환)
- Across BridgeStatusTracker: 영향 없음
- Lido WithdrawalTracker: 영향 없음
- Jito EpochTracker: 영향 없음
- GasConditionTracker: 영향 없음
- 신규 off-chain tracker 예시: ClobOrderTracker (Polymarket/CoW), CexOrderTracker (Binance) — PARTIALLY_FILLED/FILLED/CANCELED/EXPIRED 사용

**6. 알림 연동 설계**
- 기존 AsyncPollingService 알림 콜백(emitNotification) 재사용
- 신규 이벤트 타입: external_action_filled, external_action_canceled, external_action_expired, external_action_partially_filled
- WalletNotificationChannel 이벤트 카테고리 매핑
