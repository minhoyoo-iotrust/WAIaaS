From signer-capability-interface.md:
```typescript
export interface ISignerCapability {
  readonly scheme: SigningScheme;
  canSign(params: SigningParams): boolean;
  sign(params: SigningParams): Promise<SigningResult>;
}

export interface ISignerCapabilityRegistry {
  register(capability: ISignerCapability): void;
  get(scheme: SigningScheme): ISignerCapability | undefined;
  resolve(action: SignedDataAction | SignedHttpAction): ISignerCapability;
  listSchemes(): readonly SigningScheme[];
}
```

HMAC/RSA-PSS SigningParams:
```typescript
export interface HmacSigningParams {
  scheme: 'hmac-sha256';
  data: string;      // signing target
  secret: string;    // HMAC secret from CredentialVault
}

export interface RsaPssSigningParams {
  scheme: 'rsa-pss';
  data: string;      // signing target
  privateKey: string; // PEM-encoded RSA private key from CredentialVault
  saltLength?: number; // default: 32
}
```

설계 문서를 작성한다. 구현 코드가 아닌 설계 명세만 산출한다.

**1. HmacSignerCapability 상세 설계:**
- scheme: 'hmac-sha256'
- 구현 기반: node:crypto의 createHmac('sha256', secret).update(data).digest('hex')
- canSign(): params.scheme === 'hmac-sha256', data 존재, secret 존재 확인
- sign() 흐름:
  1. params에서 data, secret 추출
  2. createHmac('sha256', secret) 생성
  3. .update(data).digest('hex') 계산
  4. secret 메모리 즉시 클리어 (Buffer.from(secret).fill(0))
  5. SigningResult { signature: hexDigest } 반환
- CEX API 인증 패턴 지원:
  - Binance: HMAC-SHA256(timestamp + method + path + body, secret)
  - OKX: HMAC-SHA256(timestamp + method + path + body, secret) → Base64
  - data 필드에 이미 조합된 signing target 전달 (조합은 ActionProvider 책임)
  - Base64 변환이 필요한 경우 metadata.encoding: 'base64' 옵션 지원
- 에러 매핑: node:crypto 에러 -> SigningError('SIGNING_FAILED')
- 보안 고려: secret은 CredentialVault에서 주입, sign() 후 즉시 클리어

**2. RsaPssSignerCapability 상세 설계:**
- scheme: 'rsa-pss'
- 구현 기반: node:crypto의 sign('RSA-SHA256', data, { key, padding: RSA_PKCS1_PSS_PADDING, saltLength })
- canSign(): params.scheme === 'rsa-pss', data 존재, privateKey(PEM) 존재 확인
- sign() 흐름:
  1. params에서 data, privateKey, saltLength 추출
  2. Buffer.from(data) 변환
  3. node:crypto sign() with RSA-PSS padding
  4. privateKey 메모리 클리어
  5. SigningResult { signature: base64Signature } 반환
- PEM 키 검증: '-----BEGIN RSA PRIVATE KEY-----' 또는 '-----BEGIN PRIVATE KEY-----' 형식 확인
- saltLength 기본값: 32 (SHA-256 해시 길이)
- 에러 매핑: INVALID_KEY (PEM 파싱 실패), SIGNING_FAILED (서명 연산 실패)

**3. SignerCapabilityRegistry 상세 설계:**
- 내부 저장소: Map<SigningScheme, ISignerCapability>
- register(): 같은 scheme 덮어쓰기 허용 (테스트 mock 주입 용이)
- get(): scheme으로 조회, undefined 반환 가능
- resolve(action):
  1. action.signingScheme 추출
  2. registry.get(scheme) 조회
  3. 없으면 SigningError('CAPABILITY_NOT_FOUND') throw (code 추가 필요 — SigningErrorCode에 'CAPABILITY_NOT_FOUND' 추가)
  4. capability.canSign() 검사 (false면 SigningError('INVALID_PARAMS'))
  5. capability 반환
- listSchemes(): 등록된 모든 scheme 배열 반환 (connect-info API 노출용)

**4. daemon 부팅 시 초기 등록 절차:**
- bootstrapSignerCapabilities(registry) 함수 설계
- 등록 순서: Eip712 -> Personal -> Erc8128 -> Hmac -> RsaPss -> EcdsaSignBytes -> Ed25519SignBytes
- TransactionSignerCapability는 등록하지 않음 (기존 pipeline 사용)
- 등록 후 listSchemes() 결과가 connect-info API에 노출

**5. connect-info 확장:**
- capabilities.signing: SigningScheme[] (registry.listSchemes() 반환값)
- 에이전트가 어떤 서명 방식을 사용할 수 있는지 자기 발견 가능

**6. SigningErrorCode 확장:**
- 'CAPABILITY_NOT_FOUND' 추가 (resolve()에서 미등록 scheme 요청 시)
- Phase 380 설계의 5종 + 1종 = 총 6종
