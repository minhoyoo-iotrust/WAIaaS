설계 문서를 작성한다. 다음 섹션을 포함해야 한다:

**섹션 1: 개요** — Phase 380~382 설계를 바탕으로 파이프라인 라우팅의 목적과 설계 원칙 기술. 기존 파이프라인(6-stage, sign-only, sign-message, sign-http) 무변경 원칙 재확인.

**섹션 2: 3-way 파이프라인 라우팅 (PIPE-01)** — ActionProviderRegistry.executeAction() 내부의 kind별 분기 흐름을 상세 설계한다.
- resolve() 결과를 normalizeResolvedActions()로 정규화한 후 kind별 switch 분기
- contractCall: 기존 6-stage pipeline 진입 (변경 없음)
- signedData: 새로운 sign-data pipeline (credential 해소 -> SigningParams 조립 -> canSign() 검증 -> sign() 실행 -> 결과 DB 기록 -> 응답)
- signedHttp: 새로운 sign-http pipeline (credential 해소 -> Erc8128/HMAC/RSA-PSS 서명 -> HTTP 요청 발송 -> 결과 DB 기록 -> 응답)
- ApiDirectResult: 기존 isApiDirectResult() 분기 유지 (정규화 전에 분기)
- mixed-kind 배열 처리: ResolvedAction[]에서 contractCall + signedData 혼재 시 순차 실행
- 전체 흐름도 (ASCII art) 포함

**섹션 3: signedData 파이프라인 상세** — 5단계 흐름 상세 설계
1. credentialRef 해소: CredentialVault.get(ref, walletId) — Phase 381 resolveCredentialRef() 활용
2. SigningParams 조립: scheme별 discriminated union 매핑 (eip712→Eip712SigningParams, hmac-sha256→HmacSigningParams, etc.)
3. canSign() 사전 검증: credential 주입 후 완전한 params로 검증 (Phase 382 설계 참조)
4. sign() 실행: SignerCapabilityRegistry.resolve(action) → ISignerCapability.sign(params)
5. 결과 처리: DB 기록 + 응답 조립 + credential 메모리 클리어
- credential이 없는 경우(credentialRef undefined): wallet key 직접 사용 (EIP-712 등)
- payload에서 scheme별 SigningParams 필드 추출 로직

**섹션 4: signedHttp 파이프라인 상세** — 6단계 흐름 상세 설계
1. credentialRef 해소 (signedData와 동일)
2. HTTP 요청 서명 (Erc8128/HMAC/RSA-PSS)
3. 서명된 헤더 첨부
4. HTTP 요청 발송 (fetch)
5. 응답 처리 + DB 기록
6. 결과 반환
- 기존 ERC-8128 sign-http REST API와의 관계 (별도 경로, 무변경)
- signedHttp에서 fetch까지 수행하는지, 서명만 반환하는지 설계 결정 필요 (권장: 서명만 반환, fetch는 ActionProvider 책임)

**섹션 5: off-chain action DB 기록 (PIPE-03)** — transactions 테이블 확장 설계
- action_kind 컬럼 추가: TEXT NOT NULL DEFAULT 'contractCall' (기존 레코드 호환)
- txHash를 nullable로 변경: off-chain 액션은 txHash가 없음
  - 기존 코드의 txHash NOT NULL 가정 전수 조사 결과 기술 (STATE.md에 명시된 blocker)
  - nullable 전환 시 영향받는 코드 경로 식별
- off-chain 전용 필드 추가: venue TEXT, operation TEXT, external_id TEXT (nullable)
- DB 마이그레이션 v55 또는 v56 설계 (wallet_credentials가 v55이면 v56)
- Drizzle 스키마 초안
- 인덱스: idx_transactions_action_kind, idx_transactions_venue

**섹션 6: 정책 평가 시점 보장 (PIPE-02)** — 모든 kind에서 resolve() 후 서명 전 정책 평가
- 기존 contractCall 정책 평가 위치 확인 (Stage 2: 정책 평가)
- signedData/signedHttp 경로에서 동일 시점 보장
  - policyContext 필드를 ActionPolicyParam으로 변환
  - venue, actionCategory, notionalUsd 등을 정책 엔진에 전달
  - 기존 PolicyService.evaluate() 호출 흐름 재사용
- 정책 거부 시 서명하지 않고 에러 반환
- Phase 384에서 상세화될 정책 확장(VENUE_WHITELIST, ACTION_CATEGORY_LIMIT)의 placeholder 설계

**섹션 7: 에러 처리** — kind별 에러 코드 매핑
- CREDENTIAL_NOT_FOUND, CREDENTIAL_EXPIRED, CREDENTIAL_ACCESS_DENIED → Phase 381 에러
- CAPABILITY_NOT_FOUND → Phase 382 에러
- SIGNING_FAILED, INVALID_KEY, INVALID_PARAMS → Phase 382 에러
- POLICY_DENIED → 기존 정책 에러 재사용
- HTTP_REQUEST_FAILED → signedHttp 전용 (선택적)
