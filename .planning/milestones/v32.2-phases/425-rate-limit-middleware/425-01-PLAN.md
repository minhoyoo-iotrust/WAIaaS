**Step 2: Create SlidingWindowRateLimiter class**

Create `packages/daemon/src/api/middleware/rate-limiter.ts`:

```typescript
/**
 * In-memory sliding-window rate limiter with 3-tier Hono middleware factories.
 *
 * Architecture:
 * - SlidingWindowRateLimiter: generic sliding-window counter with TTL auto-cleanup
 * - createIpRateLimiter: IP-based global rate limit (all endpoints)
 * - createSessionRateLimiter: session-based rate limit (sessionAuth endpoints)
 * - createTxRateLimiter: transaction-specific rate limit (POST /v1/transactions, /v1/actions)
 *
 * All three read limit values from SettingsService on each request (hot-reload).
 * Expired window entries are automatically cleaned every 60 seconds.
 */
```

SlidingWindowRateLimiter implementation:
- `private buckets: Map<string, { timestamps: number[]; }>` -- stores request timestamps per key
- `constructor(opts: { cleanupIntervalMs?: number })` -- starts cleanup interval (default 60_000)
- `check(key: string, limit: number, windowMs: number = 60_000): { allowed: boolean; remaining: number; retryAfterSec: number }` -- sliding window check. Filter timestamps within window, if count < limit: add timestamp, return allowed=true. Else return allowed=false with retryAfterSec = ceil((oldest_in_window + windowMs - now) / 1000).
- `cleanup()` -- iterate buckets, remove entries whose ALL timestamps are older than max windowMs (60_000). Delete empty buckets.
- `destroy()` -- clearInterval for cleanup timer.
- `get size(): number` -- number of active keys (for testing).

**Step 3: Create 3 middleware factories**

In the same file, export three factory functions. Each receives `{ settingsService: SettingsService }`:

`createIpRateLimiter(deps)`:
- Uses `createMiddleware` from `hono/factory`
- Key = IP from `c.req.header('x-forwarded-for')?.split(',')[0]?.trim() || c.req.header('x-real-ip') || 'unknown'`
- Limit = `parseInt(deps.settingsService.get('security.rate_limit_global_ip_rpm'), 10) || 1000`
- On reject: `throw new WAIaaSError('RATE_LIMITED', { message: 'IP rate limit exceeded', hint: 'Reduce request frequency or increase rate_limit_global_ip_rpm in Admin Settings' })` -- but FIRST set `c.header('Retry-After', String(result.retryAfterSec))` before throwing.
- **Important**: use `c.res = new Response(...)` pattern or set header on response. Since WAIaaSError triggers errorHandler which creates the response, use a different approach: return `c.json({ error: { code: 'RATE_LIMITED', ... } }, 429)` directly with `Retry-After` header set via `c.header()`.
  Actually, the cleanest approach: set `Retry-After` header, then throw WAIaaSError. The errorHandler in error-handler.ts already uses `c.json()` which will include headers set before the throw. Verify this works by testing; if not, return a manual 429 response instead.

`createSessionRateLimiter(deps)`:
- Key = `c.get('sessionId' as never) as string | undefined`
- If no sessionId (unauthenticated request), skip (call `await next()`)
- Limit = `parseInt(deps.settingsService.get('security.rate_limit_session_rpm'), 10) || 300`

`createTxRateLimiter(deps)`:
- Key = sessionId (same as session limiter, but with tighter limit)
- If no sessionId, skip
- Limit = `parseInt(deps.settingsService.get('security.rate_limit_tx_rpm'), 10) || 10`

All three share one global SlidingWindowRateLimiter instance each (module-level singleton created on first factory call, or passed as option). Actually, create one limiter per factory call since createApp() is called once:
- Each factory function creates its own `new SlidingWindowRateLimiter()`.
