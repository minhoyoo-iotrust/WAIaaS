  <action>
    1. In openapi-schemas.ts, add a reusable PaginationQuerySchema:
       ```typescript
       export const PaginationQuerySchema = z.object({
         limit: z.coerce.number().int().min(1).max(200).default(50).optional(),
         offset: z.coerce.number().int().min(0).default(0).optional(),
       });
       ```
       Add factory function `createPaginatedSchema(itemSchema, name)` that returns:
       ```typescript
       z.object({
         data: z.array(itemSchema),
         total: z.number().int(),
         limit: z.number().int(),
         offset: z.number().int(),
       }).openapi(name)
       ```
       Create `PaginatedSessionListSchema` and `PaginatedPolicyListSchema` using this factory.

    2. In sessions.ts `listSessionsRoute`:
       - Merge PaginationQuerySchema into existing query schema: `z.object({ walletId: ... }).merge(PaginationQuerySchema)`
       - Change response schema from `z.array(SessionListItemSchema)` to `PaginatedSessionListSchema`
       - In handler: extract `limit` (default 50) and `offset` (default 0) from validated query
       - After building the full `result` array, compute `total = result.length`
       - Return `{ data: result.slice(offset, offset + limit), total, limit, offset }`
       - Note: Since sessions are already fetched with batch wallet joins, we apply pagination in-memory after the full query. For the current scale (sessions are bounded by max_sessions_per_wallet), this is appropriate. The DB query fetches all matching rows, then we slice. This preserves the existing NQ-01 batch optimization.

    3. In policies.ts `listPoliciesRoute`:
       - Same pattern: merge PaginationQuerySchema into query, update response schema
       - In handler: extract limit/offset, compute total from full rows, return paginated wrapper
       - For sessionAuth path: same pagination applies

    4. Add tests to api-sessions.test.ts:
       - Create 3+ sessions, then test limit=2&offset=0 returns 2 items with total=3+
       - Test offset beyond total returns empty data
       - Test default pagination (no params) returns paginated wrapper format

    5. Add tests to api-policies.test.ts:
       - Create 3+ policies, test limit=1&offset=1 returns 1 item
       - Test offset beyond total returns empty data
       - Test default pagination returns paginated wrapper format

    IMPORTANT: The response format changes from `[items]` to `{data: [items], total, limit, offset}`. This is a BREAKING CHANGE for raw API consumers, but PAG-03 requires backward compatibility via defaults. Since limit defaults to 50 and offset to 0, existing clients that don't send pagination params will get the same items wrapped in the new format. SDK/MCP will be updated in Plan 02 to handle this.

    NOTE on existing test compatibility: Some existing tests check `const body = await res.json()` and expect an array. These tests MUST be updated to use `body.data` instead. Review all GET /v1/sessions and GET /v1/policies test assertions and update them to expect the paginated wrapper format.
