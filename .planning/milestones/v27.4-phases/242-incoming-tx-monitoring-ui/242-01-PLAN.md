**3) Create `packages/admin/src/pages/incoming.tsx`:**

The page has two visual sections in a single scrollable page:

**Section A: Settings Panel** (collapsible, default expanded)
- Header: "Incoming TX Monitoring Settings" with a collapse/expand toggle
- Reuses the same settings save/discard pattern from settings.tsx (fetch settings via `apiGet(API.ADMIN_SETTINGS)`, track dirty state, save via `apiPut(API.ADMIN_SETTINGS, { settings: entries })`)
- Fields (same as the removed IncomingSettings from settings.tsx):
  - `incoming.enabled` — select (Yes/No)
  - `incoming.poll_interval` — number, min=5, max=3600
  - `incoming.retention_days` — number, min=1, max=365
  - `incoming.suspicious_dust_usd` — number, min=0, max=1000
  - `incoming.suspicious_amount_multiplier` — number, min=1, max=1000
  - `incoming.cooldown_minutes` — number, min=1, max=1440
  - `incoming.wss_url` — text, placeholder "wss://custom-rpc.example.com"
- Only save keys prefixed with `incoming.` when saving from this page (filter dirty entries by prefix before sending to PUT /admin/settings)
- Info box with same description text as the removed section

**Section B: Per-Wallet Monitoring Toggles**
- Header: "Per-Wallet Monitoring"
- Fetch wallet list via `apiGet<WalletListItem[]>(API.WALLETS)` on mount (same pattern as transactions.tsx)
- The response items (WalletCrudResponse) do NOT include monitorIncoming. So for each wallet, also call `apiGet(API.WALLET_PATCH(id))` — WRONG, that's for PATCH. Instead, use the GET /wallets/:id detail endpoint which should have monitorIncoming. Actually looking at WalletDetailResponseSchema it doesn't either.
- **Alternative approach**: Fetch wallets list, then for each wallet display a toggle. On toggle, call `apiPatch(API.WALLET_PATCH(id), { monitorIncoming: newValue })`. The response returns `{ id, monitorIncoming }` — use that to update local state.
- **Initial state**: Since the list API doesn't return monitorIncoming, we need to get it another way. The simplest approach: query the DB via a lightweight endpoint. But since no such endpoint exists yet, use an alternative: create a local signal map `walletMonitorState: Record<string, boolean>` and for each wallet in the list, call GET /wallets/:id to check the detail. **Actually**, looking more carefully at WalletDetailResponseSchema — it doesn't include monitorIncoming either. So we need to add it to the response, OR use a different approach.
- **Pragmatic approach**: Add `monitorIncoming` to WalletDetailResponseSchema and the wallet detail handler response. But that changes daemon code. Instead, for the Admin UI, we can call PATCH with the current state to read it — but that's a mutation.
- **Best approach**: For each wallet, make a single PATCH request on load? No. The cleanest approach: add `monitorIncoming` field to the wallet list response in the wallet CRUD handler. This is a simple one-line change in the wallets route handler.

So:
- In `packages/daemon/src/api/routes/wallets.ts`, in the `listWalletsRoute` handler, add `monitorIncoming: a.monitorIncoming ?? false` to the map callback.
- In `packages/daemon/src/api/routes/openapi-schemas.ts`, add `monitorIncoming: z.boolean()` to `WalletCrudResponseSchema`.
- This makes the wallet list return `monitorIncoming` for each wallet.
- In incoming.tsx, the wallet list section renders a table: Wallet Name | Chain | Status | Monitor Incoming (toggle)
- Toggle calls `apiPatch(API.WALLET_PATCH(walletId), { monitorIncoming: !currentValue })` and updates local state on success.

**Section C: Incoming TX Table**
- Header: "Incoming Transactions"
- FilterBar with fields:
  - wallet_id — select, dynamically populated from wallet list (same pattern as transactions.tsx)
  - chain — select, options: `[{ value: 'solana', label: 'Solana' }, { value: 'evm', label: 'EVM' }]`
  - status — select, options: `[{ value: 'DETECTED', label: 'DETECTED' }, { value: 'CONFIRMED', label: 'CONFIRMED' }]`
  - suspicious — select, options: `[{ value: 'true', label: 'Yes' }, { value: 'false', label: 'No' }]`
- Table columns: Time | Wallet | Sender | Amount | Chain | Network | Status | Suspicious
  - Time: `formatDate(detectedAt)`
  - Wallet: walletName or walletId.slice(0,8)
  - Sender: `formatAddress(fromAddress)`
  - Amount: raw amount string
  - Chain: chain value
  - Network: network value
  - Status: Badge with variant (CONFIRMED=success, DETECTED=warning)
  - Suspicious: Badge `Yes` (danger) / `No` (success)
- Server-side offset/limit pagination (same pattern as transactions.tsx)
- Row click expands detail grid showing all fields (id, txHash, walletId, walletName, fromAddress, amount, tokenAddress, chain, network, status, blockNumber, detectedAt, confirmedAt, suspicious)
- Data fetched via `apiGet(API.ADMIN_INCOMING + '?' + params)` with filter/pagination query params
- ExplorerLink not needed here since incoming TX viewer doesn't have a hash column that links to explorer (txHash is in expand detail)
