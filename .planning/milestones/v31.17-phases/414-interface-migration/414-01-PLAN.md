각 페이지에 대해 dashboard.tsx 마이그레이션 패턴을 반복한다:

**페이지 마이그레이션 (8개):**
1. `import { apiGet, apiPost, ... } from '../api/client'` → `import { api, ApiError } from '../api/typed-client'`
2. manual `interface Xxx { ... }` 삭제 → types.aliases.ts 또는 직접 `components['schemas'][...]`에서 type alias 선언
3. `apiGet<Type>(url)` → `const { data } = await api.GET(path as typed)` 형식으로 교체
4. `apiPost<Type>(url, body)` → `const { data } = await api.POST(path, { body })` 형식으로 교체
5. `apiPut`, `apiDelete`, `apiPatch` 동일 패턴 적용
6. API.ENDPOINT 상수는 그대로 유지하되, typed client의 path 자동완성과 일치하도록 리터럴 경로 문자열 사용 권장 (endpoints.ts의 동적 경로는 `\`/v1/wallets/${id}\`` 형태로 변환)

**주의사항:**
- `ApiError` import는 `'../api/typed-client'`에서 가져온다 (re-export됨)
- typed client의 GET/POST/PUT/DELETE는 `{ data, error }` 구조를 반환한다. error가 있으면 middleware에서 이미 throw하므로, `data!` 또는 non-null assertion 사용
- 동적 경로(예: `/v1/wallets/${id}`)는 openapi-fetch의 params 패턴 사용: `api.GET('/v1/wallets/{walletId}', { params: { path: { walletId: id } } })`
- query string이 포함된 URL은 `params: { query: { ... } }` 사용

**테스트 마이그레이션 (10개):**
1. `vi.mock('../api/client', ...)` → `vi.mock('../api/typed-client', ...)`
2. mock 구조를 openapi-fetch 패턴으로 변경: `api: { GET: vi.fn(), POST: vi.fn(), ... }`
3. mock return 값을 `{ data: ... }` 구조로 변경
4. mock 객체에 `satisfies components['schemas'][...]` 추가하여 타입 검증
5. `import { apiGet } from '../api/client'` → `import { api } from '../api/typed-client'`
