    3. Replace apiGet/apiPost calls with typed client calls:
       - `apiGet<AdminStatus>(API.ADMIN_STATUS)` -> `api.GET('/v1/admin/status').then(r => { if (r.error) throw new ApiError(...); return r.data!; })`
       - Create a small helper `unwrap()` to reduce repetition:
         ```typescript
         function unwrap<T>(result: { data?: T; error?: unknown; response: Response }): T {
           if (result.error || !result.data) {
             const err = result.error as { code?: string; message?: string } | undefined;
             throw new ApiError(result.response.status, err?.code ?? 'UNKNOWN', err?.message ?? 'Unknown error');
           }
           return result.data;
         }
         ```
       - Actually, openapi-fetch middleware already handles error throwing from Plan 413-01. The response middleware throws ApiError on non-ok responses. So we can simplify:
         - Since the middleware throws on error, `api.GET()` will throw before returning `{ error }`. The caller just needs `(await api.GET('/v1/admin/status')).data!`
         - BUT: openapi-fetch wraps errors differently — the middleware `onResponse` can throw, which means the promise rejects. So existing try/catch patterns work.
         - Replace: `const result = await apiGet<AdminStatus>(API.ADMIN_STATUS)` with `const { data } = await api.GET('/v1/admin/status'); const result = data!;`
         - The error handling (catch ApiError) remains unchanged since middleware throws ApiError.

    4. Handle query parameter APIs:
       - For defi positions with query params: `api.GET('/v1/admin/defi/positions', { params: { query: { wallet_id: ..., category: ... } } })`
       - For transactions with query: `api.GET('/v1/admin/transactions', { params: { query: { status: 'APPROVED', limit: 1 } } })`
       - For wallets list: `api.GET('/v1/wallets')`

    5. Handle POST calls:
       - `apiPost<AgentPromptResult>(API.ADMIN_AGENT_PROMPT, {})` -> `const { data } = await api.POST('/v1/admin/agent-prompt', { body: {} }); const result = data!;`

    6. Remove `API` endpoint constant imports that are no longer needed (path strings are now inline in typed client calls).

    7. Keep the `API` import if any endpoint constants are still used (e.g., for non-typed-client usage). If ALL calls are migrated, remove the `API` import entirely.

    IMPORTANT: openapi-fetch returns `{ data, error, response }`. When the response middleware throws ApiError on errors, the promise rejects — so `data` is always defined on the success path. Use `data!` (non-null assertion) in the success path since error cases throw before reaching that point.

    NOTE: If query parameter types in generated types use snake_case but the URL requires it, openapi-fetch handles the mapping through the path definition. Check the generated types for exact query param names.
