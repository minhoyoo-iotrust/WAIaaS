1. `scripts/check-api-contract.ts` 생성 -- CI에서 독립 실행 가능한 contract 검증 스크립트:
   - `packages/admin/openapi.json`을 로드하여 모든 endpoint의 200 응답 스키마 property 키를 추출한다.
   - 중첩 스키마는 `$ref`를 resolve하여 components.schemas에서 재귀적으로 키를 수집한다.
   - `packages/admin/src/pages/*.tsx` 및 `packages/admin/src/api/types.aliases.ts`를 읽어 실제 사용되는 필드 패턴을 추출한다:
     - `data.{field}` 패턴 (typed-client 응답 접근)
     - `components['schemas']['{Name}']` 패턴 (타입 alias 참조)
   - 각 사용된 스키마에 대해: Admin UI가 접근하는 키가 OpenAPI spec 스키마에 존재하는지 검증한다.
   - 누락 키 발견 시 에러 출력 + exit code 1. 모두 일치 시 성공 메시지 + exit code 0.
   - 기존 `scripts/check-api-types-freshness.ts` 패턴을 따른다 (ROOT resolve, console output, process.exit).

2. `packages/admin/src/__tests__/api-contract.test.ts` 생성 -- vitest 환경에서 실행되는 구조 검증:
   - `openapi.json`을 import하여 `components.schemas`의 모든 스키마를 순회한다.
   - `types.aliases.ts`에서 참조하는 스키마 이름 목록을 추출한다 (정규식으로 `components['schemas']['XXX']` 파싱).
   - 각 참조된 스키마가 openapi.json의 components.schemas에 존재하는지 검증한다.
   - 각 스키마의 property 키가 비어있지 않은지 (빈 object 아닌지) 검증한다.
   - Negative test 케이스: 존재하지 않는 스키마 참조를 시뮬레이션하여 탐지되는지 검증한다.

3. `package.json` 루트의 scripts에 `"check:api-contract": "tsx scripts/check-api-contract.ts"` 추가한다. 기존 `check:api-types-freshness` 바로 아래에 위치한다.
