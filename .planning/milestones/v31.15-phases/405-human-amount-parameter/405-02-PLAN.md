  **Step 1: 공유 헬퍼 생성 (packages/actions/src/common/resolve-human-amount.ts)**

  ```typescript
  import { parseAmount } from '@waiaas/core';

  /**
   * Resolve humanAmount variant in provider params.
   * If humanAmountField is present, converts to smallest-unit and sets amountField.
   * Mutates params in place and removes the humanAmount field.
   *
   * @param params - Provider input params (will be mutated)
   * @param amountField - Original amount field name (e.g., 'amount', 'sellAmount', 'amountIn')
   * @param humanAmountField - Human amount field name (e.g., 'humanAmount', 'humanSellAmount', 'humanAmountIn')
   * @param decimals - Token decimals for conversion
   */
  export function resolveProviderHumanAmount(
    params: Record<string, unknown>,
    amountField: string,
    humanAmountField: string,
    decimals: number,
  ): void {
    const humanValue = params[humanAmountField];
    if (typeof humanValue === 'string' && humanValue.length > 0) {
      params[amountField] = parseAmount(humanValue, decimals).toString();
      delete params[humanAmountField];
    }
  }
  ```

  **Step 2: 각 provider 스키마에 humanAmount 필드 추가**

  각 provider의 inputSchema에 humanAmount 변형을 optional로 추가하고, superRefine 또는 refine으로 XOR 검증을 추가한다. 변환 자체는 provider의 resolve() 메서드 내에서 resolveProviderHumanAmount를 호출한다.

  **중요 패턴:** Zod schema에 humanAmount를 추가할 때, 기존 amount 필드를 `.optional()`로 변경하고 `.superRefine()`으로 XOR 검증한다. 그러나 ActionProviderRegistry.executeResolve()에서 `inputSchema.parse(params)`를 호출하므로, schema 수준에서 XOR이 검증된다. resolve() 내에서는 이미 검증된 params에서 humanAmount -> amount 변환만 수행하면 된다.

  **대안 전략 (superRefine이 inputSchema.parse에서 문제 발생 시):** amount는 required로 유지하되, provider resolve() 메서드 시작 부분에서 humanAmount가 있으면 amount로 변환하고 amount가 없으면 에러를 throw한다. Schema에는 amount를 optional + humanAmount를 optional로 두되 refine 없이, resolve()에서 수동 XOR 검증. 빌드가 통과하는 접근을 선택한다.

  **Provider별 매핑:**
  - jupiter_swap: amount -> humanAmount (decimals는 inputMint 기반이므로, 공통 decimals 필드를 추가하거나 resolve에서 registry 조회 -- 단 provider는 registry 접근 불가, decimals를 schema에 required로 추가)
  - zerox_swap: sellAmount -> humanSellAmount
  - dcent_swap: amount -> humanAmount
  - across_bridge: amount -> humanAmount
  - aave_v3: amount -> humanAmount (4개 action 모두)
  - kamino: amount -> humanAmount (4개 action 모두)
  - lido_staking: amount -> humanAmount (2개 action)
  - jito_staking: amount -> humanAmount (2개 action)
  - pendle: amountIn -> humanAmountIn (mintPt, mintYt, mintSy, redeemSy), amount -> humanAmount (redeemMatured)

  **핵심 설계 결정 - decimals 조회:**
  Provider의 resolve()는 TokenRegistryService에 접근할 수 없다. humanAmount 변환에 decimals가 필요하므로:
  - humanAmount 사용 시 `decimals` 필드를 schema에 required로 추가 (humanAmount가 있을 때만 필요)
  - 또는 ActionContext에 tokenRegistry를 주입 (너무 큰 변경)
  - **선택: humanAmount 사용 시 decimals 필드를 필수로 요구.** 스키마에 `decimals: z.number().int().min(0).max(24).optional().describe('Token decimals. Required when using humanAmount.')`를 추가하고, humanAmount가 있는데 decimals가 없으면 XOR refine에서 에러.

  LI.FI는 amount가 nested (fromToken.amount)이므로 별도 처리가 필요하다. LI.FI의 경우 fromToken 객체에 humanAmount 필드를 추가한다.

  **Step 3: resolve() 메서드 수정**

  각 provider의 resolve() 시작 부분에 resolveProviderHumanAmount 호출을 추가한다. 예시 (jupiter_swap):
  ```typescript
  async resolve(actionName, params, context) {
    // humanAmount -> amount conversion (before schema parse)
    const resolvedParams = { ...params };
    if (resolvedParams.humanAmount && resolvedParams.decimals) {
      resolveProviderHumanAmount(resolvedParams, 'amount', 'humanAmount', resolvedParams.decimals as number);
      delete resolvedParams.decimals; // schema parse 전에 decimals 제거 (기존 schema에 없으므로)
    }
    const input = SwapInputSchema.parse(resolvedParams);
    // ...
  }
  ```

  그러나 ActionProviderRegistry.executeResolve()가 inputSchema.parse(params)를 먼저 호출하므로, humanAmount가 inputSchema에 포함되어 있어야 한다. 즉 schema 레벨에서 humanAmount+decimals를 추가하고, resolve()에서 변환 후 기존 로직을 실행해야 한다.

  **최종 접근:**
  1. 각 provider의 inputSchema에 humanAmount/humanSellAmount/humanAmountIn + decimals 필드를 optional로 추가
  2. inputSchema에 superRefine으로 XOR 검증 (amount OR humanAmount, 동시 지정 불가)
  3. humanAmount가 있으면 decimals 필수
  4. resolve() 시작에서 resolveProviderHumanAmount로 변환하여 기존 amount 필드에 값 설정
  5. 이후 기존 로직 그대로 실행
