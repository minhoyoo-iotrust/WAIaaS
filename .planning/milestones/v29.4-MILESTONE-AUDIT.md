### KPROV-08: Borrow/withdraw simulate HF and block self-liquidation

**Status: PASS**

`hf-simulation.ts` implements `simulateKaminoHealthFactor()` (pure function).
`index.ts` `checkBorrowSafety()` and `checkWithdrawSafety()` fetch obligation, simulate, throw if `!result.safe`.

Key behavior:
- Borrow: simulates increased debt, blocks if HF < `KAMINO_LIQUIDATION_THRESHOLD` (1.0)
- Withdraw: simulates decreased collateral, blocks if HF < threshold
- Max withdraw: skips HF simulation (closing position)
- No obligation: skips simulation (no existing position)
- SDK failure: graceful degradation (skips simulation)

Tests: `describe('HF simulation on borrow')` — 4 cases; `describe('HF simulation on withdraw')` — 3 cases.
Separate test file: `kamino-hf-simulation.test.ts` — 20 unit tests for `calculateHealthFactor`, `simulateKaminoHealthFactor`, `hfToStatus`.

### KINT-07: LendingPolicyEvaluator applies max_ltv_pct to Kamino borrow actions

**Status: FAIL**

**Issue:** `packages/daemon/src/pipeline/database-policy-engine.ts` line 1662:
```ts
if (transaction.actionName !== 'borrow') return null;
```

`ActionProviderRegistry.resolve()` auto-tags `actionName` with the full action name (`entry.action.name`). For Kamino, this is `'kamino_borrow'`, not `'borrow'`. The check `!== 'borrow'` never matches Kamino borrow actions, so `LENDING_LTV_LIMIT` policy is never evaluated.

Similarly, `LENDING_ASSET_WHITELIST` (line 1607-1609):
```ts
const LENDING_ACTIONS = new Set(['supply', 'borrow', 'repay', 'withdraw']);
if (!transaction.actionName || !LENDING_ACTIONS.has(transaction.actionName)) return null;
```

`'kamino_borrow'` is not in the set, so the default-deny whitelist check is also skipped for Kamino.

**Note:** This is also a pre-existing bug for Aave V3 (`'aave_borrow'` would also fail the check). The unit tests pass because `lendingTx()` constructs `TransactionParam` with bare action names directly, bypassing the ActionProviderRegistry tagging.

**Production impact:** Kamino borrow/supply/repay/withdraw actions bypass `LENDING_ASSET_WHITELIST` and `LENDING_LTV_LIMIT` policies entirely. They pass through the pipeline via provider-trust bypass of `CONTRACT_WHITELIST`, then skip lending-specific guards, then are classified as regular `CONTRACT_CALL` spending.

**Fix required:** In `evaluateLendingAssetWhitelist()` and `evaluateLendingLtvLimit()`, replace bare name matching with suffix-based or normalized matching:
```ts
// Instead of:
const LENDING_ACTIONS = new Set(['supply', 'borrow', 'repay', 'withdraw']);
if (!transaction.actionName || !LENDING_ACTIONS.has(transaction.actionName)) return null;

// Use:
const LENDING_VERBS = ['supply', 'borrow', 'repay', 'withdraw'];
const isLendingAction = LENDING_VERBS.some(v => transaction.actionName?.endsWith(v));
if (!isLendingAction) return null;
```

No tests in `kamino-lending-e2e.test.ts` cover the LENDING_LTV_LIMIT path for Kamino.
