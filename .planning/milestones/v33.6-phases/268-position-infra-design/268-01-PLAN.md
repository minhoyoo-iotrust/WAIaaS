**섹션 5.1: defi_positions 테이블 DDL**

- 칼럼 설계:
  - `id TEXT PRIMARY KEY` (UUID v7)
  - `wallet_id TEXT NOT NULL REFERENCES wallets(id) ON DELETE CASCADE`
  - `category TEXT NOT NULL` CHECK 제약: LENDING, YIELD, PERP, STAKING (SSoT 배열 POSITION_CATEGORIES에서 생성)
  - `provider TEXT NOT NULL` (aave_v3, kamino, lido, jito, pendle, drift 등)
  - `chain TEXT NOT NULL` CHECK 제약 (CHAIN_TYPES SSoT)
  - `network TEXT` CHECK 제약 (NETWORK_TYPES SSoT, nullable)
  - `asset_id TEXT` (CAIP-19 식별자, nullable)
  - `amount TEXT NOT NULL` (소수점 문자열)
  - `amount_usd REAL` (IPriceOracle 환산, nullable)
  - `metadata TEXT` (카테고리별 JSON, nullable)
  - `status TEXT NOT NULL DEFAULT 'ACTIVE'` CHECK 제약: ACTIVE, CLOSED, LIQUIDATED
  - `opened_at INTEGER NOT NULL` (Unix epoch seconds)
  - `closed_at INTEGER` (nullable)
  - `last_synced_at INTEGER NOT NULL`
  - `created_at INTEGER NOT NULL`
  - `updated_at INTEGER NOT NULL`
- 인덱스 전략: 4개 인덱스 정의 (wallet+category, wallet+provider, status, UNIQUE(wallet_id, provider, asset_id, category))
- CHECK 제약은 buildCheckSql() 패턴으로 SSoT 배열에서 생성

**섹션 6.1: IPositionProvider 인터페이스**
- @waiaas/core에 위치할 인터페이스 정의
- IActionProvider를 확장하지 않음 (IPositionProvider는 조회 전용, IActionProvider는 실행 전용)
- 메서드:
  - `getPositions(walletId: string): Promise<PositionUpdate[]>` -- 특정 월렛의 현재 포지션 상태를 온체인/API에서 조회
  - `getProviderName(): string` -- 프로바이더 식별자 반환
  - `getSupportedCategories(): PositionCategory[]` -- 지원 카테고리 반환
- PositionUpdate 타입: { walletId, category, provider, chain, network?, assetId?, amount, amountUsd?, metadata, status }
- IActionProvider와의 관계: 동일 프로바이더 클래스가 IActionProvider + IPositionProvider 모두 구현 가능 (예: AaveV3Provider implements IActionProvider, IPositionProvider)

**섹션 6.2: PositionTracker 스케줄러**
- 독립 서비스 클래스 (BackgroundWorkers 직접 등록이 아닌, BalanceMonitorService 패턴의 내부 setInterval)
- 카테고리별 차등 폴링 주기 테이블:
  - PERP: 60,000ms (1분) -- 마진 청산 위험으로 높은 빈도
  - LENDING: 300,000ms (5분) -- 헬스 팩터 모니터링
  - STAKING: 900,000ms (15분) -- 느린 변동
  - YIELD: 3,600,000ms (1시간) -- 만기 기반, 빈번한 변동 없음
- 오버랩 방지: 카테고리별 `running` 플래그로 이전 핸들러 완료 전 다음 tick 스킵 (Pitfall 4 대응)
- start()/stop() 라이프사이클: DaemonLifecycle Step 4에서 등록, shutdown에서 해제
- timer.unref() 호출로 프로세스 종료 차단 방지
- providers Map<string, IPositionProvider>: registerProvider(name, provider) / unregisterProvider(name)

**섹션 6.3: PositionWriteQueue (배치 쓰기 전략)**
- IncomingTxQueue 패턴 적용:
  - Map<string, PositionUpsert> 기반 메모리 큐 (키: `${walletId}:${provider}:${assetId}:${category}`)
  - MAX_BATCH = 100 (IncomingTxQueue와 동일)
  - flush()에서 prepared statement + SQLite transaction(BEGIN IMMEDIATE) 사용
  - ON CONFLICT(wallet_id, provider, asset_id, category) DO UPDATE SET amount=?, amount_usd=?, metadata=?, status=?, last_synced_at=?, updated_at=?
- 플러시 주기: PositionTracker의 각 syncCategory() 완료 후 즉시 flush() 호출 (BackgroundWorkers 주기와 독립)
- Map 키를 통한 자동 중복 제거: 동일 포지션의 다중 업데이트는 최신 값만 유지

**섹션 6.4: 데몬 라이프사이클 연동**
- DaemonLifecycle 연동 지점 명시:
  - Step 4 (after provider registration): PositionTracker.start()
  - Shutdown hook: PositionTracker.stop() + PositionWriteQueue.flush() (잔여 큐 소진)
- 설정 연동: config.toml [position_tracker] 섹션 (enabled, intervals.perp, intervals.lending, intervals.staking, intervals.yield)
- Admin Settings 런타임 오버라이드: position_tracker.* 카테고리로 5개 키 등록 (enabled + 4개 interval)
- EventBus 연동: POSITION_UPDATED / POSITION_OPENED / POSITION_CLOSED 이벤트 발행 (NotificationService에서 소비)

**설계 결정 기록:**
- IPositionProvider가 IActionProvider를 확장하지 않는 이유 (조회 vs 실행 관심사 분리)
- PositionTracker가 BackgroundWorkers에 등록하지 않고 독립 서비스인 이유 (다중 타이머 필요)
- 플러시를 별도 타이머가 아닌 syncCategory() 완료 후 즉시 실행하는 이유 (지연 최소화)
