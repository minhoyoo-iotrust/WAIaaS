<objective>
Integrate RpcPool as the RPC URL resolution layer in AdapterPool and seed it from config.toml single-URL settings and environment variables at daemon startup.

Purpose: AdapterPool currently receives a raw RPC URL string on each resolve() call. This plan wires RpcPool into AdapterPool so it can leverage priority-based fallback and cooldown. Config.toml backward compatibility is maintained by seeding the pool with existing single-URL settings as 1-entry pools.

Output: Modified AdapterPool with RpcPool dependency, daemon startup seeding, tests.

**AdapterPool changes (adapter-pool.ts):**

1. Add `RpcPool` import from `@waiaas/core`.

2. Make AdapterPool accept an optional `RpcPool` in its constructor:
   ```ts
   constructor(private readonly rpcPool?: RpcPool) {}
   ```

3. Modify `resolve()` signature to make `rpcUrl` optional (third parameter defaults to `''`):
   ```ts
   async resolve(chain: ChainType, network: NetworkType, rpcUrl?: string): Promise<IChainAdapter>
   ```

4. In `resolve()`, derive the actual RPC URL:
   - If `this.rpcPool` is available, call `this.rpcPool.getUrl(network as string)` to get the URL. If that throws (no endpoints registered), fall back to the provided `rpcUrl`.
   - If `this.rpcPool` is NOT available (legacy path), use the provided `rpcUrl` as-is.
   - This ensures backward compatibility: all existing `adapterPool.resolve(chain, network, rpcUrl)` callers work unchanged.

5. Add a public method `reportRpcFailure(network: string, url: string)` that delegates to `this.rpcPool?.reportFailure(network, url)`.

6. Add a public method `reportRpcSuccess(network: string, url: string)` that delegates to `this.rpcPool?.reportSuccess(network, url)`.

7. Add a public getter `pool` that returns the RpcPool instance (or undefined) for inspection by other services.

**IMPORTANT DESIGN DECISION -- SolanaAdapter withRpcRetry vs RpcPool cooldown:**
The existing SolanaAdapter.withRpcRetry does 3 retries with 1-10s backoff on the SAME URL. RpcPool cooldown is 60s+ and switches to a DIFFERENT URL. These are complementary, not conflicting:
- withRpcRetry: short-term retry on transient errors (same endpoint)
- RpcPool cooldown: long-term rotation when an endpoint is persistently failing
The adapter-level retry runs first. If all 3 retries fail, the error propagates up. The caller (pipeline stage or route handler) can then report failure to RpcPool and retry with a new URL from the pool. No changes to SolanaAdapter.withRpcRetry needed.

**IMPORTANT DESIGN DECISION -- viem fallback transport vs RpcPool:**
viem's built-in `fallback()` transport creates a client that internally rotates URLs. However, using RpcPool at a higher level provides:
- Unified pool state across both SolanaAdapter and EvmAdapter
- Pool-level cooldown visible in Admin UI monitoring (Phase 264)
- Consistent config/settings management
Therefore: EvmAdapter.connect() continues to use `http(rpcUrl)` (single URL). If the URL fails, the caller reports to RpcPool and creates a new adapter with the next URL (via evict+resolve pattern already in hot-reload). No viem fallback transport needed.

**Daemon startup changes (daemon.ts):**

1. Import `RpcPool` from `@waiaas/core`.

2. In the `DaemonLifecycle` class, add a private field `private rpcPool: RpcPool | null = null;`.

3. In Step 4 (adapter initialization), before creating AdapterPool:
   a. Create `this.rpcPool = RpcPool.createWithDefaults()` -- loads 13 built-in networks.
   b. Seed config.toml URLs: iterate over `this._config.rpc` entries. For each key like `solana_mainnet`, `evm_ethereum_sepolia`, etc., extract the network name and prepend that URL to the pool using `this.rpcPool.register(networkName, [url])`. The `register()` method deduplicates, so if the config URL matches a built-in default, it won't create a duplicate -- but since register() appends, the config URL needs to be registered FIRST (before createWithDefaults). So instead:
      - Create `this.rpcPool = new RpcPool()`.
      - Seed config.toml URLs first (highest priority): For each config.rpc entry that is a network URL (not `evm_default_network`), map the config key to network name and register `[url]`.
      - Then register built-in defaults: `for (const [network, urls] of Object.entries(BUILT_IN_RPC_DEFAULTS)) { this.rpcPool.register(network, [...urls]); }`. Since register() deduplicates and appends, built-in URLs come after config URLs in priority.

   Config key to network name mapping (reuse existing pattern from rpcConfigKey):
   - `solana_mainnet` -> `mainnet`
   - `solana_devnet` -> `devnet`
   - `solana_testnet` -> `testnet`
   - `evm_ethereum_mainnet` -> `ethereum-mainnet`
   - `evm_ethereum_sepolia` -> `ethereum-sepolia`
   - `evm_polygon_mainnet` -> `polygon-mainnet`
   - etc.

   The reverse mapping function (`configKeyToNetwork`): strip `solana_` prefix for Solana networks, strip `evm_` prefix and replace `_` with `-` for EVM networks. Add this as an exported helper in adapter-pool.ts.

4. Pass the RpcPool to AdapterPool: `this.adapterPool = new AdapterPool(this.rpcPool)`.

5. Make rpcPool accessible for IncomingTxMonitor (Plan 261-03) via a getter: `get rpcPoolInstance(): RpcPool | null { return this.rpcPool; }`.

**Environment variable override for CONF-04:**
The existing `applyEnvOverrides` in config/loader.ts already handles `WAIAAS_RPC_*` env vars by writing them into the config.rpc object. Since we seed from config.rpc (which includes env overrides), CONF-04 is automatically satisfied. No additional changes needed in loader.ts.
