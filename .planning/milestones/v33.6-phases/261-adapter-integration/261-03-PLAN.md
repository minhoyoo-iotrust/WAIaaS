**Updated pattern:**
```ts
const subscriberFactory = async (chain: string, network: string) => {
  const sSvc = this._settingsService!;

  // Resolve RPC URL: prefer RpcPool, fallback to SettingsService
  const resolveSubscriberRpcUrl = (chain: string, network: string): string => {
    if (this.rpcPool) {
      try {
        return this.rpcPool.getUrl(network);
      } catch {
        // Pool has no entry or all in cooldown -- fall back to settings
      }
    }
    return sSvc.get(`rpc.${rpcConfigKey(chain, network)}`);
  };

  if (chain === 'solana') {
    const rpcUrl = resolveSubscriberRpcUrl(chain, network);
    const wssUrl = sSvc.get('incoming.wss_url') || rpcUrl.replace(/^https:\/\//, 'wss://');
    const { SolanaIncomingSubscriber } = await import('@waiaas/adapter-solana');
    return new SolanaIncomingSubscriber({ rpcUrl, wsUrl: wssUrl });
  }

  // EVM chains
  const rpcUrl = resolveSubscriberRpcUrl(chain, network);
  const { EvmIncomingSubscriber } = await import('@waiaas/adapter-evm');
  const ns = this.notificationService;
  return new EvmIncomingSubscriber({
    rpcUrl,
    onRpcAlert: ns ? (alert) => {
      ns.notify(alert.type, alert.walletId, {
        network: alert.network,
        errorCount: String(alert.errorCount),
        lastError: alert.lastError,
        ...(alert.fromBlock ? { fromBlock: alert.fromBlock } : {}),
        ...(alert.toBlock ? { toBlock: alert.toBlock } : {}),
      });
    } : undefined,
  });
};
```

**Key points:**
- `resolveSubscriberRpcUrl` is a local helper within the subscriberFactory closure
- It tries RpcPool.getUrl(network) first
- Falls back to SettingsService `rpc.{key}` if pool unavailable or throws
- WSS URL derivation continues from the resolved HTTP URL (same as before)
- The EvmIncomingSubscriber's `onRpcAlert` callback is unchanged

**No changes needed to IncomingTxMonitorService or SubscriptionMultiplexer:**
The subscriberFactory is a dependency-injected callback. Changing what it does internally (using RpcPool vs SettingsService) doesn't affect the monitor service's logic. The monitor still calls subscriberFactory(chain, network) and gets back an IChainSubscriber.

**RPC failure reporting from IncomingTxMonitor:**
When EvmIncomingSubscriber encounters RPC errors, it already has per-wallet backoff logic (errorCount, backoffUntil). For RpcPool integration, we could add a callback to report failures to the pool. However, this is better deferred to Phase 264 (Monitoring + Alerts) since:
- The subscriber creates its own viem PublicClient at construction time
- Rotating URLs mid-polling would require destroying and recreating the subscriber
- The current subscriber backoff is sufficient for transient errors
- Pool-level rotation happens naturally when the subscription multiplexer recreates subscribers (e.g., after destroy+reconnect cycle)

For now, the RpcPool URL is used at subscriber creation time only. This is consistent with how AdapterPool works (URL used at connect() time, cached after that).

```ts
export function resolveRpcUrlFromPool(
  rpcPool: RpcPool | null | undefined,
  settingsGet: (key: string) => string,
  chain: string,
  network: string,
): string {
  if (rpcPool) {
    try {
      return rpcPool.getUrl(network);
    } catch {
      // Pool unavailable or all in cooldown -- fall back
    }
  }
  return settingsGet(`rpc.${rpcConfigKey(chain, network)}`);
}
```

Add this to adapter-pool.ts and use it in both the subscriberFactory and the test. Then the daemon.ts subscriberFactory becomes:
```ts
const rpcUrl = resolveRpcUrlFromPool(this.rpcPool, sSvc.get.bind(sSvc), chain, network);
```

This makes the URL resolution testable without mocking the entire DaemonLifecycle.
