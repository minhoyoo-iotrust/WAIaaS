**Section 13: ILendingProvider 인터페이스**

13.1 인터페이스 정의 — ILendingProvider extends IActionProvider (from packages/core/src/interfaces/action-provider.types.ts):
- Inherits: metadata, actions, resolve() from IActionProvider
- Adds 3 query methods:
  - `getPosition(walletId: string, context: ActionContext): Promise<LendingPositionSummary[]>` — current lending positions for a wallet
  - `getHealthFactor(walletId: string, context: ActionContext): Promise<HealthFactor>` — health factor (collateral/debt ratio) for a wallet
  - `getMarkets(chain: string, network?: string): Promise<MarketInfo[]>` — available lending markets
- File path: `packages/core/src/interfaces/lending-provider.types.ts`
- Re-export from `packages/core/src/interfaces/index.ts`

13.2 ActionDefinition 4개 — define the standard lending actions with input schemas:
- `supply`: { asset (CAIP-19), amount (string, human-readable), onBehalfOf? (address) } — risk: medium, defaultTier: DELAY
- `borrow`: { asset (CAIP-19), amount (string), interestRateMode? (1|2, default 2=variable) } — risk: high, defaultTier: APPROVAL
- `repay`: { asset (CAIP-19), amount (string), interestRateMode? } — risk: low, defaultTier: INSTANT
- `withdraw`: { asset (CAIP-19), amount (string), to? (address) } — risk: medium, defaultTier: DELAY
- Each action.chain is set per-provider (EVM for Aave/Morpho, solana for Kamino)

13.3 IPositionProvider 동시 구현 패턴:
- Each lending provider class implements BOTH ILendingProvider AND IPositionProvider (from Phase 268)
- `IPositionProvider.getPositions()` returns PositionUpdate[] for PositionTracker write queue
- `ILendingProvider.getPosition()` returns richer LendingPositionSummary[] for API/AI agent
- Same underlying RPC calls serve both interfaces; return types differ by purpose
- Pattern example: `class AaveV3Provider implements ILendingProvider, IPositionProvider { ... }`

13.4 설계 결정 기록 — Record decisions:
- DEC-LEND-01: ILendingProvider extends IActionProvider (not separate interface) — keeps resolve() in same class for 6-stage pipeline
- DEC-LEND-02: 3 query methods added directly to ILendingProvider (not via separate service) — avoids service lookup indirection, provider knows its own data best
- DEC-LEND-03: borrow action defaults to interestRateMode=2 (variable) — Aave V3 stable rate deprecated on most markets
- DEC-LEND-04: resolve() returns ContractCallRequest | ContractCallRequest[] — multi-step approve+supply pattern (same as LidoStaking)

**Section 14: LendingPosition + HealthFactor 타입**

14.1 LendingPosition Zod 스키마:
- LendingPositionSchema extends Phase 268's LendingMetadataSchema concept
- Fields: positionType (z.enum(['SUPPLY', 'BORROW'])), provider (string), asset (CAIP-19 string), symbol (string), amount (string, raw units), amountUsd (number), apy (number, decimal e.g., 0.032), interestRateMode ('STABLE' | 'VARIABLE', optional), chain (ChainTypeEnum), network (string)
- This schema is for API response; the defi_positions table uses the metadata JSON column

14.2 HealthFactor 타입:
- HealthFactorSchema = z.object:
  - healthFactor: z.number() — collateral/debt ratio (> 1 = safe, < 1 = liquidatable), null if no debt
  - totalCollateralUsd: z.number()
  - totalDebtUsd: z.number()
  - availableBorrowUsd: z.number()
  - currentLtv: z.number() — current LTV (decimal, e.g., 0.65)
  - maxLtv: z.number() — max allowed LTV before liquidation
  - status: z.enum(['SAFE', 'WARNING', 'DANGER', 'CRITICAL', 'NO_POSITIONS']) — maps to Phase 269 MonitorSeverity
  - positions: z.array(LendingPositionSummarySchema) — per-asset breakdown
- Threshold mapping: SAFE (hf >= 2.0), WARNING (1.5 <= hf < 2.0), DANGER (1.2 <= hf < 1.5), CRITICAL (hf < 1.2)
- Note: These thresholds align with Phase 269 HealthFactorMonitor severity levels

14.3 MarketInfo 타입:
- MarketInfoSchema = z.object:
  - marketId (string, protocol-specific), asset (CAIP-19), symbol (string)
  - supplyApy (number), borrowApy (number)
  - totalSupply (string), totalBorrow (string)
  - ltv (number, max LTV for this market), liquidationThreshold (number)
  - isActive (boolean), chain (ChainTypeEnum), network (string)
  - provider (string, provider name)

14.4 설계 결정 기록:
- DEC-LEND-05: HealthFactor status enum maps 1:1 with Phase 269 MonitorSeverity — enables consistent alert classification
- DEC-LEND-06: amount fields use string (not bigint) for JSON serialization — bigint serialization requires custom handling
- DEC-LEND-07: CAIP-19 asset identifiers used throughout (not raw addresses) — consistent with Phase 232 CAIP-19 integration

**Section 15: LendingPolicyEvaluator**

15.1 LENDING_LTV_LIMIT 정책 타입:
- New policy type added to DatabasePolicyEngine's policy type registry
- Rules JSON schema (LendingLtvLimitRules):
  - maxLtv: number (maximum allowed LTV ratio, e.g., 0.75 = 75%)
  - warningLtv: number (warning threshold, e.g., 0.65 = 65%)
- Evaluation logic:
  - Only applies to `borrow` actions (not supply/repay/withdraw)
  - Computes projected LTV: (currentDebtUsd + newBorrowAmountUsd) / totalCollateralUsd
  - If projectedLtv > maxLtv → deny with reason "Borrow would exceed max LTV ({maxLtv})"
  - If projectedLtv > warningLtv → allow but upgrade tier to DELAY (force human review)
  - Current position data read from defi_positions table (cached by PositionTracker)
- Integration point: DatabasePolicyEngine.evaluate() step 4h (after APPROVE_TIER_OVERRIDE step 4g)
- Wallet + global scoping: same priority resolution as existing policy types (wallet+network > wallet+null > global+network > global+null)

15.2 LENDING_ASSET_WHITELIST 정책 타입:
- Default-deny policy following CONTRACT_WHITELIST pattern (per CLAUDE.md: "Contracts default-deny (CONTRACT_WHITELIST opt-in)")
- Rules JSON schema (LendingAssetWhitelistRules):
  - collateralAssets: Array<{ assetId: string (CAIP-19), symbol?: string }> — allowed assets for supply/withdraw
  - borrowAssets: Array<{ assetId: string (CAIP-19), symbol?: string }> — allowed assets for borrow/repay
- Evaluation logic:
  - When no LENDING_ASSET_WHITELIST policy exists → deny ALL lending actions (default-deny)
  - For supply/withdraw actions: check asset against collateralAssets list
  - For borrow/repay actions: check asset against borrowAssets list
  - Asset matching uses CAIP-19 identifier comparison (exact match)
- Integration point: DatabasePolicyEngine.evaluate() step 4h (before LENDING_LTV_LIMIT check)
- Order: asset whitelist first (4h-a), then LTV limit (4h-b) — deny invalid assets before computing LTV

15.3 PolicyEngine 통합 지점:
- Add LENDING_LTV_LIMIT and LENDING_ASSET_WHITELIST to the policy type union in Zod SSoT
- Existing policies table schema (type, rules JSON, wallet_id, priority, enabled, network) fully reused
- Lending action identification: use ContractCallRequest.metadata field to carry `{ actionProvider: string, actionName: string }`. Policy evaluator checks metadata.actionName is one of ['supply', 'borrow', 'repay', 'withdraw'] to identify lending actions. This avoids adding new types to the discriminatedUnion 5-type.
- Admin UI policies page: extend the policy type dropdown to include LENDING_LTV_LIMIT and LENDING_ASSET_WHITELIST. The rules editor renders appropriate form fields (LTV slider for LTV limit, asset picker for whitelist).
- Document the policy evaluation order clearly:
  1. Existing checks (4a-4g): WHITELIST, ALLOWED_NETWORKS, ALLOWED_TOKENS, CONTRACT_WHITELIST, METHOD_WHITELIST, APPROVED_SPENDERS, APPROVE_AMOUNT_LIMIT, APPROVE_TIER_OVERRIDE
  2. New checks (4h): LENDING_ASSET_WHITELIST (4h-a) → LENDING_LTV_LIMIT (4h-b)
  3. Existing check (5): SPENDING_LIMIT

15.4 HealthFactor 정책-모니터 임계값 관계:
- Clearly distinguish: LendingPolicyEvaluator.maxLtv prevents new borrows exceeding LTV ratio, while HealthFactorMonitor thresholds alert on existing position degradation
- Conversion formula: healthFactor = liquidationThreshold / currentLtv
- Example: if maxLtv=0.75 and liquidationThreshold=0.825, the resulting minimum health factor at max borrow = 0.825/0.75 = 1.10 (which is in DANGER range)
- Recommendation: Set warningLtv such that resulting health factor stays in SAFE range (>= 2.0). This ensures policy-approved borrows don't immediately trigger monitoring alerts.

15.5 설계 결정 기록:
- DEC-LEND-08: LENDING_ASSET_WHITELIST is default-deny — follows CLAUDE.md contract whitelist pattern
- DEC-LEND-09: Lending action identification via ContractCallRequest.metadata (not new discriminatedUnion type) — preserves 5-type pipeline integrity
- DEC-LEND-10: LTV check reads from defi_positions cache (not live RPC) — avoids blocking pipeline with RPC latency; PositionTracker ensures data freshness
- DEC-LEND-11: warningLtv forces DELAY tier (not deny) — allows borrow with human review for borderline cases
