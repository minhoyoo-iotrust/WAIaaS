1. packages/daemon/src/infrastructure/database/migrations/v62.ts 생성 (per INFRA-07):
   - SQLite는 CHECK 제약조건을 ALTER TABLE로 수정할 수 없으므로 12-step table recreation 사용
   - managesOwnTransaction: true (foreign_keys OFF 필요)
   - 영향받는 테이블: wallets, transactions, incoming_transactions, defi_positions
   - 각 테이블에 대해:
     a. PRAGMA foreign_keys = OFF
     b. BEGIN
     c. CREATE TABLE {table}_new (...) — chain CHECK에 'ripple' 포함
     d. INSERT INTO {table}_new SELECT * FROM {table}
     e. DROP TABLE {table}
     f. ALTER TABLE {table}_new RENAME TO {table}
     g. 인덱스/트리거 재생성
     h. COMMIT
     i. PRAGMA foreign_keys = ON
   - 중요: getCreateTableStatements()의 DDL과 동일한 컬럼/CHECK 조합을 사용
   - 기존 데이터 보존이 핵심 — SELECT * 복사로 데이터 손실 방지
   - transactions 테이블의 복잡한 CHECK/인덱스도 모두 재생성 필요

   주의: schema-ddl.ts에서 inList(CHAIN_TYPES)를 사용하므로 Plan 01이 완료되면 CHAIN_TYPES에 'ripple'이 이미 포함됨. 하지만 migration v62에서는 하드코딩된 CHECK 값을 사용해야 함 (마이그레이션 코드는 런타임 상수에 의존하지 않아야 하므로). 실제로는 CHAIN_TYPES를 import해서 사용하는 것이 이 프로젝트의 패턴이므로 (schema-ddl.ts가 그렇게 함) 동일 패턴을 따른다.

   더 간단한 접근: SQLite CHECK 제약조건 대신 트리거로 대체하는 방법도 있으나, 이 프로젝트는 테이블 재생성 패턴을 사용해왔으므로 그 패턴을 따른다. 단, 테이블이 4개로 많으므로 헬퍼 함수를 만들어 중복을 줄인다.
