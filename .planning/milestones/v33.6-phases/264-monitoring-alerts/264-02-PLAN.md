2. In `packages/daemon/src/lifecycle/daemon.ts`:

   Find the daemon initialization step where `RpcPool.createWithDefaults()` is called (or where `new RpcPool()` is constructed). This is in the adapter pool setup section.

   a. Pass an `onEvent` callback to the RpcPool constructor options:
   ```typescript
   onEvent: (event) => {
     // RPC pool health notifications -- use SYSTEM_WALLET_ID as walletId
     // since these are infrastructure-level alerts, not wallet-specific.
     if (this.notificationService) {
       const vars: Record<string, string> = {
         network: event.network,
         url: event.url,
         errorCount: String(event.failureCount),
         totalEndpoints: String(event.totalEndpoints),
       };
       void this.notificationService.notify(
         event.type as import('@waiaas/core').NotificationEventType,
         'system',  // system-level alert, not wallet-specific
         vars,
       );
     }
   }
   ```

   b. The `event.type` values ('RPC_HEALTH_DEGRADED', 'RPC_ALL_FAILED', 'RPC_RECOVERED') are all valid NotificationEventType values after Plan 264-01.

   c. Use `'system'` as walletId since these are infrastructure alerts not tied to a specific wallet. The NotificationService.notify accepts any string walletId for lookup purposes.

   **IMPORTANT:** The existing `RPC_HEALTH_DEGRADED` notification from `EvmIncomingSubscriber.onRpcAlert` is a different trigger point (subscriber-level consecutive errors). The new RpcPool-level notification fires on individual cooldown entry. Both coexist without conflict -- different source, different semantics. The EvmIncomingSubscriber tracks per-wallet consecutive polling failures; the RpcPool tracks per-URL cooldown transitions.
