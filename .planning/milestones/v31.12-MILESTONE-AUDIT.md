## 4. Critical Findings

### FINDING-01: `bootstrapSignerCapabilities()` never called at startup [CRITICAL]

**Location:** `packages/daemon/src/lifecycle/daemon.ts` lines 1475-1517

**Issue:** `createApp()` is called without `signerRegistry` in the deps object. The `SignerCapabilityRegistry` is never instantiated, and `bootstrapSignerCapabilities()` is never called during daemon startup. The `signerRegistry` field in `CreateAppDeps` is optional (`signerRegistry?`), so TypeScript does not flag this as an error.

**Impact:**
- `external_actions` capability never appears in connect-info (gated on `signerRegistry?.listSchemes().length > 0`)
- Kind-based routing in `actions.ts` returns "External action pipeline not configured (credentialVault/signerRegistry missing)" for any signedData/signedHttp action
- The entire external action pipeline is non-functional at runtime despite all unit tests passing

**Affected Requirements:** PIPE-01, PIPE-02, PIPE-03, PIPE-04, PIPE-07, SIGN-07 (all depend on signerRegistry being available)

**Fix:** Add to `daemon.ts` before `createApp()`:
```typescript
const { SignerCapabilityRegistry, bootstrapSignerCapabilities } = await import('../signing/index.js');
const signerRegistry = new SignerCapabilityRegistry();
bootstrapSignerCapabilities(signerRegistry);
```
Then pass `signerRegistry` in the `createApp()` deps object.

### FINDING-02: 10 lint errors in Phase 387 test files [BLOCKING]

**Location:**
- `packages/daemon/src/__tests__/signer-capabilities.test.ts` (6 errors: unused `vi`, `privateKeyToAccount`, `SigningParams`, `SigningResult`, `SigningErrorCode`, `ed25519PublicKey`)
- `packages/daemon/src/__tests__/signer-registry.test.ts` (4 errors: unused `PersonalSignCapability`, `RsaPssSignerCapability`, `EcdsaSignBytesCapability`, `Ed25519SignBytesCapability`)

**Impact:** CI will fail on lint check. Per CLAUDE.md: "Run `pnpm turbo run lint` and `pnpm turbo run typecheck` before creating the PR."

**Fix:** Remove unused imports or prefix with `_`.

### FINDING-03: TRACK-02 column rename not performed [DEVIATION]

**Location:** `packages/daemon/src/infrastructure/database/migrate.ts`, `schema.ts`

**Issue:** The objective document (R4-2) and REQUIREMENTS.md (TRACK-02) specify renaming `bridge_status`/`bridge_metadata` to `tracking_status`/`tracking_metadata`. This was not done. Instead, the implementation expanded `BRIDGE_STATUS_VALUES` from 6 to 11 values to include the new external action states. Column names remain `bridge_status`/`bridge_metadata`.

**Impact:** Lower risk approach (no rename means no risk of breaking existing bridge trackers). The columns work for both bridge and external action tracking. However, the naming is now misleading since "bridge_status" holds non-bridge states like PARTIALLY_FILLED.

**Severity:** LOW -- intentional design deviation, documented in Phase 389-01 summary.

### FINDING-04: ExternalActionTracker class not implemented (TRACK-03) [DEVIATION]

**Issue:** Instead of creating a dedicated `ExternalActionTracker` class implementing `IAsyncStatusTracker`, the AsyncPollingService was directly extended to handle the new states. The `checkStatus()` method is defined on `IActionProvider` (TRACK-04 fulfilled) but no mediator class delegates to it.

**Impact:** LOW for current milestone (no actual off-chain ActionProviders exist yet). When off-chain providers are added in future milestones, the polling integration will need either an ExternalActionTracker class or direct wiring in AsyncPollingService.

### FINDING-05: EventBus events for external actions not registered (TRACK-05) [DEVIATION]

**Issue:** The objective requires `action:status-changed`, `action:completed`, `action:failed` EventBus events. These were not added to `packages/core/src/events/event-types.ts`. The implementation uses notification callbacks (`emitNotification()`) instead.

**Impact:** LOW -- webhook consumers and autostop rules will not react to external action status changes. Since no off-chain providers exist yet, no user-visible impact.

### FINDING-06: Objective file status not updated [DOCUMENTATION]

**Location:** `internal/objectives/m31-12-external-action-impl.md`

**Issue:** Status is still `PLANNED`, should be `SHIPPED` per CLAUDE.md: "Update objective status on milestone completion."

### FINDING-07: REQUIREMENTS.md checkbox mismatch [DOCUMENTATION]

**Issue:** TRACK-01, TRACK-04, TRACK-06 are implemented in code but marked `[ ]` (unchecked) in REQUIREMENTS.md. These should be `[x]`.

## 9. Architecture Quality

### Strengths
- Clean separation of concerns: types (core) -> capabilities (daemon/signing) -> vault (daemon/credential) -> pipeline (daemon/pipeline) -> API routes
- Backward compatibility maintained: 13 existing ActionProviders unchanged, existing pipelines untouched
- Security model correct: masterAuth for credential writes, sessionAuth for reads only, values never in responses
- HKDF domain separation prevents key reuse between credential vault and settings encryption
- AAD binding prevents cross-credential substitution attacks

### Concerns
- The `signerRegistry` optional pattern masks the wiring gap (FINDING-01). Consider making it required when external action features are enabled
- Column names `bridge_status`/`bridge_metadata` are now semantically incorrect for external action tracking
