From packages/actions/src/providers/aave-v3/aave-rpc.ts:
```typescript
export interface IRpcCaller {
  call(params: { to: string; data: string; chainId: number }): Promise<string>;
}
```

Current Aave V3 getPositions: Only queries 'ethereum-mainnet' hardcoded. Uses this.rpcCaller for all calls. The rpcCaller.call() already accepts chainId parameter, so multichain is structurally possible.

Key insight: IRpcCaller.call() takes chainId. For multichain, we need to either:
(a) Create per-network rpcCallers, or (b) use raw fetch with ctx.rpcUrls like Lido.
Since IRpcCaller abstracts RPC and already takes chainId, and the provider uses this.rpcCaller from constructor, the simplest approach is: for each supported network in ctx.networks, create a raw-fetch-based caller using ctx.rpcUrls[network], OR extend the existing pattern.

Decision: Use raw fetch with ctx.rpcUrls[network] for multichain position queries (consistent with Lido pattern). Keep this.rpcCaller for single-network ILendingProvider methods (getPosition, getHealthFactor).
