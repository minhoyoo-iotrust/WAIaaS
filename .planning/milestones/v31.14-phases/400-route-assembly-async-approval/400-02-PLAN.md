**1. Add `from` field validation in `rpc-proxy.ts` route handler** (SEC-02, SEC-03):

Before dispatching `eth_sendTransaction` or `eth_signTransaction` requests, validate the `from` field:

In the route handler, after parsing the JSON-RPC body and before dispatching:
- For single requests where method is `eth_sendTransaction`, `eth_signTransaction`, `personal_sign`, `eth_sign`, or `eth_signTypedData_v4`:
  - Extract `params[0]` for tx methods, check the `from` field.
  - **SEC-03 Auto-fill**: If `from` is undefined/null/empty, set `params[0].from = wallet.publicKey` (the checksummed address).
  - **SEC-02 Mismatch check**: If `from` is present and `from.toLowerCase() !== wallet.publicKey.toLowerCase()`, return `jsonRpcError(id, JSON_RPC_ERRORS.INVALID_PARAMS, 'from address does not match session wallet')`.
  - For `personal_sign` params: [message, address] -- validate `address` (params[1]) matches if present, auto-fill if missing.
  - For `eth_sign` params: [address, message] -- validate `address` (params[0]) matches if present.
  - For `eth_signTypedData_v4` params: [address, typedData] -- validate `address` (params[0]) matches if present.

Implement this as a helper function `validateAndFillFrom(method: string, params: unknown[], walletAddress: string): JsonRpcErrorResponse | null` that mutates params in-place for auto-fill and returns an error response if validation fails, or null if OK.

**2. Async timeout error formatting** (ASYNC-04):

The SyncPipelineExecutor already handles long-poll via CompletionWaiter. The timeout error from CompletionWaiter includes the txId in the message. In the `dispatch()` or route handler catch block, when the error message contains "timed out":

```typescript
catch (err) {
  const message = err instanceof Error ? err.message : 'Transaction execution failed';
  // ASYNC-04: Include txId in timeout error data
  const txIdMatch = message.match(/Transaction ([a-f0-9-]+) timed out/);
  if (txIdMatch) {
    return jsonRpcError(id, JSON_RPC_ERRORS.SERVER_ERROR, message, { txId: txIdMatch[1] });
  }
  return jsonRpcError(id, JSON_RPC_ERRORS.SERVER_ERROR, message);
}
```

Note: `jsonRpcError` already accepts optional `data` parameter. If not, add it (check json-rpc.ts -- the `data` field is part of JsonRpcError interface).
