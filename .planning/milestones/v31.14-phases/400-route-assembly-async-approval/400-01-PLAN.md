**1. Create `packages/daemon/src/rpc-proxy/dispatcher.ts` -- RpcDispatcher class:**

```typescript
export interface RpcDispatcherDeps {
  methodHandlers: RpcMethodHandlers;
  passthrough: RpcPassthrough;
  nonceTracker: NonceTracker;
}
```

- Constructor takes `RpcDispatcherDeps`.
- `async dispatch(request: JsonRpcRequest, ctx: HandlerContext, pipelineDeps: any): Promise<JsonRpcResponse>`:
  1. Call `classifyMethod(request.method, this.passthrough)` for 3-way classification.
  2. `'intercept'` -> `this.methodHandlers.handle(request.method, request.params as unknown[], request.id ?? null, ctx, pipelineDeps)`
  3. `'passthrough'` -> `this.passthrough.forward(request, network)` where network is `ctx.network`
  4. `'unsupported'` -> `jsonRpcError(request.id ?? null, JSON_RPC_ERRORS.METHOD_NOT_FOUND, 'Method not found: ' + request.method)`
- `async dispatchBatch(requests: JsonRpcRequest[], ctx: HandlerContext, pipelineDeps: any): Promise<JsonRpcResponse[]>`:
  1. Map each request through `dispatch()` with `Promise.all()`.
  2. Filter out notifications (requests where `isNotification(request)` is true -- they produce no response per JSON-RPC 2.0 spec).
  3. Return array of responses.

Route handler for `POST /rpc-evm/:walletId/:chainId`:
1. **Content-Type check** (RPC-06): If `Content-Type` header is not `application/json`, return 415 JSON-RPC error response (not Hono error -- still JSON-RPC format): `jsonRpcError(null, JSON_RPC_ERRORS.INVALID_REQUEST, 'Content-Type must be application/json')` with status 415.
2. **Parse body** via `parseJsonRpcBody(await c.req.json())`. If `type: 'error'`, return the error response immediately.
3. **Resolve walletId**: Extract `:walletId` param. Call `verifyWalletAccess(sessionId, walletId, deps.db)` to confirm session has access.
4. **Resolve chainId**: Extract `:chainId` param, parse to number. Look up in `EVM_CHAIN_ID_TO_NETWORK`. If not found, return `jsonRpcError(id, JSON_RPC_ERRORS.INVALID_PARAMS, 'Unknown chainId: ' + chainId)`.
5. **Look up wallet**: Query `wallets` table by `walletId`, get `publicKey` (which IS the EVM address), `chain`, `environment`, `accountType`, `aaProvider`, etc.
6. **Check rpc_proxy.enabled** via SettingsService: if `'false'`, return `jsonRpcError(null, JSON_RPC_ERRORS.SERVER_ERROR, 'RPC proxy is disabled')` with status 503.
7. **Build HandlerContext**:
   ```typescript
   const ctx: HandlerContext = {
     walletId, walletAddress: wallet.publicKey, chainId: numericChainId,
     network: resolvedNetwork, chain: wallet.chain, sessionId,
   };
   ```
8. **Build pipeline deps** (same pattern as `transactions.ts`): resolve adapter from AdapterPool, build the full deps object needed by RpcMethodHandlers.
9. **Construct RpcDispatcher** with lazy-initialized components (or construct per-request -- they are lightweight):
   - `CompletionWaiter(deps.eventBus)` -- construct once per route factory and reuse (has global EventBus listeners).
   - `SyncPipelineExecutor(completionWaiter, deps.settingsService)` -- construct once.
   - `RpcTransactionAdapter()` -- stateless, construct once.
   - `NonceTracker()` -- construct once (per-address state shared across requests).
   - `RpcMethodHandlers(syncPipeline, txAdapter, nonceTracker)` -- construct once.
   - `RpcPassthrough(rpcPool)` -- needs per-network rpcPool, may need to get from adapterPool. Use `deps.adapterPool.pool` if available, or create a simple fetch-based forwarding using `resolveRpcUrl`.
   - `RpcDispatcher({ methodHandlers, passthrough, nonceTracker })` -- construct once.

   **Important**: CompletionWaiter, SyncPipelineExecutor, NonceTracker, and RpcMethodHandlers should be constructed ONCE in the route factory function (not per-request). RpcPassthrough needs the RPC URL which varies per chainId, so either construct per-request or use a passthrough that accepts network at forward time (it already does: `forward(request, network)` takes network param).

10. **Dispatch**: For `type: 'single'`, call `dispatcher.dispatch(...)`. For `type: 'batch'`, call `dispatcher.dispatchBatch(...)`.
11. **Return**: `c.json(response)` for single, `c.json(responses)` for batch. Always use status 200 (JSON-RPC errors are in the response body per spec).
