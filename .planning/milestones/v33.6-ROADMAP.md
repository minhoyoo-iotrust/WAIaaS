# Roadmap: WAIaaS v33.6 XRP 메인넷 지원

## Overview

XRP Ledger를 WAIaaS의 3번째 ChainType으로 통합한다. SSoT 열거형 확장과 DB 마이그레이션으로 시작하여, `@waiaas/adapter-ripple` 패키지로 네이티브 XRP 전송을 구현하고, Trust Line 토큰과 XLS-20 NFT를 추가한 후, REST/MCP/SDK/Admin UI 전 인터페이스에서 ripple 체인이 동작하도록 완성한다.

## Phases

**Phase Numbering:**
- Integer phases (470, 471, 472, 473): Planned milestone work
- Decimal phases (e.g., 470.1): Urgent insertions (marked with INSERTED)

- [ ] **Phase 470: SSoT Extension + DB Migration** - ripple ChainType/NetworkType/CAIP 등록과 DB v62 마이그레이션
- [ ] **Phase 471: Adapter Package + Native XRP Transfer** - @waiaas/adapter-ripple 패키지와 XRP 네이티브 전송 전체 구현
- [ ] **Phase 472: Trust Line Token Support** - Trust Line 설정/IOU 전송/자산 조회 구현
- [ ] **Phase 473: XLS-20 NFT + Integration Completeness** - NFT 전송과 REST/MCP/SDK/Admin UI 전 인터페이스 통합

## Phase Details

### Phase 470: SSoT Extension + DB Migration
**Goal**: 'ripple'이 시스템 전체에서 유효한 ChainType으로 인식되고 DB에 ripple 지갑 행을 저장할 수 있다
**Depends on**: Nothing (first phase of milestone)
**Requirements**: INFRA-01, INFRA-02, INFRA-03, INFRA-04, INFRA-05, INFRA-06, INFRA-07, INFRA-08
**Plans:** 3 plans
Plans:
- [ ] 470-01-PLAN.md — Shared SSoT + Core chain/network 타입 등록 (INFRA-01~04, INFRA-08)
- [ ] 470-02-PLAN.md — CAIP-2/19 레지스트리 XRPL 확장 (INFRA-05, INFRA-06)
- [ ] 470-03-PLAN.md — DB v62 마이그레이션 + AdapterPool stub (INFRA-07)
**Success Criteria** (what must be TRUE):
  1. chain='ripple'인 지갑 생성 요청이 Zod 유효성 검사를 통과한다
  2. xrpl-mainnet/xrpl-testnet/xrpl-devnet 네트워크가 ENVIRONMENT_NETWORK_MAP에서 올바른 환경에 매핑된다
  3. CAIP-2 식별자 xrpl:0/1/2가 네트워크로 변환되고 CAIP-19 slip44:144/token:{}.{} 자산 ID가 파싱된다
  4. DB v62 마이그레이션이 기존 데이터 손실 없이 chain_type CHECK 제약조건에 'ripple'을 추가한다

### Phase 471: Adapter Package + Native XRP Transfer
**Goal**: 사용자가 AI 에이전트를 통해 XRP를 r-address로 안전하게 전송하고 잔액을 정확히 조회할 수 있다
**Depends on**: Phase 470
**Requirements**: ADAPT-01, ADAPT-02, ADAPT-03, ADAPT-04, ADAPT-05, ADAPT-06, XRP-01, XRP-02, XRP-03, XRP-04, XRP-05, XRP-06, XRP-07, XRP-08, XRP-09, XRP-10
**Success Criteria** (what must be TRUE):
  1. XRP를 r-address로 전송하면 drops 단위 변환이 자동 적용되어 validated ledger에서 확인된다
  2. 잔액 조회 시 총 잔액과 가용 잔액(reserve 차감)이 구분되어 표시된다
  3. Destination Tag가 있는 전송과 X-address 입력이 올바르게 처리된다
  4. WebSocket 연결이 자동 재연결되고 동시 요청 10개 제한이 적용된다
  5. simulate(dry-run) 요청이 autofill 기반으로 트랜잭션 유효성을 검증한다
**Plans**: TBD

### Phase 472: Trust Line Token Support
**Goal**: 사용자가 XRPL Trust Line을 설정하고 IOU 토큰을 전송하며 보유 자산 목록을 조회할 수 있다
**Depends on**: Phase 471
**Requirements**: TRUST-01, TRUST-02, TRUST-03, TRUST-04, TRUST-05, TRUST-06
**Success Criteria** (what must be TRUE):
  1. buildApprove로 Trust Line을 설정하면 tfSetNoRipple 플래그가 자동 포함된다
  2. IOU 토큰을 buildTokenTransfer로 전송할 수 있고 3자리/40자리 hex 통화 코드가 모두 동작한다
  3. getAssets가 XRP 네이티브와 활성 Trust Line 토큰을 통합 목록으로 반환한다
**Plans**: TBD

### Phase 473: XLS-20 NFT + Integration Completeness
**Goal**: XRPL NFT 전송이 동작하고 ripple 체인이 REST/MCP/SDK/Admin UI 전 인터페이스에서 사용 가능하다
**Depends on**: Phase 472
**Requirements**: NFT-01, NFT-02, NFT-03, INTG-01, INTG-02, INTG-03, INTG-04, INTG-05, INTG-06, INTG-07
**Success Criteria** (what must be TRUE):
  1. XLS-20 NFT 전송이 CreateOffer + AcceptOffer 2단계로 동작하고 pending_accept 상태가 반환된다
  2. REST API에서 chain=ripple 지갑 생성/조회/전송이 기존 엔드포인트로 동작한다
  3. Admin UI에서 Ripple 체인 지갑 생성과 Trust Line 설정/조회 관리가 가능하다
  4. MCP와 SDK에서 ripple 체인 선택 및 지갑 조작이 가능하다
  5. Skill 파일이 ripple 체인 사용 가이드를 포함한다
**Plans**: TBD
