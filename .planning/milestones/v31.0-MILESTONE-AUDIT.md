## Executive Summary

Milestone v31.0 implements NFT support across 5 phases (333-337) with 58 requirements. The core NFT type system, indexer infrastructure, chain adapter methods, pipeline integration, and interface layers (MCP/SDK/Admin/Skills) were implemented with 78 new tests passing. However, **two critical integration gaps** prevent the REST API NFT routes from being reachable in production, and **one type error** blocks CI typecheck.

#### CRITICAL-01: NFT REST Routes Not Mounted in server.ts

**Severity:** CRITICAL
**Affected Requirements:** NFTQ-01, NFTQ-02, NFTQ-03, NFTQ-04, NFTQ-05, NFTQ-06, NFTQ-07, NFTQ-08, NFTQ-09, NFTQ-10, NFTQ-11, NFTA-05, ADUI-01, ADUI-02, ADUI-03, MCPSK-01 (list_nfts), MCPSK-02 (listNfts)

**Location:** `/Users/minho.yoo/dev/wallet/WAIaaS/packages/daemon/src/api/server.ts`

**Issue:** `nftRoutes()` and `nftApprovalRoutes()` are exported from `packages/daemon/src/api/routes/index.ts` (lines 21-22) but are never imported or mounted via `app.route()` in `server.ts`. The route factory functions exist, tests pass in isolation (routes mounted directly in test harness), but in production the HTTP server never registers these routes.

**Impact:**
- `GET /v1/wallet/nfts` -- 404 in production
- `GET /v1/wallets/{id}/nfts` -- 404 in production
- `GET /v1/wallet/nfts/{tokenIdentifier}` -- 404 in production
- `GET /v1/wallet/nfts/{tokenIdentifier}/approvals` -- 404 in production
- MCP `list_nfts` and `get_nft_metadata` tools will fail (they call these routes)
- SDK `listNfts()` and `getNftMetadata()` will fail (they call these routes)
- Admin UI NFT tab will show empty/error state (it calls `GET /v1/wallets/{id}/nfts`)

**Fix Required:**
1. Import `nftRoutes`, `NftRouteDeps`, `nftApprovalRoutes`, `NftApprovalRouteDeps` in server.ts
2. Create `NftIndexerClient` (already done for connect-info)
3. Create `NftMetadataCacheService` (optional, for metadata caching)
4. Mount: `app.route('/v1', nftRoutes({ db, nftIndexerClient, nftMetadataCacheService }))`
5. Mount: `app.route('/v1', nftApprovalRoutes({ db, adapterPool }))`

#### CRITICAL-02: TypeScript Typecheck Failure in @waiaas/core

**Severity:** CRITICAL (blocks CI)
**Affected Requirements:** CHADP-01 (IChainAdapter NFT extension)

**Location:** `/Users/minho.yoo/dev/wallet/WAIaaS/packages/core/src/__tests__/chain-adapter-interface.test.ts` (line 130), `/Users/minho.yoo/dev/wallet/WAIaaS/packages/core/src/__tests__/contracts/chain-adapter-mock.contract.test.ts` (line 46)

**Issue:** When IChainAdapter was extended from 22 to 25 methods (adding `buildNftTransferTx`, `transferNft`, `approveNft`), existing test mocks were not updated to include the new methods. The `chain-adapter-interface.test.ts` has a Record type checking all method keys but is missing the 3 NFT methods. The `chain-adapter-mock.contract.test.ts` has a ContractTestMockChainAdapter class that doesn't implement the 3 new interface methods.

**Impact:** `pnpm turbo run typecheck` fails for `@waiaas/core`, which blocks CI and prevents milestone PR merge per CLAUDE.md rules.

**Fix Required:**
1. Add `buildNftTransferTx: true`, `transferNft: true`, `approveNft: true` to the interface test Record
2. Add stub implementations for all 3 NFT methods to `ContractTestMockChainAdapter`
3. Fix minor unused import warnings in `nft-chain-adapter-types.test.ts` and `nft-indexer-types.test.ts`

#### MINOR-02: policy.nft_transfer_default_tier Setting Not Registered

**Severity:** MINOR (workaround exists)
**Affected Requirements:** PLCY-03

**Issue:** The `policy.nft_transfer_default_tier` setting key used in `database-policy-engine.ts` (line 352) is not registered in `setting-keys.ts`. The code uses try/catch to fallback to 'APPROVAL' when the key is not found. This means the NFT transfer default tier cannot be configured via Admin Settings UI -- it is hardcoded to APPROVAL.

**Impact:** Low -- APPROVAL is the correct default for NFT transfers (security-first). But the setting is not runtime-adjustable as the codebase conventions (CLAUDE.md) prefer.

### Conclusion

The NFT milestone has strong individual phase implementations -- schemas, indexers, adapters, pipeline, and interfaces are all well-structured with comprehensive tests. However, the critical "last mile" wiring step of mounting NFT REST routes in the HTTP server was missed. This means NFT query operations (list, metadata, approvals) are unreachable despite all the underlying infrastructure being complete. The NFT transfer flow works correctly because it uses the existing `/v1/transactions/send` endpoint.

**Recommendation:** Fix CRITICAL-01 and CRITICAL-02 before creating the milestone PR. These are mechanical fixes (adding ~15 lines to server.ts and ~20 lines to test mocks) that complete the integration.
