Pendle API client 테스트 패턴을 따른다 (setupServer + http.get + HttpResponse.json).

테스트 구조:
1. Canned mock responses: suggestedFees, limits, availableRoutes, depositStatus, swapApproval 각각 작성
2. MSW server setup: `https://app.across.to/api` base URL 기준으로 5개 엔드포인트 핸들러 등록
   - GET suggested-fees -> mock suggestedFees response
   - GET limits -> mock limits response
   - GET available-routes -> mock routes array
   - GET deposit/status -> mock status response
   - GET swap/approval -> mock swapApproval response
3. Helper: makeConfig() 함수로 ACROSS_DEFAULTS 기반 테스트용 config 생성

테스트 케이스:
- describe('getSuggestedFees')
  - returns parsed fees with totalRelayFee, timestamp, limits
  - passes query params (inputToken, outputToken, originChainId, destinationChainId, amount)
  - passes optional recipient param
- describe('getLimits')
  - returns parsed limits (minDeposit, maxDeposit, maxDepositInstant, maxDepositShortDelay)
- describe('getAvailableRoutes')
  - returns parsed route array
  - works without filter params (empty query)
  - passes filter params when provided
- describe('getDepositStatus')
  - returns parsed status response
  - handles all 4 status values: filled, pending, expired, refunded
- describe('getSwapApproval')
  - returns parsed swap approval with swapTx
- describe('error handling')
  - throws on HTTP 400 (invalid params)
  - throws on HTTP 500 (server error)
  - (msw server.use로 에러 핸들러 오버라이드)
- describe('integratorId')
  - appends integratorId as query param when config.integratorId is set
  - omits integratorId when empty string
