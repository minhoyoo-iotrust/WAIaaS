<interfaces>
Key API endpoints for x402:
- PUT /v1/admin/settings { "x402.enabled": "true" } with X-Master-Password -> 200
- POST /v1/policies { walletId, type: 'X402_ALLOWED_DOMAINS', rules: { domains: ['example.com'] } } with masterAuth -> 201
- GET /v1/policies?walletId=xxx with masterAuth -> 200 (list policies including X402_ALLOWED_DOMAINS)
- DELETE /v1/policies/:id with masterAuth -> 200

Key API endpoints for ERC-8004 (sessionAuth required):
- GET /v1/erc8004/registration-file/:walletId -> 200 { name, baseUrl, ... }
  Note: 이 엔드포인트는 온체인 호출 없이 로컬 DB에서 registration file을 빌드한다.
  다른 3개 GET 엔드포인트(/agent/:id, /agent/:id/reputation, /validation/:hash)는 온체인 RPC 호출이 필요하므로 오프체인 테스트에서는 registration-file만 테스트.

Key API endpoints for ERC-8128 (sessionAuth required):
- POST /v1/erc8128/sign { url, method, walletId? } -> 200 { signatureInput, signature, ... }
  Feature gate: erc8128.enabled must be 'true'
  Domain policy: ERC8128_ALLOWED_DOMAINS policy must allow the target domain
- POST /v1/erc8128/verify { url, method, headers } -> 200 { valid, recoveredAddress, keyid }
  Feature gate: erc8128.enabled must be 'true'

Notes:
- x402 dry-run: POST /v1/transactions/simulate with type: 'X402_PAYMENT' -- 하지만 x402는 자체 fetch flow이므로 simulate 대신 x402.enabled 설정 + X402_ALLOWED_DOMAINS 정책 CRUD로 검증
- ERC-8128 sign은 EVM wallet + domain policy 필요. erc8128.enabled + ERC8128_ALLOWED_DOMAINS 정책 설정 후 sign 호출
- ERC-8128 disabled 상태에서 sign 호출 시 ERC8128_DISABLED 에러 확인

   describe('x402-settings-crud') -- ADV-04:
   - it('enables x402 via admin settings'):
     PUT /v1/admin/settings body: key-value pairs 형태. 정확한 Settings API 형태 확인 필요.
     Admin Settings API: PUT /v1/admin/settings with body { settings: { "x402.enabled": "true" } } or flat body { "x402.enabled": "true" }
     with X-Master-Password header -> expect 200
   - it('creates X402_ALLOWED_DOMAINS policy'):
     POST /v1/policies { walletId: session.walletId, type: 'X402_ALLOWED_DOMAINS', rules: { domains: ['example.com', 'api.test.com'], max_per_request: '1000000000000000000' } } with masterAuth -> 201
     expect body.id, body.type === 'X402_ALLOWED_DOMAINS'
   - it('lists policies and finds X402_ALLOWED_DOMAINS'):
     GET /v1/policies?walletId=xxx with masterAuth -> 200
     find policy with type === 'X402_ALLOWED_DOMAINS'
   - it('deletes X402_ALLOWED_DOMAINS policy'):
     DELETE /v1/policies/:id with masterAuth -> 200
     GET /v1/policies?walletId=xxx -> confirm deleted

   describe('erc8004-registration') -- ADV-05:
   - it('retrieves registration file for a wallet'):
     GET /v1/erc8004/registration-file/:walletId with session auth (Bearer token) -> 200
     expect body to contain name field matching wallet name
     Note: sessionAuth가 필요하므로 session.http.get() 사용
   - it('returns 404 for non-existent wallet'):
     GET /v1/erc8004/registration-file/00000000-0000-0000-0000-000000000000 -> expect 404 or WALLET_NOT_FOUND error

   describe('erc8128-sign-verify') -- ADV-06:
   - it('rejects sign when erc8128 is disabled'):
     (erc8128.enabled 설정 안 한 상태에서)
     POST /v1/erc8128/sign { url: 'https://example.com/api', method: 'GET' } with session auth -> expect 403 or error with 'ERC8128_DISABLED'
   - it('enables erc8128 and creates domain policy'):
     PUT /v1/admin/settings { "erc8128.enabled": "true" } with masterAuth -> 200
     POST /v1/policies { walletId: session.walletId, type: 'ERC8128_ALLOWED_DOMAINS', rules: { domains: ['example.com'] } } with masterAuth -> 201
   - it('signs an HTTP request'):
     POST /v1/erc8128/sign { url: 'https://example.com/api', method: 'GET', walletId: session.walletId } with session auth -> 200
     expect body.signatureInput, body.signature, body.keyid to be truthy
     Note: 이 호출은 지갑의 private key를 decrypt하여 서명하므로 offchain에서 완전히 동작
   - it('verifies the signed HTTP request'):
     POST /v1/erc8128/sign으로 받은 signatureInput, signature 헤더를 사용하여
     POST /v1/erc8128/verify { url: 'https://example.com/api', method: 'GET', headers: { 'Signature-Input': signResult.signatureInput, 'Signature': signResult.signature, ...(signResult.contentDigest ? { 'Content-Digest': signResult.contentDigest } : {}) } } with session auth -> 200
