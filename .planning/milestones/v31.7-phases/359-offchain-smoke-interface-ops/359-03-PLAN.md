<!-- Relevant API endpoints -->

Audit Logs API:
- GET /v1/audit-logs (masterAuth, query: { wallet_id?, event_type?, severity?, from?, to?, tx_id?, cursor?, limit? })
  -> { items: [...], cursor?: string, total?: number }
  - Audit logs are auto-created by daemon on wallet/session/policy operations

Backup API:
- POST /v1/admin/backup (masterAuth) -> { success: true, filename: string, size: number }
- GET /v1/admin/backups (masterAuth) -> { backups: [...], total: number, retention_count: number }

Restore: The daemon CLI has a restore command, but for E2E purposes, we verify:
1. A backup can be created (POST /admin/backup)
2. The backup appears in the list (GET /admin/backups)
3. A second daemon instance can start with the backup data (or we verify the backup file exists)

Since full restore requires daemon restart with backup data, the E2E scope is:
- Create operations -> create backup -> verify backup exists -> create a second daemon from backup dir -> verify data persists
- If full restore is too complex, verify backup creation + listing + verify backup file is non-empty
</interfaces>
