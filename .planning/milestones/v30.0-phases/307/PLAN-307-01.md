## 1. 개요

Webhook Outbound 기능의 데이터 레이어와 전송 프로토콜을 설계한다. webhooks/webhook_logs 2개 테이블 스키마(Drizzle + Zod SSoT), HMAC-SHA256 서명 생성/검증 프로토콜, 비동기 재시도 큐를 정의한다.

### 1.1 핵심 설계 결정 요약

| 항목 | 결정 | 근거 |
|------|------|------|
| 테이블 수 | 2개 (webhooks, webhook_logs) | 등록 + 전송 기록 분리, 기존 notification_logs 패턴 일관 |
| Secret 저장 | SHA-256 해시만 저장 (평문 불가) | S-04 보안 요건: 비밀 평문 저장 금지 |
| 서명 알고리즘 | HMAC-SHA256 | GitHub/Stripe/Slack 업계 표준, Node.js crypto 내장 |
| 헤더 | X-WAIaaS-Signature | 프로젝트 네임스페이스 접두사 |
| 재시도 | 최대 3회, 지수 백오프 1s->2s->4s | 간단하고 예측 가능한 전략 |
| 큐 구현 | 인메모리 (setTimeout 기반) | Self-Hosted 단일 프로세스, 외부 MQ 불필요 |
| 타임아웃 | 10초 | webhook 수신자 응답 대기 상한 |

```typescript
    index('idx_webhooks_enabled').on(table.enabled),
    check('check_webhook_enabled', sql`enabled IN (0, 1)`),
  ],

```

### 2.2 컬럼 상세

| 컬럼 | 타입 | 필수 | 설명 |
|------|------|------|------|
| `id` | TEXT PK | O | UUID v7 (시간 정렬) |

| `secret_hash` | TEXT | O | SHA-256(secret) 해시값. 등록 시 한 번만 평문 반환, 이후 조회 불가 |
| `events` | TEXT | O | JSON 배열. 구독할 이벤트 타입 목록. 빈 배열 `[]` = 전체 이벤트 구독 |
| `description` | TEXT | - | 관리자용 설명 ("Slack #alerts 연동" 등) |
| `enabled` | INTEGER | O | 1=활성, 0=비활성. 비활성 webhook은 이벤트 전송 건너뜀 |
| `created_at` | INTEGER | O | Unix epoch 초 |
| `updated_at` | INTEGER | O | Unix epoch 초 |

### 2.3 Secret 보안 모델

webhook secret은 등록 시점에 서버가 `crypto.randomBytes(32).toString('hex')`로 생성한다.

```
[등록 흐름]
1. 서버: secret = randomBytes(32).toString('hex')     -- 64자 hex
2. 서버: secretHash = SHA-256(secret)                  -- DB 저장
3. 응답: { id, url, secret, ... }                      -- 평문 한 번만 반환
4. 이후 GET /v1/webhooks 조회 시 secret 필드 없음      -- 복구 불가, 재생성만 가능
```

**평문 secret은 DB에 저장하지 않는다.** SHA-256 해시만 저장하여 S-04 보안 요건을 준수한다. 클라이언트가 secret을 분실하면 webhook을 삭제하고 재등록해야 한다.

**HMAC 서명 생성 시 secret 필요 문제 해결:**
- HMAC 서명에는 원본 secret이 필요하지만, DB에는 해시만 저장한다
- 해결: secret을 AES-256-GCM으로 암호화하여 저장한다 (기존 settings-crypto 패턴 재사용)
- `secretHash` 컬럼은 **API 조회 시 secret 노출 방지** 용도로 유지
- 별도 `secretEncrypted` 컬럼을 추가하여 HMAC 서명 생성에 사용

### 2.4 수정된 Drizzle 정의 (Secret 이중 저장)

```typescript
export const webhooks = sqliteTable(
  'webhooks',
  {
    id: text('id').primaryKey(),
    url: text('url').notNull(),
    secretHash: text('secret_hash').notNull(),                // SHA-256(secret) -- 조회 API 노출 방지용
    secretEncrypted: text('secret_encrypted').notNull(),      // AES-256-GCM(secret, masterPassword) -- HMAC 서명 생성용
    events: text('events').notNull(),                         // JSON array
    description: text('description'),
    enabled: integer('enabled', { mode: 'boolean' }).notNull().default(true),
    createdAt: integer('created_at', { mode: 'timestamp' }).notNull(),
    updatedAt: integer('updated_at', { mode: 'timestamp' }).notNull(),
  },
  (table) => [
    index('idx_webhooks_enabled').on(table.enabled),
    check('check_webhook_enabled', sql`enabled IN (0, 1)`),
  ],
);
```

### 2.5 Secret 저장 흐름 (최종)

```
[등록]
1. secret = randomBytes(32).toString('hex')
2. secretHash = SHA-256(secret)                    → webhooks.secret_hash
3. secretEncrypted = AES-256-GCM(secret, mp)       → webhooks.secret_encrypted
4. 응답에 secret 평문 한 번 반환

[HMAC 서명 생성 시]
1. webhooks.secret_encrypted 읽기
2. secret = decrypt(secretEncrypted, masterPassword)
3. HMAC-SHA256(secret, payload) → X-WAIaaS-Signature

[GET 조회 시]
- secret, secretHash, secretEncrypted 모두 응답에 포함하지 않음
```

## 5. HMAC-SHA256 서명 프로토콜 (HOOK-02)

### 5.1 서명 생성

webhook 전송 시 요청 본문(payload)을 HMAC-SHA256으로 서명하여 `X-WAIaaS-Signature` 헤더에 포함한다.

```typescript
// WebhookService.sign()
import { createHmac } from 'node:crypto';

function signPayload(secret: string, payload: string): string {
  const hmac = createHmac('sha256', secret);
  hmac.update(payload, 'utf8');
  return `sha256=${hmac.digest('hex')}`;
}

// 사용 예
const body = JSON.stringify(webhookPayload);
const signature = signPayload(secret, body);
// X-WAIaaS-Signature: sha256=5d5b09f6dcb2d53a5fffc60c4ac0d55fabdf556069d6631545f42aa6e3500f2e
```

### 5.2 서명 포맷

```
X-WAIaaS-Signature: sha256={hex_digest}
```

- 접두사 `sha256=`로 알고리즘을 명시한다 (GitHub Webhooks 관례)
- hex digest는 64자 소문자 hex 문자열
- 서명 대상: HTTP POST body 원문 (JSON 직렬화된 문자열)

### 5.3 Webhook 요청 헤더

| 헤더 | 값 | 설명 |
|------|-----|------|
| `Content-Type` | `application/json` | 고정 |
| `X-WAIaaS-Signature` | `sha256={hex}` | HMAC-SHA256 서명 |
| `X-WAIaaS-Event` | `TX_SUBMITTED` | 이벤트 타입 |
| `X-WAIaaS-Delivery` | UUID v7 | 전송 고유 ID (webhook_logs.id) |
| `X-WAIaaS-Timestamp` | Unix epoch 초 | 서명 생성 시각 (재생 공격 방지) |
| `User-Agent` | `WAIaaS-Webhook/1.0` | 발신자 식별 |

### 5.5 수신 측 검증 가이드

webhook 수신 서버는 다음 절차로 요청의 진위를 검증한다:

```typescript
// --- 수신 측 검증 코드 예시 (Node.js) ---

import { createHmac, timingSafeEqual } from 'node:crypto';

function verifyWebhookSignature(
  secret: string,       // 등록 시 받은 secret
  body: string,         // req.body (raw string, parsed JSON 아님)
  signature: string,    // req.headers['x-waiaas-signature']
  timestamp: number,    // req.headers['x-waiaas-timestamp']
): boolean {
  // 1. 타임스탬프 검증 (5분 이내)
  const now = Math.floor(Date.now() / 1000);
  if (Math.abs(now - timestamp) > 300) {
    return false; // 재생 공격 방어
  }

  // 2. 서명 검증
  const expected = `sha256=${createHmac('sha256', secret).update(body, 'utf8').digest('hex')}`;
  if (signature.length !== expected.length) return false;

  // 3. 타이밍 안전 비교
  return timingSafeEqual(
    Buffer.from(signature, 'utf8'),
    Buffer.from(expected, 'utf8'),
  );
}

// Express/Hono 미들웨어 예시
app.post('/webhook', (req, res) => {
  const signature = req.headers['x-waiaas-signature'] as string;
  const timestamp = Number(req.headers['x-waiaas-timestamp']);
  const rawBody = req.rawBody; // raw string

  if (!verifyWebhookSignature(WEBHOOK_SECRET, rawBody, signature, timestamp)) {
    return res.status(401).json({ error: 'Invalid signature' });
  }

  // 서명 검증 성공 -- 이벤트 처리
  const event = JSON.parse(rawBody);
  console.log(`Received: ${event.event}`, event.data);

  // 2xx 응답으로 수신 확인 (중요: 5초 내 응답 권장)
  res.status(200).json({ received: true });
});
```

### 5.6 보안 고려사항

| 항목 | 설계 | 근거 |
|------|------|------|
| Secret 길이 | 32 bytes (256 bits) | HMAC-SHA256 키 길이 권장 |
| Secret 저장 | AES-256-GCM 암호화 + SHA-256 해시 | 평문 저장 불가 (S-04) |
| 타이밍 안전 비교 | `timingSafeEqual` 필수 | 타이밍 사이드채널 방어 |
| 재생 방지 | X-WAIaaS-Timestamp + 5분 윈도우 | 캡처된 요청 재전송 방어 |
| TLS | HTTPS 권장 (HTTP 허용) | 로컬 개발 지원, prod에서는 HTTPS 필수 |

## 6. 비동기 재시도 큐 설계 (HOOK-03)

### 6.1 설계 원칙

1. **메인 파이프라인 비블로킹**: EventBus listener에서 webhook 전송을 fire-and-forget으로 큐에 넣는다. 파이프라인 성능에 영향 없음
2. **인메모리 큐**: Self-Hosted 단일 프로세스 아키텍처이므로 외부 MQ(Redis, RabbitMQ 등) 불필요. setTimeout 기반
3. **최대 3회 재시도**: 초기 전송 1회 + 재시도 3회 = 최대 4회 시도
4. **지수 백오프**: 1초 -> 2초 -> 4초
5. **재시도별 로그**: 각 시도를 webhook_logs에 개별 기록

### 6.2 재시도 전략

| 시도 | 대기 시간 | 누적 시간 | 설명 |
|------|----------|----------|------|
| 1차 (초기) | 즉시 | 0s | 이벤트 발생 시 즉시 전송 |
| 2차 (재시도 1) | 1초 | 1s | 첫 번째 재시도 |
| 3차 (재시도 2) | 2초 | 3s | 두 번째 재시도 |
| 4차 (재시도 3) | 4초 | 7s | 세 번째(마지막) 재시도 |

최대 총 소요 시간: ~17초 (4회 시도 각 10초 타임아웃 + 백오프 7초). 실제로는 대부분 1-2초 이내 완료.

### 6.3 WebhookDeliveryQueue 인터페이스

```typescript
// packages/daemon/src/services/webhook/webhook-delivery-queue.ts

interface WebhookDeliveryJob {
  webhookId: string;
  deliveryId: string;        // UUID v7 (= webhook_logs.id 후보)
  url: string;
  secret: string;            // 복호화된 평문 (메모리에만 존재)
  payload: WebhookPayload;
  attempt: number;           // 현재 시도 횟수
  maxAttempts: number;       // 최대 시도 횟수 (4)
}

class WebhookDeliveryQueue {
  private readonly maxAttempts = 4;           // 1 + 3 retries
  private readonly timeoutMs = 10_000;        // 10초 타임아웃
  private readonly backoffMs = [0, 1000, 2000, 4000]; // 시도별 대기 시간

  constructor(
    private db: BetterSQLite3Database<typeof schema>,
  ) {}

  /**
   * 큐에 전송 작업 추가 (fire-and-forget).
   * EventBus listener에서 호출.
   */
  enqueue(job: WebhookDeliveryJob): void {
    // 즉시 전송 시작 (비동기, 호출자 블로킹 없음)
    this.deliver(job).catch(() => {
      // 최종 실패 -- webhook_logs에 이미 기록됨
    });
  }

  /**
   * 전송 + 재시도 루프.
   */
  private async deliver(job: WebhookDeliveryJob): Promise<void> {
    for (let attempt = 1; attempt <= this.maxAttempts; attempt++) {
      // 백오프 대기
      if (attempt > 1) {
        await this.sleep(this.backoffMs[attempt - 1]!);
      }

      const startTime = Date.now();
      try {
        const response = await fetch(job.url, {
          method: 'POST',
          headers: {
            'Content-Type': 'application/json',
            'X-WAIaaS-Signature': this.sign(job.secret, JSON.stringify(job.payload)),
            'X-WAIaaS-Event': job.payload.event,
            'X-WAIaaS-Delivery': job.deliveryId,
            'X-WAIaaS-Timestamp': String(job.payload.timestamp),
            'User-Agent': 'WAIaaS-Webhook/1.0',
          },
          body: JSON.stringify(job.payload),
          signal: AbortSignal.timeout(this.timeoutMs),
        });

        const duration = Date.now() - startTime;

        if (response.ok) {
          // 성공 -- 로그 기록 후 종료
          this.logAttempt(job, attempt, 'success', response.status, duration);
          return;
        }

        // HTTP 에러 (4xx, 5xx)
        this.logAttempt(job, attempt, 'failed', response.status, duration,
          `HTTP ${response.status} ${response.statusText}`);

        // 4xx는 재시도 무의미 (클라이언트 에러)
        if (response.status >= 400 && response.status < 500) {
          return;
        }
      } catch (err) {
        const duration = Date.now() - startTime;
        const error = err instanceof Error ? err.message : String(err);
        this.logAttempt(job, attempt, 'failed', null, duration, error);
      }
    }
    // 모든 재시도 소진
  }

  private sign(secret: string, body: string): string {
    return `sha256=${createHmac('sha256', secret).update(body, 'utf8').digest('hex')}`;
  }

  private logAttempt(
    job: WebhookDeliveryJob,
    attempt: number,
    status: 'success' | 'failed',
    httpStatus: number | null,
    durationMs: number,
    error?: string,
  ): void {
    try {
      // webhook_logs INSERT (fire-and-forget)
      this.db.insert(schema.webhookLogs).values({
        id: generateId(),
        webhookId: job.webhookId,
        eventType: job.payload.event,
        status,
        httpStatus,
        attempt,
        error: error ?? null,
        requestDuration: durationMs,
        createdAt: new Date(),
      }).run();
    } catch {
      // DB 에러가 전송 로직을 중단하면 안 됨
    }
  }

  private sleep(ms: number): Promise<void> {
    return new Promise((resolve) => setTimeout(resolve, ms));
  }
}
```

### 6.4 비블로킹 보장

```
[트랜잭션 파이프라인]
  Stage 5 → emit('transaction:completed') → [리턴, 파이프라인 계속]
                    │
                    ▼ (EventBus listener, 동기 호출)
  [WebhookService.onEvent()]
    │
    ├── 활성 webhook 조회 (DB SELECT, ~1ms)
    ├── 이벤트 필터 매칭
    └── queue.enqueue(job)  ← fire-and-forget (Promise 미대기)
            │
            ▼ (비동기, 별도 마이크로태스크)
      [WebhookDeliveryQueue.deliver()]
        ├── fetch(url, payload) → 성공: 로그
        └── 실패 → setTimeout(backoff) → 재시도
```

EventBus.emit()은 동기 루프이므로 listener 내에서 `await`하지 않는다. `queue.enqueue()`는 Promise를 반환하되 호출자가 대기하지 않으므로 파이프라인은 즉시 계속된다.

### 6.5 데몬 셧다운 시 동작

진행 중인 webhook 전송 작업은 graceful shutdown에서 완료를 기다리지 않는다. 인메모리 큐이므로 데몬 재시작 시 미완료 전송은 유실된다.

- **수용 가능한 트레이드오프**: webhook은 best-effort 전달이며, 수신 측은 누락 가능성을 감안해야 한다
- **대안 (v2 후보)**: webhook_delivery_queue 테이블로 영속화하여 재시작 후 재시도 (현재 범위 외)
