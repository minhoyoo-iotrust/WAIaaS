**Phase:** 307

## 1. 개요

Webhook 관리 REST API 4개 엔드포인트를 설계하고, EventBus를 통한 이벤트 필터링/전달 메커니즘을 정의한다. 기존 NotificationChannel과의 관계를 명확히 하고, 영향받는 설계 문서(25, 35, 37)의 갱신 내용을 정리한다.

### 1.1 핵심 설계 결정 요약

| 항목 | 결정 | 근거 |
|------|------|------|
| 경로 접두사 | `/v1/webhooks` | 독립 리소스, admin 서브경로 불필요 |
| 인증 | masterAuth (X-Master-Password) | webhook 관리는 관리자 전용 |
| NotificationChannel 관계 | **독립 구조** (INotificationChannel 미구현) | webhook은 HMAC+재시도 등 채널과 다른 관심사 |
| EventBus 연동 | 기존 이벤트 + 전용 webhook 이벤트 | 감사 이벤트 20개 재사용 |
| 이벤트 필터링 | webhooks.events JSON 배열 매칭 | 빈 배열 = 전체 구독 |
| 로그 조회 | createdAt 기반 cursor | webhook_logs PK는 UUID (정수 cursor 불가) |

### 2.2 POST /v1/webhooks (등록)

**요청:**

```typescript
// CreateWebhookRequestSchema (Plan 307-01에서 정의)
{
  url: string;          // 필수, URL 형식, max 2048
  events: string[];     // 선택, 빈 배열 = 전체 구독
  description?: string; // 선택, max 256
}
```

**응답 (201 Created):**

```typescript
// CreateWebhookResponseSchema
{
  id: string;           // UUID v7
  url: string;
  events: string[];
  description: string | null;
  enabled: boolean;     // true
  secret: string;       // ⚠️ 한 번만 반환, 이후 조회 불가
  createdAt: number;    // epoch 초
  updatedAt: number;    // epoch 초
}
```

**처리 흐름:**

```
1. masterAuth 검증
2. Zod 스키마 검증 (url, events, description)
3. events 배열의 각 값이 WEBHOOK_EVENT_TYPES에 포함되는지 검증
4. secret = randomBytes(32).toString('hex')
5. secretHash = SHA-256(secret)
6. secretEncrypted = encryptSettingValue(secret, masterPassword)
7. DB INSERT (webhooks)
8. 응답에 secret 포함하여 반환
```

**에러:**

| 코드 | HTTP | 조건 |
|------|------|------|
| `INVALID_MASTER_PASSWORD` | 401 | 인증 실패 |
| `VALIDATION_ERROR` | 400 | 입력 검증 실패 (잘못된 URL, 잘못된 이벤트 타입 등) |

**예시:**

```
POST /v1/webhooks
X-Master-Password: my-secret-password
Content-Type: application/json

{
  "url": "https://example.com/webhook",
  "events": ["TX_SUBMITTED", "TX_CONFIRMED", "KILL_SWITCH_ACTIVATED"],
  "description": "Production alert webhook"
}

→ 201
{
  "id": "01936d3c-7f8a-7b00-9e4d-aaaaaa000001",
  "url": "https://example.com/webhook",
  "events": ["TX_SUBMITTED", "TX_CONFIRMED", "KILL_SWITCH_ACTIVATED"],
  "description": "Production alert webhook",
  "enabled": true,
  "secret": "a1b2c3d4e5f6...64자hex",
  "createdAt": 1772525000,
  "updatedAt": 1772525000
}
```

## 3. EventBus 연동 설계 (HOOK-05)

### 3.1 WebhookService 아키텍처

```
┌─────────────────────────────────────────────────────────┐
│                    EventBus                              │
│  transaction:completed, transaction:failed,              │
│  wallet:activity, kill-switch:state-changed, ...         │
└────────────────────┬────────────────────────────────────┘
                     │ on(event, data)
                     ▼
┌─────────────────────────────────────────────────────────┐
│               WebhookService                             │
│                                                          │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │ Event        │  │ Webhook      │  │ Delivery     │  │
│  │ Mapper       │→ │ Filter       │→ │ Queue        │  │
│  │              │  │              │  │              │  │
│  │ EventBus →   │  │ webhook.     │  │ enqueue()    │  │
│  │ AuditEvent   │  │ events[]     │  │ fire-forget  │  │
│  │ 매핑         │  │ 매칭         │  │              │  │
│  └──────────────┘  └──────────────┘  └──────────────┘  │
│                                              │           │
│                                              ▼           │
│                                    ┌──────────────┐     │
│                                    │ webhook_logs │     │
│                                    │ INSERT       │     │
│                                    └──────────────┘     │
└─────────────────────────────────────────────────────────┘
```

### 3.2 EventBus 이벤트 → Webhook 이벤트 매핑

기존 EventBus의 WaiaasEventMap 이벤트를 Phase 305에서 정의한 AuditEventType으로 매핑한다.

| EventBus 이벤트 | 매핑되는 AuditEventType | payload 매핑 |
|----------------|----------------------|-------------|
| `transaction:completed` | `TX_CONFIRMED` | `{ txId, txHash, walletId, chain, network, type, amount }` |
| `transaction:failed` | `TX_FAILED` | `{ txId, walletId, error, network, type }` |
| `transaction:incoming` | (별도 이벤트) | `{ walletId, txHash, fromAddress, amount, chain, network }` |
| `transaction:incoming:suspicious` | (별도 이벤트) | `{ walletId, txHash, fromAddress, amount, suspiciousReasons }` |
| `wallet:activity` (TX_SUBMITTED) | `TX_SUBMITTED` | `{ walletId, details }` |
| `wallet:activity` (SESSION_CREATED) | `SESSION_CREATED` | `{ walletId, details }` |
| `wallet:activity` (OWNER_SET) | `OWNER_REGISTERED` | `{ walletId, details }` |
| `kill-switch:state-changed` | `KILL_SWITCH_ACTIVATED` / `KILL_SWITCH_RECOVERED` | `{ state, previousState, activatedBy }` |

**매핑 불가한 감사 이벤트 (별도 삽입 필요):**

일부 AuditEventType은 EventBus를 통해 발행되지 않으므로 직접 호출이 필요하다:

| AuditEventType | 발생 위치 | WebhookService 호출 방법 |
|----------------|---------|------------------------|
| `WALLET_CREATED` | POST /wallets 핸들러 | 핸들러에서 `webhookService.dispatch()` 직접 호출 |
| `WALLET_SUSPENDED` | PATCH /wallets/:id 핸들러 | 동일 |
| `SESSION_REVOKED` | DELETE /sessions/:id 핸들러 | 동일 |
| `POLICY_DENIED` | Stage 3 | 동일 |
| `MASTER_AUTH_FAILED` | master-auth 미들웨어 | 동일 |
| `AUTO_STOP_TRIGGERED` | AutoStopService | 동일 |

### 3.3 WebhookService 인터페이스

```typescript
// packages/daemon/src/services/webhook/webhook-service.ts

import type { EventBus, WaiaasEventMap } from '@waiaas/core';
import type { BetterSQLite3Database } from 'drizzle-orm/better-sqlite3';
import type * as schema from '../../infrastructure/database/schema.js';

export class WebhookService {
  private queue: WebhookDeliveryQueue;

  constructor(
    private db: BetterSQLite3Database<typeof schema>,
    private eventBus: EventBus,
    private getMasterPassword: () => string,  // 런타임 masterPassword 접근
  ) {
    this.queue = new WebhookDeliveryQueue(db);
    this.registerEventListeners();
  }

  /**
   * EventBus 이벤트 리스너 등록.
   * 각 이벤트를 AuditEventType으로 매핑하여 dispatch()에 전달.
   */
  private registerEventListeners(): void {
    this.eventBus.on('transaction:completed', (data) => {
      this.dispatch('TX_CONFIRMED', {
        txId: data.txId,
        txHash: data.txHash,
        walletId: data.walletId,
        chain: undefined,   // WaiaasEventMap에 없으면 생략
        network: data.network,
        type: data.type,
        amount: data.amount,
      });
    });

    this.eventBus.on('transaction:failed', (data) => {
      this.dispatch('TX_FAILED', {
        txId: data.txId,
        walletId: data.walletId,
        error: data.error,
        network: data.network,
        type: data.type,
      });
    });

    this.eventBus.on('wallet:activity', (data) => {
      const eventMap: Record<string, string> = {
        'TX_SUBMITTED': 'TX_SUBMITTED',
        'TX_REQUESTED': 'TX_SUBMITTED',
        'SESSION_CREATED': 'SESSION_CREATED',
        'OWNER_SET': 'OWNER_REGISTERED',
      };
      const auditEvent = eventMap[data.activity];
      if (auditEvent) {
        this.dispatch(auditEvent, {
          walletId: data.walletId,
          ...data.details,
        });
      }
    });

    this.eventBus.on('kill-switch:state-changed', (data) => {
      const event = data.state === 'ACTIVE'
        ? 'KILL_SWITCH_ACTIVATED'
        : 'KILL_SWITCH_RECOVERED';
      this.dispatch(event, {
        state: data.state,
        previousState: data.previousState,
        activatedBy: data.activatedBy,
      });
    });

    this.eventBus.on('transaction:incoming', (data) => {
      this.dispatch('TX_INCOMING' as any, {
        walletId: data.walletId,
        txHash: data.txHash,
        fromAddress: data.fromAddress,
        amount: data.amount,
        chain: data.chain,
        network: data.network,
      });
    });
  }

  /**
   * 이벤트를 구독 중인 모든 활성 webhook에 전달.
   * EventBus listener 또는 직접 호출 모두 가능.
   */
  dispatch(eventType: string, data: Record<string, unknown>): void {
    try {
      // 활성 webhook 조회
      const activeWebhooks = this.db
        .select()
        .from(schema.webhooks)
        .where(eq(schema.webhooks.enabled, true))
        .all();

      if (activeWebhooks.length === 0) return;

      const now = Math.floor(Date.now() / 1000);
      const masterPassword = this.getMasterPassword();

      for (const webhook of activeWebhooks) {
        // 이벤트 필터링
        const subscribedEvents = JSON.parse(webhook.events) as string[];
        if (subscribedEvents.length > 0 && !subscribedEvents.includes(eventType)) {
          continue; // 이 webhook은 해당 이벤트를 구독하지 않음
        }

        // secret 복호화
        const secret = decryptSettingValue(webhook.secretEncrypted, masterPassword);

        const deliveryId = generateId();
        const payload: WebhookPayload = {
          id: deliveryId,
          event: eventType,
          timestamp: now,
          data,
        };

        // 큐에 전송 작업 추가 (fire-and-forget)
        this.queue.enqueue({
          webhookId: webhook.id,
          deliveryId,
          url: webhook.url,
          secret,
          payload,
          attempt: 1,
          maxAttempts: 4,
        });
      }
    } catch (err) {
      // webhook 전송 실패가 메인 로직을 중단하면 안 됨
      console.error('[WebhookService] dispatch error:', err);
    }
  }

  /**
   * 리소스 정리 (데몬 셧다운 시).
   */
  destroy(): void {
    this.eventBus.removeAllListeners();
  }
}
```

### 3.4 이벤트 필터링 메커니즘

```
[이벤트 발생]
  │
  ▼
[WebhookService.dispatch(eventType, data)]
  │
  ├── SELECT * FROM webhooks WHERE enabled = 1
  │
  ├── for each webhook:
  │     │
  │     ├── events = JSON.parse(webhook.events)
  │     │
  │     ├── events === [] ?  → 전체 구독 → 전송
  │     │
  │     └── events.includes(eventType) ?
  │           ├── YES → 전송
  │           └── NO  → skip
  │
  └── queue.enqueue(job) × N개 webhook
```

**필터링 규칙:**

| webhooks.events 값 | 동작 |
|---------------------|------|
| `[]` (빈 배열) | 모든 이벤트 수신 (와일드카드) |
| `["TX_SUBMITTED"]` | TX_SUBMITTED만 수신 |
| `["TX_SUBMITTED", "TX_CONFIRMED", "TX_FAILED"]` | 3개 이벤트만 수신 |
| `["KILL_SWITCH_ACTIVATED", "KILL_SWITCH_RECOVERED"]` | Kill Switch 이벤트만 수신 |

## 4. NotificationChannel과의 관계 (HOOK-05)

### 4.1 독립 구조 결정

webhook은 `INotificationChannel`을 구현하지 않는다. **독립적인 서비스로 운영한다.**

**근거:**

| 비교 항목 | INotificationChannel | WebhookService |
|----------|---------------------|----------------|
| 전송 대상 | 단일 채널 (Telegram/Discord/ntfy) | N개 구독자 (webhook URL) |
| 인증 | 각 채널 고유 (Bot Token, Webhook URL) | HMAC 서명 프로토콜 |
| 재시도 | NotificationService 폴백 체인 | 자체 재시도 큐 (지수 백오프) |
| 이벤트 필터 | SettingsService 카테고리 필터 | 개별 webhook 이벤트 배열 |
| 전송 기록 | notification_logs 공유 | webhook_logs 전용 |
| Secret 관리 | settings 테이블 (category별) | webhooks 테이블 (AES-GCM 암호화) |
| 구독 관리 | 설정 ON/OFF | CRUD REST API |

**INotificationChannel의 `send(payload)` 시그니처는 webhook의 "N개 URL에 HMAC 서명 포함 전송 + 개별 재시도"와 맞지 않는다.**

### 4.2 아키텍처 다이어그램

```
┌──────────────────────────────────────────────────────────────┐
│                       EventBus                                │
└───────┬────────────────────────────────────────┬──────────────┘
        │                                        │
        ▼                                        ▼
┌───────────────────┐                  ┌────────────────────┐
│ NotificationService│                  │  WebhookService     │
│                    │                  │                     │
│ ┌────────────────┐│                  │ ┌─────────────────┐│
│ │ TelegramChannel││                  │ │ Event Mapper    ││
│ ├────────────────┤│                  │ ├─────────────────┤│
│ │ DiscordChannel ││                  │ │ Webhook Filter  ││
│ ├────────────────┤│                  │ ├─────────────────┤│
│ │ NtfyChannel    ││                  │ │ DeliveryQueue   ││
│ ├────────────────┤│                  │ └─────────────────┘│
│ │ SlackChannel   ││                  │                     │
│ └────────────────┘│                  │ webhooks 테이블     │
│                    │                  │ webhook_logs 테이블 │
│ notification_logs  │                  │                     │
└───────────────────┘                  └────────────────────┘
      ↑ (사이드 채널)
┌─────────────────────┐
│ WalletNotification  │
│ Channel (지갑 앱)    │
└─────────────────────┘
```

**핵심 차이:**

- NotificationService는 `notify(eventType, walletId, vars)`로 **단일 이벤트를 설정된 채널에 전송**
- WebhookService는 `dispatch(eventType, data)`로 **단일 이벤트를 N개 구독 webhook에 전송**
- 두 서비스 모두 EventBus를 소비하지만 독립적으로 동작한다
