## 2. REST API 엔드포인트 스펙 (SIM-03)

### 2.1 POST /v1/transactions/simulate

**경로:** `POST /v1/transactions/simulate`
**인증:** sessionAuth (Authorization: Bearer wai_sess_...)
**태그:** Transactions
**설명:** 트랜잭션을 실제 실행하지 않고 정책 평가 결과, 예상 수수료, 잔액 변화를 시뮬레이션한다.

#### 2.1.1 요청 바디

기존 `TransactionRequestSchema` (discriminatedUnion 5-type)을 **그대로 재사용**한다. 별도의 시뮬레이션 전용 스키마를 만들지 않는다.

```typescript

// TRANSFER | TOKEN_TRANSFER | CONTRACT_CALL | APPROVE | BATCH

// 추가 필드 없음 -- 기존 TransactionRequest 그대로 사용
// gasCondition 필드가 있어도 무시됨 (dry-run에서 가스 조건 의미 없음)

// 예시: TRANSFER
{
  "type": "TRANSFER",
  "to": "0x1234...abcd",
  "amount": "1000000000000000000",
  "network": "ethereum-mainnet"
}

// 예시: TOKEN_TRANSFER
{
  "type": "TOKEN_TRANSFER",
  "to": "0x1234...abcd",
  "amount": "1000000",
  "token": {
    "address": "0xA0b86991c6218b36c1d19D4a2e9Eb0cE3606eB48",
    "decimals": 6,
    "symbol": "USDC"
  },
  "network": "ethereum-mainnet"

```

**재사용 근거:**
- 에이전트가 `sendToken()` 요청 바디를 그대로 `simulate()`에 전달 가능 (복사-붙여넣기 워크플로우)
- 별도 스키마 관리 부담 없음
- Zod 검증 로직 공유 (Stage 1' Validate)

**설계 결정: HTTP 상태 코드**

| 상황 | HTTP 상태 | 근거 |
|------|----------|------|
| 시뮬레이션 성공 (정책 허용) | 200 | 정상 완료 |
| 시뮬레이션 성공 (정책 거부) | 200 | 시뮬레이션 자체는 성공. `success=false`로 결과 전달. |
| 요청 검증 실패 (Zod) | 422 | 기존 파이프라인과 동일한 에러 포맷 |
| 세션 인증 실패 | 401 | 기존 sessionAuth 미들웨어 동작 |
| 지갑 미존재 | 404 | WALLET_NOT_FOUND |
| 지갑 종료 상태 | 409 | WALLET_TERMINATED |
| 내부 서버 에러 | 500 | 예상치 못한 에러 |

> **근거:** 정책 거부를 4xx로 반환하면, 에이전트가 "시뮬레이션 실패"와 "정책 거부"를 구분하기 어렵다.
> HTTP 200 + `success: false` + `policy.allowed: false`로 반환하면 에이전트가 프로그래밍 방식으로 명확히 구분 가능.

#### 2.1.4 라우트 핸들러 의사코드

```typescript
router.openapi(simulateTransactionRoute, async (c) => {
  // 1. 요청 바디 읽기 (raw JSON)
  const request = await c.req.json();

  // 2. walletId 해결 (body.walletId > query > single-wallet auto-resolve)
  const walletId = resolveWalletId(c, deps.db, request.walletId);

  // 3. 지갑 조회 + 종료 상태 확인
  const wallet = await deps.db.select().from(wallets).where(eq(wallets.id, walletId)).get();
  if (!wallet) throw new WAIaaSError('WALLET_NOT_FOUND');
  if (wallet.status === 'TERMINATED') throw new WAIaaSError('WALLET_TERMINATED');

  // 4. 네트워크 해결 + 어댑터 풀 해결
  const resolvedNetwork = resolveNetwork(...);
  const adapter = await deps.adapterPool.resolve(...);

  // 5. TransactionPipeline.executeDryRun() 호출
  const pipeline = new TransactionPipeline(deps);
  const result = await pipeline.executeDryRun(walletId, request, sessionId);

  // 6. 200 응답 (success 여부와 무관하게 200)
  return c.json(result, 200);
});
```

### 2.2 walletId 해결 패턴

기존 `POST /v1/transactions/send`와 동일한 패턴:

```
body.walletId > session에 단일 지갑이면 자동 해결 > multi-wallet이면 필수
```

요청 바디에 `walletId` 필드를 추가하되, 기존 TransactionRequestSchema에는 포함하지 않는다. 라우트 핸들러에서 `request.walletId`를 별도 추출한다. (기존 `/transactions/send` 패턴 그대로.)

### 3.2 메서드 구현 설계

```typescript
// packages/sdk/src/client.ts 추가

export class WAIaaSClient {
  // ... 기존 메서드 ...

  /**
   * Simulate a transaction without executing it.
   *
   * Returns policy evaluation, fee estimates, balance changes, and warnings.
   * No side effects: no DB writes, no signing, no on-chain submission.
   *
   * @param params - Same parameters as sendToken()
   * @returns SimulateResponse with policy tier, fees, balance changes, warnings
   *
   * @example
   * ```typescript
   * // Simulate before sending
   * const sim = await client.simulate({
   *   type: 'TRANSFER',
   *   to: '0x1234...abcd',
   *   amount: '1000000000000000000',
   *   network: 'ethereum-mainnet',
   * });
   *
   * if (sim.success && sim.policy.tier === 'INSTANT') {
   *   const tx = await client.sendToken(params); // proceed
   * } else {
   *   console.log('Warnings:', sim.warnings);
   * }
   * ```
   */
  async simulate(params: SimulateParams): Promise<SimulateResponse> {
    // Pre-validate (same as sendToken)
    validateSendToken(params);
    return withRetry(
      () => this.http.post<SimulateResponse>(
        '/v1/transactions/simulate',
        params,
        this.authHeaders(),
      ),
      this.retryOptions,
    );
  }
}
```

### 3.3 SDK 사전 검증

`simulate()`는 `sendToken()`과 동일한 `validateSendToken()` 사전 검증을 수행한다. 동일한 스키마(TransactionRequestSchema)를 사용하므로 자연스러운 재사용.

### 3.4 SDK 에러 처리

| HTTP 상태 | SDK 동작 |
|----------|---------|
| 200 | SimulateResponse 반환 (success 필드 확인은 호출자 책임) |
| 401 | WAIaaSError(UNAUTHORIZED) throw |
| 404 | WAIaaSError(WALLET_NOT_FOUND) throw |
| 422 | WAIaaSError(VALIDATION_ERROR) throw |
| 429 | 재시도 (retryOptions에 따라) |
| 5xx | 재시도 후 WAIaaSError throw |

## 4. MCP waiaas_simulate_transaction Tool (SIM-04 - MCP)

### 4.1 Tool 등록

```typescript
// packages/mcp/src/tools/simulate-transaction.ts

import { z } from 'zod';
import type { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js';
import { type ApiClient, toToolResult } from '../api-client.js';
import { type WalletContext, withWalletPrefix } from '../server.js';

export function registerSimulateTransaction(
  server: McpServer,
  apiClient: ApiClient,
  walletContext?: WalletContext,
): void {
  server.tool(
    'waiaas_simulate_transaction',
    withWalletPrefix(
      'Simulate a transaction without executing it. ' +
      'Returns policy tier, estimated fees, balance changes, and warnings. ' +
      'No side effects -- use this to check before sending.',
      walletContext?.walletName,
    ),
    {
      to: z.string().describe('Destination wallet address'),
      amount: z.string().describe('Amount in smallest unit (lamports/wei)'),
      type: z.enum(['TRANSFER', 'TOKEN_TRANSFER', 'CONTRACT_CALL', 'APPROVE', 'BATCH'])
        .optional()
        .describe('Transaction type. Default: TRANSFER'),
      token: z.object({
        address: z.string().describe('Token mint (SPL) or contract address (ERC-20)'),
        decimals: z.number().describe('Token decimals (e.g., 6 for USDC)'),
        symbol: z.string().describe('Token symbol (e.g., USDC)'),
        assetId: z.string().optional().describe('CAIP-19 asset identifier'),
      }).optional().describe('Required for TOKEN_TRANSFER'),
      // CONTRACT_CALL 전용 필드
      calldata: z.string().optional().describe('Hex-encoded calldata (CONTRACT_CALL)'),
      abi: z.array(z.record(z.unknown())).optional().describe('ABI fragment (CONTRACT_CALL)'),
      value: z.string().optional().describe('Native token value (CONTRACT_CALL)'),
      programId: z.string().optional().describe('Solana program ID (CONTRACT_CALL)'),
      // APPROVE 전용 필드
      spender: z.string().optional().describe('Spender address (APPROVE)'),
      // BATCH 전용 필드
      instructions: z.array(z.record(z.unknown())).optional()
        .describe('Batch instructions (BATCH)'),
      // 공통 필드
      network: z.string().optional().describe('Target network (e.g., ethereum-mainnet)'),
      wallet_id: z.string().optional().describe('Target wallet ID (multi-wallet sessions)'),
    },
    async (args) => {
      const body: Record<string, unknown> = {
        to: args.to,
        amount: args.amount,
      };
      if (args.type) body.type = args.type;
      if (args.token) body.token = args.token;
      if (args.calldata !== undefined) body.calldata = args.calldata;
      if (args.abi) body.abi = args.abi;
      if (args.value !== undefined) body.value = args.value;
      if (args.programId !== undefined) body.programId = args.programId;
      if (args.spender !== undefined) body.spender = args.spender;
      if (args.instructions) body.instructions = args.instructions;
      if (args.network !== undefined) body.network = args.network;
      if (args.wallet_id) body.walletId = args.wallet_id;

      const result = await apiClient.post('/v1/transactions/simulate', body);
      return toToolResult(result);
    },
  );
}
```

### 4.2 Tool 이름 결정

| 후보 | 선택 | 근거 |
|------|------|------|
| `simulate_transaction` | X | 기존 MCP tool 네이밍에 `waiaas_` 접두사 없음 (send_token, get_balance) |
| `waiaas_simulate_transaction` | X | 접두사 불필요 -- 이미 WAIaaS MCP 서버 컨텍스트 내 |
| `simulate_transaction` | O | 기존 패턴과 일관: send_token, get_balance, sign_transaction 등 |

**최종 결정: `simulate_transaction`**

기존 MCP tool 이름 패턴을 따라 `simulate_transaction`으로 한다. 위 코드의 `waiaas_simulate_transaction`은 초안이며, 실제 구현 시 `simulate_transaction`으로 등록한다.

### 4.3 MCP Tool 응답 포맷

MCP tool의 응답은 `toToolResult()`를 통해 JSON 문자열로 변환된다. AI 에이전트는 이 JSON을 해석하여 다음을 판단한다:

1. `success` -- 트랜잭션 실행 가능 여부
2. `policy.tier` -- 어떤 보안 계층으로 분류되는지
3. `fee.estimatedFee` -- 예상 수수료
4. `warnings` -- 주의사항 (잔액 부족, 높은 수수료 등)

```

```

## 6. 타임아웃 설계

### 6.1 시뮬레이션 전체 타임아웃

```
기본값: 30초 (config.toml simulation_timeout_ms 또는 Admin Settings)
```

시뮬레이션은 다음 I/O를 포함하므로 타임아웃이 필요:
1. DB 조회 (지갑, 정책) -- 로컬, 빠름
2. 가격 오라클 HTTP 호출 -- 외부 API, 가변
3. RPC buildTransaction -- 외부 RPC, 가변
4. RPC simulateTransaction -- 외부 RPC, 가변
5. RPC getBalance -- 외부 RPC, 가변

**타임아웃 전략**: AbortController를 사용하여 전체 시뮬레이션에 타임아웃을 적용하되, 개별 RPC 호출 타임아웃은 어댑터 레벨에서 기존 메커니즘을 사용한다.

### 6.2 부분 결과 반환

타임아웃 시 Stage 진행 상태에 따라 부분 결과를 반환:
- Stage 3까지 완료 → 정책 결과 + 수수료 null + 시뮬레이션 null
- Stage 5a까지 완료 → 정책 결과 + 수수료 있음 + 시뮬레이션 null
- Stage 5b 타임아웃 → 정책 결과 + 수수료 있음 + 시뮬레이션 timeout error
