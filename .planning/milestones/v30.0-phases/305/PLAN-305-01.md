# Plan 305-01: AuditEventType 확대 + 삽입 지점 매핑 + 쿼리/응답 스키마 설계

## 목표

감사 이벤트 커버리지를 현재 구현 9개에서 18개+로 확대하고, 각 이벤트의 발생 지점(서비스, 파이프라인 스테이지, 정확한 파일/함수)을 매핑한다. AuditLogQuerySchema와 AuditLogResponseSchema를 Zod로 정의하여 다음 플랜(305-02)의 REST API 스펙 기반을 제공한다.

## 컨텍스트

### 현재 코드베이스 감사 이벤트 현황

**audit_log 테이블 스키마 (schema.ts):**

```typescript
export const auditLog = sqliteTable('audit_log', {
  id: integer('id').primaryKey({ autoIncrement: true }),
  timestamp: integer('timestamp', { mode: 'timestamp' }).notNull(),
  eventType: text('event_type').notNull(),
  actor: text('actor').notNull(),
  walletId: text('wallet_id'),
  sessionId: text('session_id'),
  txId: text('tx_id'),
  details: text('details').notNull(),  // JSON
  severity: text('severity').notNull().default('info'),  // CHECK: info | warning | critical
  ipAddress: text('ip_address'),
});
```

**인덱스 5개:**
- `idx_audit_log_timestamp` (timestamp)
- `idx_audit_log_event_type` (event_type)
- `idx_audit_log_wallet_id` (wallet_id)
- `idx_audit_log_severity` (severity)
- `idx_audit_log_wallet_timestamp` (wallet_id, timestamp)

**현재 구현된 이벤트 (9개):**

| # | event_type | 위치 | severity | 구현 방식 |
|---|-----------|------|----------|-----------|
| 1 | `UNLISTED_TOKEN_TRANSFER` | `pipeline/stages.ts` Stage 3 | warning | Drizzle `db.insert(auditLog)` |
| 2 | `AUTO_STOP_TRIGGERED` | `services/autostop-service.ts` | warning | raw SQL `INSERT INTO audit_log` |
| 3 | `KILL_SWITCH_ACTIVATED` | `services/kill-switch-service.ts` executeCascade | critical | raw SQL `INSERT INTO audit_log` |
| 4 | `KILL_SWITCH_ESCALATED` | `services/kill-switch-service.ts` escalate | critical | raw SQL `INSERT INTO audit_log` |
| 5 | `NOTIFICATION_TOTAL_FAILURE` | `notifications/notification-service.ts` | critical | Drizzle `db.insert(auditLog)` |
| 6 | `TX_APPROVED_VIA_TELEGRAM` | `infrastructure/telegram/telegram-bot-service.ts` | info | raw SQL `INSERT INTO audit_log` |
| 7 | `TX_REJECTED_VIA_TELEGRAM` | `infrastructure/telegram/telegram-bot-service.ts` | info | raw SQL `INSERT INTO audit_log` |
| 8 | `TX_CANCELLED_VIA_TELEGRAM` | `infrastructure/telegram/telegram-bot-service.ts` | info | raw SQL `INSERT INTO audit_log` |
| 9 | `SESSION_ISSUED_VIA_TELEGRAM` | `infrastructure/telegram/telegram-bot-service.ts` | info | raw SQL `INSERT INTO audit_log` |

**설계 문서 25 이벤트 타입 목록 (23개 정의됨, 대부분 미구현):**
- AGENT_CREATED, AGENT_ACTIVATED, AGENT_SUSPENDED, AGENT_TERMINATED
- SESSION_ISSUED, SESSION_REVOKED, SESSION_EXPIRED
- TX_REQUESTED, TX_QUEUED, TX_SUBMITTED, TX_CONFIRMED, TX_FAILED, TX_CANCELLED
- POLICY_VIOLATION, POLICY_UPDATED
- KEYSTORE_UNLOCKED, KEYSTORE_LOCKED, KEY_ROTATED
- KILL_SWITCH_ACTIVATED
- DAEMON_STARTED, DAEMON_STOPPED
- AUTH_FAILED, RATE_LIMIT_EXCEEDED

### 격차 분석

설계 문서 25에 23개 이벤트 타입이 정의되어 있으나 실제 구현은 9개뿐이다. 또한 설계 문서 25는 v0.2 시점 정의로 이후 추가된 기능(Telegram bot, notification failure, autostop, owner lifecycle 등)의 이벤트가 반영되지 않았다. 구현 과정에서 설계와 다른 이름으로 이벤트가 추가되었다 (예: `AUTO_STOP_TRIGGERED`는 설계 문서에 없음).

**결론:** 설계 문서 25의 기존 목록을 기반으로 하되, 현재 코드베이스의 실제 기능을 반영하여 이벤트 목록을 재정의한다.

## 설계 내용

### 1. AuditEventType 확대 (9개 → 18개+)

기존 9개 구현 이벤트를 유지하면서, 설계 문서 25의 미구현 핵심 이벤트 + 코드베이스 기능 기반 신규 이벤트를 추가한다.

#### 1.1 기존 유지 (9개)

이미 코드에 구현되어 있어 변경하지 않는다.

| # | event_type | severity | 비고 |
|---|-----------|----------|------|
| 1 | `UNLISTED_TOKEN_TRANSFER` | warning | Pipeline Stage 3 |
| 2 | `AUTO_STOP_TRIGGERED` | warning | AutoStopService |
| 3 | `KILL_SWITCH_ACTIVATED` | critical | KillSwitchService cascade |
| 4 | `KILL_SWITCH_ESCALATED` | critical | KillSwitchService escalate |
| 5 | `NOTIFICATION_TOTAL_FAILURE` | critical | NotificationService |
| 6 | `TX_APPROVED_VIA_TELEGRAM` | info | TelegramBotService |
| 7 | `TX_REJECTED_VIA_TELEGRAM` | info | TelegramBotService |
| 8 | `TX_CANCELLED_VIA_TELEGRAM` | info | TelegramBotService |
| 9 | `SESSION_ISSUED_VIA_TELEGRAM` | info | TelegramBotService |

#### 1.2 신규 추가 (11개)

설계 문서 25의 미구현 이벤트 중 운영 가시성에 필수적인 것과, 코드베이스의 현재 기능에서 도출한 이벤트를 추가한다.

| # | event_type | severity | 트리거 조건 | 근거 |
|---|-----------|----------|------------|------|
| 10 | `WALLET_CREATED` | info | 월렛 생성 (POST /v1/wallets) | 설계 문서 25 `AGENT_CREATED` 대응. 용어 변경 반영 |
| 11 | `WALLET_SUSPENDED` | warning | 월렛 정지 (status→SUSPENDED) | 설계 문서 25 `AGENT_SUSPENDED` 대응 |
| 12 | `SESSION_CREATED` | info | 세션 생성 (POST /v1/sessions) | 설계 문서 25 `SESSION_ISSUED` 대응 |
| 13 | `SESSION_REVOKED` | info | 세션 폐기 (DELETE /v1/sessions/:id) | 설계 문서 25 `SESSION_REVOKED` |
| 14 | `TX_SUBMITTED` | info | Pipeline Stage 5d 온체인 제출 완료 | 설계 문서 25 + 32 명시 |
| 15 | `TX_CONFIRMED` | info | Pipeline Stage 6 온체인 확정 | 설계 문서 25 + 32 명시 |
| 16 | `TX_FAILED` | warning | Pipeline 실패 (Stage 5 또는 Stage 6) | 설계 문서 25 + 32 명시 |
| 17 | `POLICY_DENIED` | warning | Pipeline Stage 3 정책 거부 | 설계 문서 25 `POLICY_VIOLATION` 대응 |
| 18 | `OWNER_REGISTERED` | info | Owner 주소 최초 등록 (PUT /v1/wallets/:id/owner) | 보안 이벤트. Owner lifecycle 추적 |
| 19 | `MASTER_AUTH_FAILED` | critical | masterAuth 미들웨어 인증 실패 | 보안 감사 필수. 무차별 대입 시도 탐지 |
| 20 | `KILL_SWITCH_RECOVERED` | warning | Kill Switch 복구 (POST /v1/admin/recover) | ACTIVATED/ESCALATED 대칭. 복구 추적 |

**전체 합계: 20개 이벤트**

#### 1.3 의도적 제외 이벤트

설계 문서 25에 정의되어 있으나 현재 구현에서 제외하는 이벤트와 그 사유:

| event_type | 제외 사유 |
|-----------|----------|
| `AGENT_ACTIVATED` | v1.4.2에서 agent→wallet 변경. 월렛 생성 시 바로 ACTIVE. 별도 활성화 단계 없음 |
| `AGENT_TERMINATED` | 월렛 삭제 기능 미구현 (RESTRICT FK). SUSPENDED만 존재 |
| `SESSION_EXPIRED` | 만료는 수동 이벤트가 아님. JWT exp 검증 시 거부하지만 audit 기록 불필요 (조회 시 필터 가능) |
| `TX_REQUESTED` | Stage 1에서 transactions 테이블에 INSERT. 별도 audit 기록 시 중복 (TX 자체가 기록) |
| `TX_QUEUED` | transactions.status 변경으로 추적 가능. audit 중복 |
| `TX_CANCELLED` | 일반 취소는 TX 상태로 추적. Telegram 경유 취소만 별도 기록 (이미 구현) |
| `POLICY_UPDATED` | 정책 CRUD 자체보다 정책 위반이 감사 대상. CRUD는 masterAuth 보호 |
| `KEYSTORE_UNLOCKED/LOCKED` | Self-hosted 데몬 시작/종료 시 자동. 별도 추적 불필요 |
| `KEY_ROTATED` | 키 로테이션 미구현 (sweepAll 기반 — 별도 마일스톤) |
| `DAEMON_STARTED/STOPPED` | 운영 이벤트이지 보안 이벤트가 아님. 로그 파일로 충분 |
| `RATE_LIMIT_EXCEEDED` | 현재 rate limiting 미구현 (Self-hosted, localhost-only 전제) |

#### 1.4 AuditEventType enum 정의 (Zod SSoT)

```typescript
// packages/core/src/schemas/audit.ts (신규)

import { z } from 'zod';

/**
 * 감사 이벤트 타입 enum.
 *
 * 카테고리별 분류:
 * - WALLET_*: 월렛 생명주기
 * - SESSION_*: 세션 생명주기
 * - TX_*: 트랜잭션 파이프라인
 * - POLICY_*: 정책 엔진
 * - KILL_SWITCH_*: Kill Switch
 * - AUTO_STOP_*: AutoStop
 * - MASTER_AUTH_*: 관리자 인증
 * - OWNER_*: Owner 생명주기
 * - NOTIFICATION_*: 알림 시스템
 */
export const AUDIT_EVENT_TYPES = [
  // 월렛 생명주기
  'WALLET_CREATED',
  'WALLET_SUSPENDED',

  // 세션 생명주기
  'SESSION_CREATED',
  'SESSION_REVOKED',
  'SESSION_ISSUED_VIA_TELEGRAM',

  // 트랜잭션 파이프라인
  'TX_SUBMITTED',
  'TX_CONFIRMED',
  'TX_FAILED',
  'TX_APPROVED_VIA_TELEGRAM',
  'TX_REJECTED_VIA_TELEGRAM',
  'TX_CANCELLED_VIA_TELEGRAM',
  'UNLISTED_TOKEN_TRANSFER',

  // 정책 엔진
  'POLICY_DENIED',

  // Kill Switch
  'KILL_SWITCH_ACTIVATED',
  'KILL_SWITCH_ESCALATED',
  'KILL_SWITCH_RECOVERED',

  // AutoStop
  'AUTO_STOP_TRIGGERED',

  // 관리자 인증
  'MASTER_AUTH_FAILED',

  // Owner 생명주기
  'OWNER_REGISTERED',

  // 알림 시스템
  'NOTIFICATION_TOTAL_FAILURE',
] as const;

export const AuditEventTypeSchema = z.enum(AUDIT_EVENT_TYPES);
export type AuditEventType = z.infer<typeof AuditEventTypeSchema>;

export const AUDIT_SEVERITIES = ['info', 'warning', 'critical'] as const;
export const AuditSeveritySchema = z.enum(AUDIT_SEVERITIES);
export type AuditSeverity = z.infer<typeof AuditSeveritySchema>;
```

### 2. 이벤트별 삽입 지점 매핑

각 신규 이벤트가 코드베이스의 어느 파일, 어느 함수에서 삽입되어야 하는지 매핑한다.

#### 2.1 WALLET_CREATED

| 항목 | 값 |
|------|-----|
| **파일** | `packages/daemon/src/api/routes/wallets.ts` |
| **함수** | `createWalletRoute` 핸들러 (POST /wallets) |
| **삽입 시점** | wallets 테이블 INSERT 직후 |
| **actor** | `'master'` (masterAuth 보호 엔드포인트) |
| **walletId** | 생성된 wallet.id |
| **severity** | `'info'` |
| **details** | `{ chain, environment, publicKey }` |
| **구현 방식** | Drizzle `db.insert(auditLog)` |

#### 2.2 WALLET_SUSPENDED

| 항목 | 값 |
|------|-----|
| **파일** | `packages/daemon/src/api/routes/wallets.ts` (수동 정지) / `packages/daemon/src/services/autostop-service.ts` (자동 정지) / `packages/daemon/src/services/kill-switch-service.ts` (cascade) |
| **함수** | 수동: PATCH /wallets/:id 핸들러, 자동: AutoStopService.suspend(), cascade: executeCascade() |
| **삽입 시점** | wallets.status = 'SUSPENDED' UPDATE 직후 |
| **actor** | 수동: `'master'`, 자동: `'autostop'`, cascade: `activatedBy` |
| **walletId** | 정지된 wallet.id |
| **severity** | `'warning'` |
| **details** | `{ reason, previousStatus, trigger: 'manual' | 'autostop' | 'kill_switch' }` |
| **비고** | AutoStopService에서 이미 `AUTO_STOP_TRIGGERED`를 기록하므로, WALLET_SUSPENDED는 상태 변경 자체에 대한 별도 기록. Kill Switch cascade에서도 각 월렛 정지를 개별 기록하면 과도 -- cascade 전체를 `KILL_SWITCH_ACTIVATED` 하나로 커버. **따라서 수동 정지 (PATCH /wallets/:id)에서만 삽입.** |

#### 2.3 SESSION_CREATED

| 항목 | 값 |
|------|-----|
| **파일** | `packages/daemon/src/api/routes/sessions.ts` |
| **함수** | `createSessionRoute` 핸들러 (POST /sessions) |
| **삽입 시점** | sessions 테이블 INSERT + JWT 발급 직후 |
| **actor** | `'master'` (masterAuth 보호 엔드포인트) |
| **walletId** | 세션에 연결된 첫 번째 wallet.id |
| **sessionId** | 생성된 session.id |
| **severity** | `'info'` |
| **details** | `{ walletIds, constraints: { maxAmountPerTx, maxTotalAmount, allowedRecipients } }` |

#### 2.4 SESSION_REVOKED

| 항목 | 값 |
|------|-----|
| **파일** | `packages/daemon/src/api/routes/sessions.ts` |
| **함수** | `revokeSessionRoute` 핸들러 (DELETE /sessions/:id) |
| **삽입 시점** | sessions.revoked_at UPDATE 직후 |
| **actor** | `'master'` (masterAuth 보호) 또는 `'session'` (sessionAuth 자기 폐기 — 현재 미지원이지만 향후 가능) |
| **sessionId** | 폐기된 session.id |
| **severity** | `'info'` |
| **details** | `{ sessionId, revokedAt }` |

#### 2.5 TX_SUBMITTED

| 항목 | 값 |
|------|-----|
| **파일** | `packages/daemon/src/pipeline/stages.ts` |

| **txId** | `ctx.txId` |

| **actor** | `recoveredBy` 파라미터 |
| **severity** | `'warning'` |
| **details** | `{ previousState: 'SUSPENDED' | 'LOCKED', recoveredBy }` |

### 3. 이벤트별 details JSON 스키마

각 이벤트의 details 필드에 저장되는 JSON 구조를 Zod로 정의한다. 구현 시 type-safe 보장.

```typescript
// packages/core/src/schemas/audit.ts (계속)

// ─── details 스키마 (이벤트별) ───

export const WalletCreatedDetailsSchema = z.object({
  chain: z.string(),
  environment: z.string(),
  publicKey: z.string(),
});

export const WalletSuspendedDetailsSchema = z.object({
  reason: z.string(),
  previousStatus: z.string(),
  trigger: z.enum(['manual', 'autostop', 'kill_switch']),
});

export const SessionCreatedDetailsSchema = z.object({
  walletIds: z.array(z.string()),
  constraints: z.record(z.unknown()).optional(),
});

export const SessionRevokedDetailsSchema = z.object({
  sessionId: z.string(),
  revokedAt: z.number(),
});

export const TxSubmittedDetailsSchema = z.object({
  txHash: z.string(),
  chain: z.string(),
  network: z.string(),
  type: z.string(),
```

### 4. AuditLogQuerySchema 설계 (AUDIT-01)

외부에서 감사 로그를 조회할 때 사용하는 쿼리 파라미터 스키마.

```typescript
// packages/core/src/schemas/audit.ts (계속)

/**
 * GET /v1/audit-logs 쿼리 파라미터 스키마.
 *
 * 필터 조합:
 * - wallet_id: 특정 월렛의 감사 로그만 조회
 * - event_type: 특정 이벤트 타입만 조회 (AUDIT_EVENT_TYPES enum)
 * - severity: 특정 심각도만 조회 (info | warning | critical)
 * - from / to: 시간 범위 (Unix epoch 초)
 * - tx_id: 특정 트랜잭션 관련 감사 로그
 * - cursor: id 기반 커서 (다음 페이지)
 * - limit: 페이지 크기 (기본 50, 최대 200)
 */
export const AuditLogQuerySchema = z.object({
  // ─── 필터 ───
  wallet_id: z.string().uuid().optional().openapi({
    description: '월렛 ID 필터',
    example: '01936d3c-7f8a-7b00-9e4d-abcdef123456',
  }),
  event_type: AuditEventTypeSchema.optional().openapi({
    description: '이벤트 타입 필터 (AUDIT_EVENT_TYPES 중 하나)',
    example: 'KILL_SWITCH_ACTIVATED',
  }),
  severity: AuditSeveritySchema.optional().openapi({
    description: '심각도 필터 (info | warning | critical)',
    example: 'critical',
  }),
  from: z.coerce.number().int().optional().openapi({
    description: '시작 시각 (Unix epoch 초, inclusive)',
    example: 1772438400,
  }),
  to: z.coerce.number().int().optional().openapi({
    description: '종료 시각 (Unix epoch 초, inclusive)',
    example: 1772524800,
  }),
  tx_id: z.string().uuid().optional().openapi({
    description: '트랜잭션 ID 필터',
  }),

  // ─── 페이지네이션 ───
  cursor: z.coerce.number().int().positive().optional().openapi({
    description: 'ID 기반 커서 (이전 응답의 nextCursor 값)',
  }),
  limit: z.coerce.number().int().min(1).max(200).default(50).optional().openapi({
    description: '페이지 크기 (기본 50, 최대 200)',
    example: 50,
  }),
});

export type AuditLogQuery = z.infer<typeof AuditLogQuerySchema>;
```

**설계 결정:**

| 항목 | 결정 | 근거 |
|------|------|------|
| cursor 타입 | `number` (id AUTOINCREMENT) | UUID cursor보다 단순. audit_log.id는 AUTOINCREMENT로 단조 증가 보장 |
| cursor 인코딩 | 평문 정수 | Base64url 인코딩 불필요. id는 연속 정수이므로 클라이언트가 조작해도 보안 위험 없음 (조회 전용) |
| limit 기본값 | 50 | 기존 transactions (20)보다 크지만, 감사 로그는 관리자 전용 저빈도 조회. 200 상한으로 대량 내보내기도 가능 |
| from/to 타입 | `number` (epoch 초) | 프로젝트 전체 타임스탬프 규칙 (SQLite 초 단위). 기존 incoming.ts since/until 패턴과 동일 |
| event_type 필터 | enum 기반 | 잘못된 이벤트 타입 입력 시 Zod 검증에서 즉시 거부 |

```typescript
    description: '관련 트랜잭션 ID (없으면 null)',

    description: '이벤트별 상세 데이터 (JSON)',
  }),

    description: '심각도',
    example: 'critical',
  }),
  ipAddress: z.string().nullable().openapi({

  }),

export type AuditLogItem = z.infer<typeof AuditLogItemSchema>;

 *

export const AuditLogResponseSchema = z.object({
  data: z.array(AuditLogItemSchema).openapi({
    description: '감사 로그 항목 목록 (id DESC 정렬)',
  }),
  nextCursor: z.number().int().nullable().openapi({
    description: '다음 페이지 커서 (null이면 마지막 페이지). 다음 요청의 cursor 파라미터에 전달',

  }),
});
```

**응답 필드 설계 결정:**

| 항목 | 결정 | 근거 |
|------|------|------|
| `data` (not `items`) | 기존 incoming.ts는 `data`, transactions.ts는 `items` 사용. 신규 API이므로 `data` 채택 (REST 일반 관례) |
| `nextCursor` | `number` (id 값) | Base64 불필요. audit_log.id는 공개 가능한 AUTOINCREMENT 정수 |
| `total` | optional | 전체 건수 조회는 COUNT(*) 필요. 대량 데이터에서 성능 이슈 가능. 관리자가 필요 시만 제공 |
| `details` | `Record<string, unknown>` | 이벤트별 details 구조가 다르므로 응답 스키마에서는 generic. 클라이언트가 eventType에 따라 파싱 |

### 7. 감사 로그 헬퍼 함수 설계

현재 코드베이스에서 audit_log 삽입은 두 가지 방식으로 혼재한다:
1. Drizzle ORM: `db.insert(auditLog).values({...})` (stages.ts, notification-service.ts)
2. raw SQL: `sqlite.prepare('INSERT INTO audit_log ...').run(...)` (kill-switch, autostop, telegram-bot)

11개 이벤트를 추가할 때 일관된 헬퍼 함수를 제공한다.

```typescript
// packages/daemon/src/infrastructure/database/audit-helper.ts (신규)

import type { Database as SQLiteDatabase } from 'better-sqlite3';
import type { AuditEventType, AuditSeverity } from '@waiaas/core';

export interface AuditEntry {
  eventType: AuditEventType;
  actor: string;
  walletId?: string;
  sessionId?: string;
  txId?: string;
  details: Record<string, unknown>;
  severity: AuditSeverity;
  ipAddress?: string;
}

/**
 * Best-effort 감사 로그 삽입.
 *
 * try-catch로 감싸서 감사 로그 실패가 메인 로직을 중단하지 않도록 한다.
 * raw SQL 사용 (better-sqlite3 동기 API) — Drizzle ORM 의존 없이 사용 가능.
 */
export function insertAuditLog(sqlite: SQLiteDatabase, entry: AuditEntry): void {
  try {
    const now = Math.floor(Date.now() / 1000);
    sqlite
      .prepare(
        'INSERT INTO audit_log (timestamp, event_type, actor, wallet_id, session_id, tx_id, details, severity, ip_address) VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?)',
      )
      .run(
        now,
        entry.eventType,
        entry.actor,
        entry.walletId ?? null,
        entry.sessionId ?? null,
        entry.txId ?? null,
        JSON.stringify(entry.details),
        entry.severity,
        entry.ipAddress ?? null,
      );
  } catch {
    // Best-effort: 감사 로그 실패가 메인 로직을 중단해서는 안 됨
  }
}
```

**설계 결정:**
- raw SQL 방식 채택 (better-sqlite3 동기 API). 이유: master-auth 미들웨어 등 Drizzle DB 인스턴스 접근이 어려운 위치에서도 사용 가능
- `try-catch` 내장. 기존 코드 패턴과 동일 (best-effort audit logging)
- 기존 구현된 9개 이벤트는 일괄 리팩토링하지 않음 (하위 호환). 신규 11개만 헬퍼 사용

### 8. 인덱스 최적화 검토

현재 5개 인덱스가 AuditLogQuerySchema의 필터를 커버하는지 검증한다.

| 필터 | 사용 인덱스 | 커버 여부 |

| `event_type` | `idx_audit_log_event_type` | O |
| `severity` | `idx_audit_log_severity` | O |
| `from/to` (timestamp) | `idx_audit_log_timestamp` | O |
| `wallet_id + from/to` | `idx_audit_log_wallet_timestamp` | O |
| `tx_id` | (없음) | **X** |
| `cursor` (id) | PRIMARY KEY | O |

**신규 인덱스 필요:**

```sql
CREATE INDEX idx_audit_log_tx_id ON audit_log(tx_id);
```

tx_id 필터를 지원하기 위해 인덱스 1개를 추가한다. 설계 문서 25에 반영 필요.

### 9. 설계 문서 25 갱신 내용 요약

설계 문서 25 (sqlite-schema.md) 섹션 2.5에 다음을 갱신한다:

1. **이벤트 타입 목록 테이블**: 기존 23개 → 20개로 재정의 (현재 코드베이스 기준)
2. **각 이벤트의 details JSON 스키마**: 이벤트별 details 구조 명시
3. **인덱스**: `idx_audit_log_tx_id` 추가 (5개 → 6개)
4. **용어 변경**: agent_id → wallet_id 반영 (v1.4.2)

## 검증 기준

1. AuditEventType enum이 20개 이벤트를 정의하고, AUDIT_EVENT_TYPES 배열이 Zod enum의 SSoT가 된다
2. 각 신규 이벤트(11개)의 삽입 지점이 정확한 파일/함수/시점으로 매핑된다
3. AuditLogQuerySchema가 6개 필터(wallet_id, event_type, from/to, tx_id, severity)를 지원한다
4. AuditLogResponseSchema가 cursor pagination(nextCursor, hasMore)과 data 배열을 포함한다
5. 이벤트별 details JSON 스키마가 Zod로 정의되어 type-safety를 보장한다

## 산출물

- AuditEventType enum (20개) + AUDIT_EVENT_TYPES 배열
- 이벤트별 삽입 지점 매핑 (11개 신규, 6개 파일)
- AuditLogQuerySchema, AuditLogResponseSchema (Zod)
- 이벤트별 details JSON 스키마 (11개)
- insertAuditLog 헬퍼 함수 설계
- idx_audit_log_tx_id 신규 인덱스 사양
