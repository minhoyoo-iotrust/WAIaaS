# Plan 306-01: 아카이브 바이너리 포맷 + BackupService 암호화 확장 설계

## 목표

백업 아카이브 바이너리 포맷을 바이트 수준으로 사양화하고, BackupService가 기존 인터페이스를 유지하면서 Argon2id KDF + AES-256-GCM 암호화 + VACUUM INTO 원자적 스냅샷을 수행하도록 확장 설계한다.

## 컨텍스트

### 현재 상태

- `BackupService` (packages/daemon/src/infrastructure/backup/backup-service.ts): `createBackup(version)`, `restoreLatest()`, `restore(backupDir)`, `listBackups()`, `pruneBackups(keep=5)` 메서드 제공
- 백업 방식: `copyFileSync`로 평문 파일 복사 (DB, WAL, SHM, config.toml)
- 백업 경로: `{dataDir}/backups/pre-upgrade-{version}-{timestamp}/` (디렉토리 기반)
- `waiaas update --rollback` 내부에서만 사용 (독립 CLI 커맨드 없음)
- **암호화 없음** -- 백업 파일이 평문으로 디스크에 저장됨

### 기존 암호화 인프라 (재사용 대상)

- **Argon2id KDF**: `packages/daemon/src/infrastructure/keystore/crypto.ts`
  - 파라미터: `memoryCost=65536` (64 MiB), `timeCost=3`, `parallelism=4`, `hashLength=32`
  - `deriveKey(password, salt?)` -- 32바이트 AES-256 키 유도
  - `encrypt(plaintext, password)` -- AES-256-GCM 암호화
  - `decrypt(encrypted, password)` -- AES-256-GCM 복호화
- **AES-256-GCM**: 12바이트 IV (nonce), 16바이트 authTag
- CSPRNG salt: 16바이트 (`crypto.randomBytes(16)`)

### 데이터 디렉토리 구조 (doc 24)

```
~/.waiaas/
  config.toml              -- 데몬 설정 (600)
  .master-password         -- 마스터 패스워드 파일 (600)
  data/waiaas.db           -- SQLite 데이터베이스 (600)
  data/waiaas.db-wal       -- WAL 파일 (선택적)
  data/waiaas.db-shm       -- SHM 파일 (선택적)
  keystore/<wallet-id>.json -- 월렛별 키스토어 파일 (600)
  logs/                    -- 로그 디렉토리
  backups/                 -- 백업 디렉토리 (700)
```

## 1. 백업 아카이브 바이너리 포맷 (BKUP-01)

### 1.1 설계 원칙

| 원칙 | 설명 |
|------|------|
| **단일 파일** | 디렉토리 기반 → 단일 `.waiaas-backup` 파일로 전환. USB/클라우드 이동 편의 |
| **자기 기술적** | 파일 하나에 복호화에 필요한 모든 메타데이터 포함 (KDF 파라미터, 솔트, 논스) |
| **AEAD 무결성** | AES-256-GCM authTag로 변조 탐지. 잘못된 비밀번호도 authTag 불일치로 즉시 감지 |
| **버전 호환** | 포맷 버전 필드로 향후 확장 (알고리즘 변경 등) 대비 |
| **스트리밍 불가** | 페이로드 전체를 메모리에 올린 후 암호화/복호화 (SQLite DB 파일은 수~수십 MB, 메모리 부담 미미) |

### 1.2 파일 확장자

- `.waiaas-backup` -- 암호화된 백업 아카이브
- 파일명 규칙: `waiaas-backup-{YYYYMMDD-HHmmss}.waiaas-backup`
- 예: `waiaas-backup-20260303-143022.waiaas-backup`

### 1.3 전체 바이너리 레이아웃

```
Offset   Size      Field                  Description
──────   ────      ─────                  ───────────
0x0000   8B        Magic Number           "WAIAAS\x00\x01" (ASCII 6바이트 + 2바이트 padding/version hint)
0x0008   2B        Format Version         uint16 LE (현재: 0x0001)
0x000A   2B        Reserved               0x0000 (향후 플래그 용도)
0x000C   4B        Metadata Length         uint32 LE (JSON 메타데이터 바이트 길이)
0x0010   16B       KDF Salt               Argon2id 솔트 (CSPRNG 128비트)
0x0020   12B       AES-GCM Nonce          IV (96비트 논스)
0x002C   16B       AES-GCM Auth Tag       인증 태그 (128비트)
0x003C   NB        Metadata (JSON)        평문 JSON 메타데이터 (N = Metadata Length 값)
0x003C+N MB        Encrypted Payload      AES-256-GCM 암호문 (M = EOF까지)
```

**총 고정 헤더 크기:** 60바이트 (0x3C)

### 1.4 각 필드 상세

#### 1.4.1 Magic Number (8바이트)

```
바이트: 0x57 0x41 0x49 0x41 0x41 0x53 0x00 0x01
텍스트:  W    A    I    A    A    S   \0   \x01
```

- 처음 6바이트: ASCII "WAIAAS" -- 파일 포맷 식별자
- 7번째 바이트: 0x00 (NULL terminator / separator)
- 8번째 바이트: 0x01 (format family hint, 향후 완전히 다른 포맷 계열 구분용)

검증: 파일 읽기 시 첫 8바이트를 비교하여 WAIaaS 백업 파일인지 확인한다. 불일치 시 `INVALID_BACKUP_FORMAT` 에러.

#### 1.4.2 Format Version (2바이트)

- uint16 LE (Little-Endian)
- 현재 값: `0x0001` (버전 1)
- 향후 KDF 알고리즘 변경, 압축 추가 등의 경우 버전을 증가시킨다
- 복원 시 지원하지 않는 버전이면 `UNSUPPORTED_BACKUP_VERSION` 에러

#### 1.4.3 Reserved (2바이트)

- 현재 0x0000
- 향후 플래그 비트맵으로 사용 가능 (예: bit 0 = 압축 여부, bit 1 = 키스토어 포함 여부)

#### 1.4.4 Metadata Length (4바이트)

- uint32 LE
- 뒤따르는 평문 JSON 메타데이터의 바이트 길이
- 최대값: 4 GiB (실제로 메타데이터는 수백 바이트 이내)

#### 1.4.5 KDF Salt (16바이트)

- `crypto.randomBytes(16)` -- CSPRNG 128비트 솔트
- Argon2id 키 유도에 사용
- 매 백업마다 새로 생성 (키스토어의 per-file salt 패턴과 동일)

#### 1.4.6 AES-GCM Nonce (12바이트)

- `crypto.randomBytes(12)` -- 96비트 GCM 논스
- 단일 키로 한 번만 암호화하므로 논스 재사용 위험 없음

#### 1.4.7 AES-GCM Auth Tag (16바이트)

- `cipher.getAuthTag()` -- 128비트 인증 태그
- 암호화 완료 후 추출하여 헤더에 기록
- 복호화 시 `decipher.setAuthTag(authTag)`으로 설정. 불일치 시 `INVALID_MASTER_PASSWORD` 에러

#### 1.4.8 Metadata (가변 길이, 평문)

평문 JSON으로 저장한다. 암호화하지 않는 이유:
- 메타데이터에 민감 정보 없음 (타임스탬프, 버전, 파일 목록 크기)
- 복호화 전에 백업 정보를 확인할 수 있어야 함 (`waiaas backup list` 등)

```json
{
  "created_at": "2026-03-03T14:30:22.000Z",
  "daemon_version": "2.9.0",
  "schema_version": 33,
  "kdf": "argon2id",
  "kdf_params": {
    "memory_cost": 65536,
    "time_cost": 3,
    "parallelism": 4,
    "hash_length": 32
  },
  "contents": {
    "database": { "name": "waiaas.db", "size": 2457600 },
    "config": { "name": "config.toml", "size": 1024 },
    "keystore_files": [
      { "name": "wallet-abc.json", "size": 512 },
      { "name": "wallet-xyz.json", "size": 512 }
    ]
  },
  "checksum": "sha256:a1b2c3d4..."
}
```

메타데이터 필드:

| 필드 | 타입 | 설명 |
|------|------|------|
| `created_at` | ISO 8601 | 백업 생성 시각 (UTC) |
| `daemon_version` | semver | 백업 생성 시점의 데몬 버전 |
| `schema_version` | integer | DB 스키마 버전 (schema_version 테이블 참조) |
| `kdf` | string | KDF 알고리즘 ("argon2id") |
| `kdf_params` | object | KDF 파라미터 (자기 기술적 포맷 유지) |
| `contents` | object | 아카이브에 포함된 파일 목록 및 크기 |
| `checksum` | string | 평문 페이로드의 SHA-256 해시 (복호화 후 무결성 검증) |

#### 1.4.9 Encrypted Payload (가변 길이)

평문 페이로드를 AES-256-GCM으로 암호화한 결과이다. 평문 페이로드는 tar-like 연결 포맷이다.

**평문 페이로드 내부 구조 (암호화 전):**

```
[Entry 1] [Entry 2] [Entry 3] ... [Entry N]
```

각 Entry 구조:

```
Offset  Size    Field           Description
──────  ────    ─────           ───────────
0x00    2B      Name Length     파일명 바이트 길이 (uint16 LE)
0x02    NB      Name            파일명 (UTF-8, 예: "waiaas.db", "keystore/wallet-abc.json")
N+2     8B      Data Length     파일 데이터 바이트 길이 (uint64 LE -- 큰 DB 대비)
N+10    MB      Data            파일 바이너리 데이터
```

**포함 파일 목록 (수집 순서):**

| # | 파일 | 필수 | 비고 |
|---|------|------|------|
| 1 | `waiaas.db` | 필수 | VACUUM INTO로 원자적 스냅샷 (WAL/SHM 불필요) |
| 2 | `config.toml` | 선택 | 존재하면 포함 |
| 3~ | `keystore/<wallet-id>.json` | 선택 | 키스토어 디렉토리의 모든 .json 파일 |

**VACUUM INTO 사용으로 WAL/SHM 파일 불필요:**
- 기존 BackupService: `copyFileSync`로 DB + WAL + SHM 3개 파일 복사 (일관성 문제 가능)
- 신규: `VACUUM INTO '/tmp/waiaas-backup-{uuid}.db'`로 일관된 단일 DB 파일 생성
- VACUUM INTO는 트랜잭션 중 호출 가능하며 원본 DB에 영향 없음

### 1.5 크기 추정

일반적인 WAIaaS 인스턴스 (월렛 2개, DB 10MB):

```
고정 헤더:      60 B
메타데이터:     ~400 B (JSON)
암호화 페이로드: ~10.5 MB (DB 10MB + config 1KB + keystore 2KB)
────────────────────────
총 아카이브:    ~10.5 MB
```

AES-256-GCM은 스트림 암호이므로 암호문 크기 = 평문 크기 (패딩 없음).

## 2. BackupService 암호화 확장 설계 (BKUP-02)

### 2.1 설계 원칙

| 원칙 | 설명 |
|------|------|
| **기존 인터페이스 유지** | `createBackup()`, `restore()`, `listBackups()`, `pruneBackups()` 시그니처 보존 |
| **KDF 재사용** | `crypto.ts`의 `deriveKey()`, KDF_PARAMS 재사용 (keystore와 동일 파라미터) |
| **VACUUM INTO** | `copyFileSync` → SQLite `VACUUM INTO`로 원자적 스냅샷 (WAL 통합) |
| **메모리 안전** | 파생 키를 사용 후 즉시 제로화 (`key.fill(0)`) |

### 2.2 확장 클래스 구조

기존 `BackupService`를 확장하지 않고, **새 클래스 `EncryptedBackupService`**를 신규 생성한다. 기존 `BackupService`는 업그레이드 롤백 전용으로 유지한다.

```
packages/daemon/src/infrastructure/backup/
  backup-service.ts            -- 기존 (비암호화, 업그레이드 롤백 전용)
  encrypted-backup-service.ts  -- 신규 (암호화 백업/복원)
  backup-format.ts             -- 신규 (아카이브 포맷 읽기/쓰기 유틸)
  index.ts                     -- export 추가
```

**분리 근거:**
- 기존 `BackupService`는 `waiaas update --rollback`에서 사용 (비밀번호 불필요, 빠른 복사)
- 신규 `EncryptedBackupService`는 `waiaas backup/restore`에서 사용 (비밀번호 필수, 암호화)
- 두 용도가 다르므로 단일 클래스 확장보다 독립 클래스가 적합

### 2.3 EncryptedBackupService 인터페이스

```typescript
export class EncryptedBackupService {
  constructor(
    private readonly dataDir: string,
    private readonly backupsDir: string,   // config.toml backup_dir 또는 기본값
    private readonly db: DatabaseType,     // better-sqlite3 인스턴스 (VACUUM INTO용)
  ) {}

  /**
   * 암호화 백업 생성.
   *
   * 1. VACUUM INTO로 DB 스냅샷 생성 (임시 파일)
   * 2. config.toml, keystore/*.json 수집
   * 3. 평문 페이로드 조립 (tar-like entries)
   * 4. SHA-256 체크섬 계산
   * 5. Argon2id KDF로 마스터 비밀번호 → AES-256 키 유도
   * 6. AES-256-GCM 암호화
   * 7. 아카이브 헤더 + 메타데이터 + 암호문 조립
   * 8. 파일 기록 (0o600)
   * 9. 보존 정책 적용 (pruneBackups)
   * 10. 임시 파일 정리
   *
   * @param masterPassword 마스터 비밀번호
   * @returns 생성된 아카이브 파일 절대 경로
   */
  async createBackup(masterPassword: string): Promise<string>;

  /**
   * 암호화 백업에서 복원.
   *
   * 1. 아카이브 파일 읽기 + 매직 넘버/버전 검증
   * 2. 메타데이터 파싱
   * 3. Argon2id KDF로 키 유도
   * 4. AES-256-GCM 복호화 (authTag 검증)
   * 5. SHA-256 체크섬 검증
   * 6. 기존 데이터 백업 (안전 장치: data/ → data.bak-{timestamp}/)
   * 7. 페이로드에서 파일 추출 + 배치
   * 8. 기존 백업 정리 (성공 시)
   *
   * @param archivePath 아카이브 파일 절대 경로
   * @param masterPassword 마스터 비밀번호
   * @throws INVALID_MASTER_PASSWORD 비밀번호 불일치
   * @throws INVALID_BACKUP_FORMAT 매직 넘버/버전 불일치
   * @throws BACKUP_CORRUPTED 체크섬 불일치
   */
  async restore(archivePath: string, masterPassword: string): Promise<void>;

  /**
   * 백업 아카이브 목록 조회 (메타데이터 포함).
   *
   * 각 파일의 헤더 + 메타데이터만 읽어 복호화 없이 정보 반환.
   *
   * @returns 최신순 정렬된 백업 정보 배열
   */
  listBackups(): BackupInfo[];

  /**
   * 보존 정책 적용.
   *
   * @param keep 보존할 최대 백업 수 (기본 7)
   * @returns 삭제된 백업 수
   */
  pruneBackups(keep?: number): number;

  /**
   * 아카이브 파일의 메타데이터를 복호화 없이 조회.
   *
   * @param archivePath 아카이브 파일 절대 경로
   * @returns 메타데이터 (created_at, daemon_version, schema_version, contents)
   */
  inspectBackup(archivePath: string): BackupMetadata;
}
```

### 2.4 타입 정의

```typescript
/** 백업 목록에 표시할 정보 (복호화 불필요) */
export interface BackupInfo {
  path: string;           // 아카이브 파일 절대 경로
  createdAt: string;      // ISO 8601
  daemonVersion: string;  // 백업 시점 데몬 버전
  schemaVersion: number;  // DB 스키마 버전
  fileSize: number;       // 아카이브 파일 크기 (bytes)
  fileCount: number;      // 포함된 파일 수
}

/** 아카이브 메타데이터 (평문 JSON 섹션) */
export interface BackupMetadata {
  created_at: string;
  daemon_version: string;
  schema_version: number;
  kdf: string;
  kdf_params: {
    memory_cost: number;
    time_cost: number;
    parallelism: number;
    hash_length: number;
  };
  contents: {
    database: { name: string; size: number };
    config?: { name: string; size: number };
    keystore_files: Array<{ name: string; size: number }>;
  };
  checksum: string;
}
```

### 2.5 암호화 플로우 상세

#### 2.5.1 백업 생성 플로우

```
createBackup(masterPassword)
    |
    v
[1] VACUUM INTO '{tmpDir}/waiaas-snap-{uuid}.db'
    -- 원본 DB 무중단, WAL 통합된 단일 파일 생성
    -- 데몬 실행 중에도 호출 가능 (읽기 스냅샷)
    |
    v
[2] 파일 수집
    -- snap DB: 필수 (없으면 에러)
    -- config.toml: {dataDir}/config.toml (선택)
    -- keystore: {dataDir}/keystore/*.json (선택, 0개 가능)
    |
    v
[3] 평문 페이로드 조립
    -- Entry 1: { name: "waiaas.db", data: <snap DB bytes> }
    -- Entry 2: { name: "config.toml", data: <config bytes> } (존재 시)
    -- Entry 3+: { name: "keystore/<wallet-id>.json", data: <keystore bytes> }
    -- 각 entry: [nameLen:2B][name:NB][dataLen:8B][data:MB]
    |
    v
[4] SHA-256 체크섬 계산
```

### 2.6 VACUUM INTO 상세

**현재 문제:**
```typescript
// 기존 BackupService -- 일관성 문제 가능
copyFileSync(dbPath, join(backupDir, 'waiaas.db'));      // DB
copyFileSync(walPath, join(backupDir, 'waiaas.db-wal')); // WAL (복사 중 변경 가능)
copyFileSync(shmPath, join(backupDir, 'waiaas.db-shm')); // SHM (복사 중 변경 가능)
```

**개선:**
```typescript
// EncryptedBackupService -- 원자적 스냅샷
const snapPath = join(tmpDir, `waiaas-snap-${uuid}.db`);
db.exec(`VACUUM INTO '${snapPath}'`);
// WAL 내용이 통합된 단일 DB 파일 생성
// 원본 DB에 영향 없음 (읽기 전용 스냅샷)
```

**VACUUM INTO 특성:**
- SQLite 3.27.0+ (better-sqlite3가 번들하는 버전에서 지원)
- 원본 DB의 읽기 잠금만 잡음 (쓰기 가능)
- WAL 내용이 통합된 자기완결적 DB 파일 생성
- 대상 파일이 이미 존재하면 에러 (충돌 방지)
- 데몬 실행 중에도 안전하게 호출 가능

### 2.7 에러 코드 추가

| 에러 코드 | HTTP | 발생 조건 |
|-----------|------|----------|
| `INVALID_BACKUP_FORMAT` | 400 | 매직 넘버 불일치, 파일이 WAIaaS 백업이 아님 |
| `UNSUPPORTED_BACKUP_VERSION` | 400 | 포맷 버전이 현재 데몬이 지원하는 범위를 초과 |
| `BACKUP_CORRUPTED` | 400 | SHA-256 체크섬 불일치 (파일 손상 또는 불완전 다운로드) |
| `BACKUP_NOT_FOUND` | 404 | 지정된 백업 파일이 존재하지 않음 |
| `DAEMON_RUNNING` | 409 | 복원 시 데몬이 실행 중 (PID 파일 확인) |
| `RESTORE_FAILED` | 500 | 복원 중 오류 (기존 데이터는 .bak에서 자동 복구) |

기존 에러 코드 재사용:
- `INVALID_MASTER_PASSWORD` -- authTag 불일치 (비밀번호 오류)

### 2.8 키스토어 포함 범위

백업에 포함되는 키스토어 파일:
- `{dataDir}/keystore/*.json` -- 모든 .json 파일
- 키스토어 파일은 이미 AES-256-GCM + Argon2id로 암호화되어 있음 (이중 암호화)
- 이중 암호화가 과도하지만, 백업의 원자성을 위해 모든 관련 파일을 포함하는 것이 올바름
- 복원 시 키스토어 디렉토리 전체를 교체하므로 부분 복원 문제 방지

### 2.9 메모리 안전성

| 단계 | 민감 데이터 | 제로화 시점 |
|------|------------|------------|
| KDF 키 유도 | 32바이트 AES 키 | 암호화/복호화 완료 직후 `key.fill(0)` |
| 평문 페이로드 | DB + keystore bytes | 암호화 완료 후 Buffer 참조 해제 (GC) |
| 복호화 결과 | 평문 페이로드 | 파일 추출 완료 후 Buffer 참조 해제 |
| 마스터 비밀번호 | 문자열 | JavaScript 문자열 불변 (Node.js 한계, 기존 패턴과 동일) |

## 성공 기준

1. 백업 아카이브 바이너리 포맷이 바이트 수준으로 사양화된다 (BKUP-01)
2. BackupService 암호화 확장이 기존 인터페이스를 유지하면서 Argon2id KDF + VACUUM INTO 원자적 스냅샷을 포함하도록 설계된다 (BKUP-02)
