모듈 경계 규칙:
- `packages/admin/src/desktop/` 내부 파일은 외부(`pages/`, `components/`, `utils/`)를 import할 수 있다
- 외부 파일은 `desktop/`를 직접 static import할 수 없다 -- 반드시 `lazy(() => import('./desktop/...'))` 패턴만 사용
- `desktop/bridge/types.ts`만 예외: 타입 정의는 static import 허용 (런타임 코드 없음, `import type` 사용)
- ESLint no-restricted-imports 규칙으로 정적 import 방지 강제

**6.6 Tree-Shaking 전략 (BLD-03):**
브라우저 번들에서 Desktop 전용 코드가 완전히 제거되는 메커니즘:

1. **Dynamic import 기반 코드 분할**: `lazy(() => import('./desktop/...'))` 패턴은 Vite/Rollup이 별도 chunk로 분리. 브라우저에서는 `isDesktop()=false`이므로 해당 chunk를 요청하지 않음.
2. **Dead code elimination**: `if (isDesktop()) { ... }` 블록은 브라우저 빌드에서 Vite define(`__DESKTOP__: false`) + terser/esbuild minifier가 제거. 단, 런타임 체크 `isDesktop()`은 minifier가 제거 불가하므로 dynamic import 패턴이 핵심.
3. **Dual build 회피**: 브라우저/Desktop 각각 빌드하지 않는다. 단일 빌드 산출물에서 런타임 분기.
4. **번들 크기 검증**: CI에서 브라우저 빌드 번들 크기를 모니터링. Desktop 의존성(@tauri-apps/*, @reown/*) 문자열이 브라우저 번들에 포함되면 CI 실패.

번들 크기 영향 분석:
- 브라우저 번들: Desktop 코드 0KB 추가 (dynamic import로 분리, 로드되지 않음)
- Desktop 번들: 기존 Admin Web UI + Desktop chunks (~200KB gzip 추가, 대부분 @reown/appkit)

기존 섹션 6.2의 "이들 Desktop 전용 의존성은 브라우저 빌드에서 tree-shake out된다" 문장에 "상세는 6.6 참조" 크로스 레퍼런스 추가.

**6.7 Vite 빌드 설정 변경 (BLD-04):**

브라우저 빌드 (기존, 변경 최소):
- `packages/admin/vite.config.ts` 기존 설정 유지
- 추가: `define: { __DESKTOP__: false }` (빌드 타임 상수, tree-shaking 보조)
- 추가: `build.rollupOptions.external` -- Desktop 전용 모듈을 external로 지정하여 빌드 에러 방지 (optional peer dep이 설치되지 않은 환경에서)

Tauri 개발 모드:
- `apps/desktop/tauri.conf.json`의 `build.devUrl` 설정: `http://localhost:5173/admin` (Vite dev server)
- Tauri CLI가 Vite dev server와 Rust 컴파일을 동시 실행
- `apps/desktop/package.json` scripts: `"dev": "tauri dev"`가 내부적으로 `beforeDevCommand`로 Admin Web UI dev server 시작

Tauri 빌드 모드:
- `build.frontendDist` 설정: Admin Web UI 빌드 산출물 경로를 URL로 지정하지 않음 (localhost에서 serve하므로)
- Sidecar가 Admin Web UI 정적 파일을 serve하므로, Tauri 앱 자체에는 프론트엔드 번들을 포함하지 않음
- `build.beforeBuildCommand`: Admin Web UI 빌드 + Sidecar SEA 바이너리 빌드를 순차 실행

**개발 워크플로우 시퀀스:**

```
[개발 모드]
1. pnpm --filter @waiaas/admin dev          # Vite dev server (port 5173)
2. pnpm --filter @waiaas/daemon dev         # Daemon dev server (port 3100)
3. cd apps/desktop && cargo tauri dev       # Tauri dev (WebView -> localhost:3100/admin)

[프로덕션 빌드]
1. pnpm --filter @waiaas/admin build        # Admin Web UI 정적 파일
2. pnpm --filter @waiaas/daemon build:sea   # Sidecar SEA 바이너리 (Admin Web UI 포함)
3. cd apps/desktop && cargo tauri build     # Tauri 앱 번들 (.dmg/.msi/.AppImage)
```

**개발 모드에서의 Admin Web UI 로드 전략:**
- 옵션 A: WebView가 Vite dev server(localhost:5173/admin)를 직접 로드 -- HMR 지원, 빠른 개발 루프
- 옵션 B: WebView가 Daemon(localhost:3100/admin)을 로드 -- 프로덕션과 동일한 경로, Vite dev 불필요
- 결정: 옵션 A (HMR 우선) -- `tauri.conf.json`의 `build.devUrl`로 Vite dev server URL 지정, 프로덕션에서만 Daemon URL 사용

**tauri.conf.json 핵심 설정 예시:**

```json
{
  "build": {
    "devUrl": "http://localhost:5173/admin",
    "beforeDevCommand": "pnpm --filter @waiaas/admin dev",
    "beforeBuildCommand": "pnpm --filter @waiaas/admin build && pnpm --filter @waiaas/daemon build:sea"
  },
  "app": {
    "windows": [{
      "url": "http://localhost:{port}/admin",
      "title": "WAIaaS Desktop",
      "width": 1200,
      "height": 800,
      "minWidth": 900,
      "minHeight": 600
    }],
    "security": {
      "csp": "..."
    }
  }
}
```

window.url의 `{port}`는 런타임에 Sidecar Manager가 할당한 포트로 대체 (Rust main.rs에서 `WebviewUrl::External` 사용).
