**Zod 스키마** (`openapi-schemas.ts`에 추가):

```typescript
// Staking position schema
export const StakingPositionSchema = z.object({
  protocol: z.enum(['lido', 'jito']),
  chain: z.enum(['ethereum', 'solana']),
  asset: z.string(),           // 'stETH' or 'JitoSOL'
  balance: z.string(),         // Token balance as string (decimal)
  balanceUsd: z.string().nullable(), // USD equivalent or null if price unavailable
  apy: z.string().nullable(),  // Current APY % as string or null
  pendingUnstake: z.object({
    amount: z.string(),
    status: z.enum(['PENDING', 'COMPLETED', 'TIMEOUT']),
    requestedAt: z.number().nullable(),
  }).nullable(),
});

export const StakingPositionsResponseSchema = z.object({
  walletId: z.string(),
  positions: z.array(StakingPositionSchema),
});
```

- OpenAPI route 정의: GET `/v1/wallets/{id}/staking`
- sessionAuth 미들웨어 적용 (기존 패턴)
- walletId 파라미터로 월렛 조회, 없으면 WALLET_NOT_FOUND 에러
- 월렛의 chain에 따라 스테이킹 포지션 결정:
  - chain='ethereum': Lido 포지션 조회 -- stETH 잔고는 adapter.getBalance()로 stETH 토큰 조회 (stethAddress 사용). APY는 v1에서 하드코딩 '~3.5%' (실제 Lido APY는 API로 조회하지만 v1은 근사치). 이미 가격 Oracle이 있으므로 USD 환산은 PriceOracleService.getPrice('stETH') 사용 (있으면).
  - chain='solana': Jito 포지션 조회 -- JitoSOL 잔고는 adapter.getBalance()로 JitoSOL mint 토큰 조회. APY는 v1에서 하드코딩 '~7.5%'.
- pendingUnstake: transactions 테이블에서 해당 walletId + bridgeStatus IN ('PENDING') + metadata에 tracker:'lido-withdrawal' 또는 'jito-epoch'가 있는 레코드 조회
- 포지션이 없으면 (balance=0, pendingUnstake=null) 빈 배열 반환
- 실제 토큰 잔고 조회가 adapter/RPC 접근 필요로 복잡할 수 있으므로, v1에서는 간단한 접근: 해당 월렛의 최근 스테이킹 트랜잭션(actions.lido_staking/jito_staking) 기록을 transactions 테이블에서 집계하여 추정 잔고 반환. `amount` 필드 합산으로 대략적 포지션 추정.
- 월렛이 ethereum이 아니면 Lido 포지션 없음, solana가 아니면 Jito 포지션 없음. 빈 배열 반환.
