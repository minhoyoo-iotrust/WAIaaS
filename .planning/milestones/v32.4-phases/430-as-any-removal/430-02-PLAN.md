    **hot-reload.ts (6건):**
    - L598, L600, L634, L639: `pool.evict('solana' as any, network as any)` / `pool.evict('ethereum' as any, network as any)` 패턴.
    - `pool.evict()`는 `(chain: ChainType, network: NetworkType)`를 받는다.
    - `'solana'`과 `'ethereum'`은 ChainType 리터럴이므로 `as any` 없이 직접 전달 가능해야 한다. 하지만 network 변수가 `string` 타입일 수 있다.
    - 해결: network 변수를 `as NetworkType`으로 단언한다. 이미 solanaNetworks Set으로 검증했으므로 안전하다. 또는 `NetworkTypeEnum.parse(network)`로 런타임 검증 후 사용.
    - 코드 흐름에서 network가 이미 알려진 네트워크 이름(solana-mainnet 등)임이 보장되므로, `as ChainType`과 `as NetworkType`으로 좁히는 것이 적절하다 (as any보다 훨씬 안전).
    - 각 evict 호출을: `await pool.evict('solana' as ChainType, network as NetworkType);` 형태로 변경.
    - import에 ChainType, NetworkType를 추가한다 (`from '@waiaas/core'`).

    **sync-pipeline.ts (1건):**
    - L52 `(ctx as any).source = 'rpc-proxy'`: Hono Context에 커스텀 속성을 추가하는 패턴.
    - 해결: Hono의 `c.set('source', 'rpc-proxy')` / `c.get('source')` 패턴을 사용하거나, Context의 Variables 제네릭에 `source` 필드를 추가한다.
    - 가장 간단한 해결: `c.set('source', 'rpc-proxy')` 사용. Hono에서 Variables 타입에 source를 선언해야 한다면, 해당 라우트의 Hono app 타입 또는 로컬 타입에 추가한다.
    - 대안: ctx가 Hono Context가 아니면, 별도 Map이나 WeakMap으로 메타데이터를 관리한다.
  </action>

    **stages.ts (6건):**
    - L996 `(ctx.request as any).type ?? 'TRANSFER'`: TransactionRequest는 discriminatedUnion 9-type으로, 모든 variant에 `.type` 필드가 있다. `ctx.request`가 `TransactionRequest` 타입이면 `.type`에 직접 접근 가능. 만약 타입이 좁혀지지 않아 `.type`을 인식하지 못하면, `'type' in ctx.request ? ctx.request.type : 'TRANSFER'` 패턴을 사용한다 (이미 비슷한 패턴이 L1543에 존재).
    - L1480 `createPublicClient({...}) as any`: viem createPublicClient의 반환 타입이 bundlerClient 생성 시 호환되지 않는 문제. `createSmartAccountBundlerClient`가 받는 client 타입과 맞추기 위해 `as PublicClient`로 좁히거나, bundlerClient 생성 함수의 타입을 확인한다.
    - viem의 제네릭이 엄격해서 exact chain type을 요구하는 경우가 있다. 이 때 `// @ts-expect-error viem chain generic mismatch` 주석으로 대체 가능하나, 먼저 `as PublicClient` 시도.
    - L1498 `publicClient as any`: 위와 같은 맥락. `as PublicClient`로 좁힌다.
    - L1506 `(bundlerClient as any).prepareUserOperation(...)`: bundlerClient의 실제 타입에 이 메서드가 있는지 확인한다. viem SmartAccountClient 또는 BundlerClient 타입을 import하여 단언. 만약 타입에 없으면 wrapper 함수를 만든다:
      ```typescript
      type BundlerOps = {
        prepareUserOperation(args: { calls: unknown[] }): Promise<UserOperationEstimate>;
        sendUserOperation(args: { calls: unknown[]; userOperation?: Partial<UserOperationEstimate> }): Promise<string>;
        waitForUserOperationReceipt(args: { hash: string; timeout?: number }): Promise<UserOperationReceipt>;
      };
      ```
      그리고 `const ops = bundlerClient as unknown as BundlerOps;`로 한 번만 단언하고 이후 `ops.prepareUserOperation(...)` 사용.
    - L1515 `(bundlerClient as any).sendUserOperation(...)`: 위 wrapper 패턴으로 통합.
    - L1569 `(bundlerClient as any).waitForUserOperationReceipt(...)`: 위 wrapper 패턴으로 통합.
