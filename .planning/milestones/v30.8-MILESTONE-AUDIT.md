## Detailed Findings

### INT-01: Notification Events Defined But Never Emitted (MEDIUM)

**Affected Requirement:** NOTIF-01
**Category:** Missing Connection

The 5 ERC-8004 notification event types are fully defined in @waiaas/core:
- `AGENT_REGISTERED` (identity category)
- `AGENT_WALLET_LINKED` (identity category)
- `AGENT_WALLET_UNLINKED` (identity category)
- `REPUTATION_FEEDBACK_RECEIVED` (identity category)
- `REPUTATION_THRESHOLD_TRIGGERED` (policy category)

**What exists:**
- Enum values in `packages/core/src/enums/notification.ts:54-58`
- Category mapping in `packages/core/src/schemas/signing-protocol.ts:238-242`
- Description mapping in `packages/core/src/schemas/signing-protocol.ts:299-303`
- i18n templates (en + ko) in `packages/core/src/i18n/{en,ko}.ts`

**What is missing:**
- No `eventBus.emit()` calls with these event types anywhere in the daemon codebase
- No post-execution hooks for ERC-8004 actions that emit these events

**Evidence:** `grep -r "emit.*AGENT_REGISTERED" packages/` returns 0 matches. Similarly for all 5 event types.

**Impact:** The notification infrastructure is complete (types, categories, descriptions, i18n templates), but users will never receive notifications for ERC-8004 events. This is a functional gap -- the plumbing exists but the faucet is not connected.

**Suggested Fix:** Add eventBus.emit() calls in the daemon's action execution post-processing, matching the pattern used for other notification events (e.g., POLICY_VIOLATION_DETECTED in the policy engine).

### INT-02: Reputation Cache Invalidation Not Wired (LOW)

**Affected Requirement:** REPU-04 (partial -- cache exists and works, invalidation path incomplete)
**Category:** Orphaned Method

**What exists:**
- `ReputationCacheService.invalidate(agentId)` at `packages/daemon/src/services/erc8004/reputation-cache-service.ts:115`
- `ReputationCacheService.invalidateAll()` at line 135
- 14 unit tests for cache behavior including invalidation

**What is missing:**
- No code path calls `invalidate()` or `invalidateAll()` after give_feedback or revoke_feedback actions complete
- The pipeline completes the on-chain transaction but does not trigger cache refresh

**Impact:** After an agent gives or revokes feedback, the cached reputation score may be stale until TTL expiration (default 300s). The stale data is bounded by the TTL, so this is a consistency delay rather than a correctness issue.

**Suggested Fix:** Add a post-execution listener in daemon.ts that detects completed give_feedback/revoke_feedback actions (by action name) and calls `reputationCacheService.invalidate(targetAgentId)`.
