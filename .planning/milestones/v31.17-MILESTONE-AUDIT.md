## Executive Summary

Milestone v31.17 successfully delivers a build-time OpenAPI spec extraction pipeline, typed API client for Admin UI, and incremental migration of ~18 pages from manual interfaces to generated types. ## Risk Assessment

**LOW RISK:**
- The 17 component files on old client still work correctly; they just don't benefit from type safety. No runtime breakage.
- The 11 remaining manual interfaces are genuinely UI-only or missing named schemas -- not integration gaps.

**MEDIUM RISK:**
- API-04 (settings schema consumption) is marked complete in REQUIREMENTS.md but is not wired. The `keyToLabel()` function in settings-helpers.ts has a hardcoded map that could drift from actual settings. The `GET /v1/admin/settings/schema` endpoint with its `deriveLabel()` function was specifically built to replace this, but the replacement was not executed.

**NO RISK:**
- All page-level API calls are type-safe via openapi-fetch.
- CI gates ensure type freshness and contract consistency on every PR.
- Backend schema changes will cause build failures (MIG-08 verified with drift test).

---

## Recommendations

1. **API-04 follow-up:** Wire `GET /v1/admin/settings/schema` into Admin UI settings display. Replace `keyToLabel()` hardcoded map with API-driven labels. This is a small task (~15 min) that completes the API-04 requirement.

2. **Component migration follow-up:** Migrate the 17 remaining component files from old `apiGet/apiPost` to typed client. Priority: `currency-select.tsx`, `spending-limit-form.tsx` (used across many pages), then Hyperliquid/Polymarket data tables.

3. **AdminStats schema:** Add a named Zod schema for `/v1/admin/stats` response so `AdminStats` interface can become a generated type alias.
