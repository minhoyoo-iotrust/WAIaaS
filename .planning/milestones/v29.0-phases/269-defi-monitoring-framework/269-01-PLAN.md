**섹션 9: 모니터링 프레임워크 — IDeFiMonitor 인터페이스**

9.1 IDeFiMonitor 인터페이스 정의:
- TypeScript 인터페이스로 정의. 메서드: `name` (readonly string), `evaluate(position: DefiPositionRow): MonitorEvaluation | null`, `getInterval(): number`, `start(): void`, `stop(): void`, `updateConfig(config: Record<string, unknown>): void`
- MonitorSeverity 타입: `'SAFE' | 'WARNING' | 'DANGER' | 'CRITICAL'` (4단계)
- MonitorEvaluation 결과 타입: `{ walletId, positionId, severity, value, threshold, message }`
- 파일 위치: `packages/core/src/interfaces/defi-monitor.types.ts`
- IChainSubscriber 6메서드 인터페이스 패턴 참조. 공통 계약으로 DeFiMonitorService가 모니터를 제네릭하게 관리 가능

9.2 MonitorSeverity 타입 상세:
- SAFE: 정상. 최대 폴링 간격. 알림 없음
- WARNING: 경고. 중간 폴링 간격. WARNING 수준 알림
- DANGER: 위험. 짧은 폴링 간격. WARNING 수준 알림 (반복 발생 시 쿨다운 적용)
- CRITICAL: 임박한 위험. 최소 폴링 간격. LIQUIDATION_IMMINENT 알림 (쿨다운 미적용)
- 각 severity에 대한 색상 코드(Admin UI 표시용): SAFE=green, WARNING=yellow, DANGER=orange, CRITICAL=red

9.3 DeFiMonitorService 오케스트레이터:
- 파일 위치: `packages/daemon/src/services/monitoring/defi-monitor-service.ts`
- 역할: 3개 모니터 인스턴스를 등록/시작/정지/설정 업데이트
- 생성자 의존성: `{ sqlite, notificationService, settingsService, config }` (BalanceMonitorService와 동일 패턴)
- `register(monitor: IDeFiMonitor)`: 모니터 등록. 시작 전 등록 필요
- `start()`: 등록된 모든 모니터 순차 시작 (fail-soft: 하나 실패해도 나머지 시작)
- `stop()`: 등록된 모든 모니터 정지 (역순)
- `updateConfig(config)`: 등록된 모든 모니터에 설정 전파
- IncomingTxMonitorService 오케스트레이터 패턴 참조 (queue + multiplexer + workers → monitors 배열)

9.4 모니터 등록 패턴:
- DeFiMonitorService 생성 시 3개 모니터를 각각 인스턴스화하여 register()
- 각 모니터는 자체 폴링 타이머를 관리 (BalanceMonitorService setInterval/clearInterval 패턴)
- 모니터 간 독립성: 한 모니터 에러가 다른 모니터에 영향 없음 (per-monitor try/catch)

**섹션 10: 3개 모니터 상세 설계**

10.1 HealthFactorMonitor (적응형 폴링):
- 파일 위치: `packages/daemon/src/services/monitoring/health-factor-monitor.ts`
- 대상: defi_positions 테이블의 category='LENDING' 포지션
- 적응형 폴링 설계:
  * 재귀 setTimeout 사용 (setInterval 금지 — 폴링 간격이 동적으로 변경되므로)
  * severity → interval 매핑: SAFE=300,000ms(5분), WARNING=60,000ms(1분), DANGER=15,000ms(15초), CRITICAL=5,000ms(5초)
  * severity 판정 기준: health factor > safe_threshold(2.0) = SAFE, > warning_threshold(1.5) = WARNING, > danger_threshold(1.2) = DANGER, 이하 = CRITICAL
  * worstSeverity 계산: 모든 LENDING 포지션 평가 후 가장 나쁜 severity를 currentSeverity로 설정
  * severity 하강(악화): 즉시 interval 변경 (다음 tick에 적용)
  * severity 상승(개선): 즉시 interval 변경 (LENDING 포지션 없으면 SAFE로 복귀, polling만 유지)
  * on-demand sync: DANGER/CRITICAL 진입 시 PositionTracker.syncCategory('LENDING') 호출하여 최신 데이터 확보
- 평가 로직 (`evaluate(position)`):
  * position.metadata에서 healthFactor 값 추출 (Phase 268 LendingMetadata.healthFactor)
  * threshold 비교하여 severity 반환
  * null 반환: category !== 'LENDING' 또는 status !== 'ACTIVE'
- 알림 발생:
  * WARNING/DANGER → LIQUIDATION_WARNING 이벤트 (notificationService.notify 직접 호출)
  * CRITICAL → LIQUIDATION_IMMINENT 이벤트 (BROADCAST_EVENTS에 포함)
  * 쿨다운: WARNING/DANGER = cooldown_hours(4h) 기반 per-wallet:position 복합 키, CRITICAL = 쿨다운 없음
- 쿨다운 맵: `Map<string, number>` (key: `${walletId}:${positionId}`, value: last alert timestamp)
- per-wallet 에러 격리: try/catch per position evaluation (BalanceMonitorService 패턴)

10.2 MaturityMonitor (1일 고정 폴링):
- 파일 위치: `packages/daemon/src/services/monitoring/maturity-monitor.ts`
- 대상: defi_positions 테이블의 category='YIELD' 포지션
- 고정 폴링: setInterval(86,400,000ms = 24시간) + 시작 시 즉시 1회 실행
- 평가 로직 (`evaluate(position)`):
  * position.metadata에서 maturity 값 추출 (Phase 268 YieldMetadata.maturity — ISO 8601 date string)
  * daysRemaining = (maturityDate - now) / MS_PER_DAY
  * daysRemaining <= 0 && status === 'ACTIVE' → severity CRITICAL (미상환), message: unredeemed position warning
  * daysRemaining <= maturity_warning_days_final(1) → severity DANGER, message: final maturity warning
  * daysRemaining <= maturity_warning_days_first(7) → severity WARNING, message: approaching maturity
  * else → severity SAFE (null 반환, 알림 없음)
  * null 반환: category !== 'YIELD' 또는 status !== 'ACTIVE'
- 알림 발생:
  * WARNING/DANGER → MATURITY_WARNING 이벤트
  * CRITICAL (미상환) → MATURITY_WARNING 이벤트 (LIQUIDATION_IMMINENT가 아님 — 만기는 자금 즉시 손실이 아님)
  * 쿨다운: 24h per-wallet:position (1일 폴링이므로 자연적 쿨다운)
- getInterval(): 고정값 86,400,000ms 반환

10.3 MarginMonitor (1분 고정 폴링):
- 파일 위치: `packages/daemon/src/services/monitoring/margin-monitor.ts`
- 대상: defi_positions 테이블의 category='PERP' 포지션
- 고정 폴링: setInterval(60,000ms = 1분) + 시작 시 즉시 1회 실행
- 평가 로직 (`evaluate(position)`):
  * position.metadata에서 margin, maintenanceMargin, liquidationPrice 추출 (Phase 268 PerpMetadata)
  * marginRatio = availableMargin / usedMargin
  * marginRatio < margin_critical_ratio(0.15) → severity CRITICAL
  * marginRatio < margin_warning_ratio(0.30) → severity WARNING
  * else → severity SAFE (null 반환)
  * 보조 판정: currentPrice와 liquidationPrice 비교
    - |currentPrice - liquidationPrice| / currentPrice < 0.05 (5% 이내) → severity CRITICAL
  * 최종 severity = max(margin ratio severity, liquidation price severity)
  * null 반환: category !== 'PERP' 또는 status !== 'ACTIVE'
- 알림 발생:
  * WARNING → MARGIN_WARNING 이벤트
  * CRITICAL → LIQUIDATION_IMMINENT 이벤트 (BROADCAST_EVENTS)
  * 쿨다운: WARNING = cooldown_hours(4h) per-wallet:position, CRITICAL = 쿨다운 없음
- getInterval(): 고정값 60,000ms 반환

10.4 설계 결정 기록:
- DEC-MON-05: "HealthFactorMonitor만 적응형 폴링, MaturityMonitor/MarginMonitor는 고정 폴링 — 폴링 빈도가 상태에 따라 극적으로 변하는 것은 health factor 뿐"
- DEC-MON-06: "MaturityMonitor CRITICAL(미상환)은 MATURITY_WARNING 이벤트 사용, LIQUIDATION_IMMINENT 아님 — 만기 미상환은 즉시 자금 손실이 아니라 기회 비용 손실"
- DEC-MON-07: "MarginMonitor는 margin ratio + liquidation price 이중 판정 — 둘 중 하나만 위험해도 경고"
- DEC-MON-08: "CRITICAL 알림은 쿨다운 미적용 — 임박한 위험은 반복 알림이 정당함"
