**Section 24.1: SignableOrder Zod 스키마**

Define SignableOrder as the Zod SSoT type for intent-based orders, following project Zod SSoT convention:

```typescript
const EIP712DomainSchema = z.object({
  name: z.string().describe('Protocol name (e.g., "Gnosis Protocol")'),
  version: z.string().describe('Protocol version (e.g., "v2")'),
  chainId: z.number().int().positive().describe('EIP-155 chain ID'),
  verifyingContract: z.string().regex(/^0x[a-fA-F0-9]{40}$/).describe('Settlement contract address'),
});

const EIP712TypeFieldSchema = z.object({
  name: z.string(),
  type: z.string(),
});

const SignableOrderSchema = z.object({
  type: z.literal('INTENT'),  // discriminatedUnion branching key
  domain: EIP712DomainSchema,
  types: z.record(z.array(EIP712TypeFieldSchema)).describe('EIP-712 type definitions keyed by struct name'),
  primaryType: z.string().describe('Top-level EIP-712 type name (e.g., "Order")'),
  message: z.record(z.unknown()).describe('The actual order data matching primaryType structure'),
  intentMetadata: z.object({
    protocol: z.string().describe('Protocol identifier: "cow_protocol", "1inch_fusion", "uniswapx"'),
    orderApiUrl: z.string().url().describe('Off-chain API endpoint for order submission'),
    statusApiUrl: z.string().url().describe('Off-chain API endpoint for status polling'),
    sellToken: z.string().describe('Sell token address (0x-prefixed)'),
    buyToken: z.string().describe('Buy token address (0x-prefixed)'),
    sellAmount: z.string().describe('Sell amount in token base units (string for bigint)'),
    buyAmount: z.string().describe('Minimum buy amount in token base units (string for bigint)'),
    validTo: z.number().int().positive().describe('Order deadline as Unix timestamp (seconds)'),
    receiver: z.string().optional().describe('Receiving address (defaults to wallet address if omitted)'),
    appData: z.string().optional().describe('Protocol-specific application data (e.g., CoW appData hash)'),
  }),
});
```

Key design points:
- `type: z.literal('INTENT')` enables discriminatedUnion branching against ContractCallRequest (which has `type: 'CONTRACT_CALL'` etc.)
- `domain`, `types`, `primaryType`, `message` map directly to viem's signTypedData parameters
- `intentMetadata` carries protocol-specific information needed for API submission and status tracking
- Amount fields use string for bigint JSON serialization (consistent with DEC-LEND-06)
- The schema is protocol-agnostic: CoW Protocol, 1inch Fusion, UniswapX all fit this structure

Include derivation chain: `SignableOrderSchema (Zod) -> SignableOrder (TypeScript type) -> OpenAPI component schema`.

**Section 24.3: ActionProviderRegistry 확장 설계**

Document the current state:
- `ActionProviderRegistry.executeResolve()` returns `Promise<ContractCallRequest[]>`
- All call sites in `actions.ts` assume ContractCallRequest[] and feed into 6-stage pipeline
- Intent orders cannot enter the 6-stage pipeline (no calldata, no gas estimation, no on-chain submission)

Design two options and recommend one:

**Option A (RECOMMENDED): Union return type extension**
```typescript
// Current
executeResolve(action: string, params: Record<string, unknown>, context: ActionContext): Promise<ContractCallRequest[]>;

// Extended
type ResolveResult = ContractCallRequest | SignableOrder;
executeResolve(action: string, params: Record<string, unknown>, context: ActionContext): Promise<ResolveResult[]>;
```

Call site branching in actions.ts:
```typescript
const results = await registry.executeResolve(action, params, context);
for (const result of results) {
  if (result.type === 'INTENT') {
    // Intent pipeline: validate -> sign typed data -> API submit -> poll
    await handleIntentOrder(result as SignableOrder, walletId, context);
  } else {
    // Existing 6-stage pipeline (unchanged)
    await handleContractCall(result as ContractCallRequest, walletId, context);
  }
}
```

Pros: Single entry point, type-safe branching via discriminatedUnion, existing providers unchanged (they return ContractCallRequest which is a valid ResolveResult).
Cons: Return type widening requires all call sites to handle both types.

**Option B: Separate executeIntentResolve method**
```typescript
// Existing (unchanged)
executeResolve(...): Promise<ContractCallRequest[]>;
// New
executeIntentResolve(...): Promise<SignableOrder[]>;
```

Pros: Zero impact on existing code. Cons: Provider must be registered for intent separately, duplicated routing logic.

**Recommendation: Option A** — Union return type extension. Justification:
1. Type-safe: discriminatedUnion on `type` field is the project's established pattern
2. Single registry: no separate intent registration, provider declares capabilities via return type
3. Forward-compatible: new result types (e.g., permit2 signatures) just add to the union
4. Minimal impact: existing providers already return ContractCallRequest, which satisfies the union

**TRANSACTION_TYPES extension:**
- Add 'INTENT' as 8th value to TRANSACTION_TYPES array: `['TRANSFER', 'TOKEN_TRANSFER', 'CONTRACT_CALL', 'APPROVE', 'BATCH', 'SIGN', 'X402_PAYMENT', 'INTENT']`
- TransactionRequestSchema (5-type discriminatedUnion for external API) is NOT modified — intent orders enter via ActionProviderRegistry, not via POST /v1/wallets/:id/transactions
