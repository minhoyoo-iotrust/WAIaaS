**섹션 7.1: GET /v1/wallets/:id/positions 엔드포인트**
- HTTP 메서드: GET
- 경로: `/v1/wallets/:walletId/positions`
- 인증: sessionAuth (기존 지갑 API와 동일)
- 경로 파라미터: walletId (UUID)
- 쿼리 파라미터:
  - `category` (optional): LENDING | YIELD | PERP | STAKING -- 카테고리 필터
  - `provider` (optional): aave_v3 | kamino | lido | jito | pendle | drift -- 프로바이더 필터
  - `status` (optional, default: ACTIVE): ACTIVE | CLOSED | LIQUIDATED -- 상태 필터
- 성공 응답 (200):
  ```
  {
    walletId: string,
    positions: Position[],  // discriminatedUnion on 'category'
    totalValueUsd: number | null,
    lastUpdatedAt: number   // 가장 최근 last_synced_at
  }
  ```

**섹션 7.2: PositionsResponseSchema (Zod)**
- Plan 01에서 정의한 BasePositionSchema + 4개 카테고리 확장을 API 응답 스키마로 활용
- 각 카테고리별 응답 스키마에 .openapi() 태그 추가:
  - LendingPositionResponseSchema.openapi('LendingPosition')
  - YieldPositionResponseSchema.openapi('YieldPosition')
  - PerpPositionResponseSchema.openapi('PerpPosition')
  - StakingPositionResponseSchema.openapi('StakingPosition')
- PositionsResponseSchema: { walletId, positions (discriminatedUnion), totalValueUsd, lastUpdatedAt }.openapi('PositionsResponse')
- PositionQuerySchema: 쿼리 파라미터용 Zod 스키마 (z.object({ category, provider, status }).partial())

**섹션 7.3: 기존 엔드포인트 호환성**
- 기존 GET /v1/wallet/staking은 deprecated 처리하되 계속 동작
- deprecated 헤더: Deprecation: true, Sunset: TBD (구현 마일스톤에서 결정)
- /wallet/staking -> /wallets/:id/positions?category=STAKING 매핑 관계 문서화
- 전환 계획: (1) 구현 마일스톤에서 PositionTracker가 STAKING 포지션을 defi_positions 테이블에 동기화, (2) /wallet/staking은 defi_positions 테이블에서 조회하도록 내부 전환, (3) SDK/MCP는 새 엔드포인트로 전환

**섹션 7.4: OpenAPIHono 라우트 정의**
- createRoute 패턴으로 라우트 정의 코드 작성 (268-RESEARCH.md의 @hono/zod-openapi 패턴 참조)
- 태그: 'Positions'
- 에러 응답: 401 (Unauthorized), 404 (Wallet not found), 500 (Internal)

**설계 결정 기록:**
- totalValueUsd를 서버에서 계산하는 이유 (AI 에이전트가 별도 집계 불필요)
- 쿼리 파라미터를 URL query로 전달하는 이유 (GET 요청, 캐시 가능)
- status 기본값 ACTIVE인 이유 (대부분의 조회가 활성 포지션)

**섹션 8.1: 포지션 목록 레이아웃**
- 페이지 위치: Admin UI의 Wallet Detail 페이지 내 "Positions" 탭 (기존 Transactions/Staking 탭과 나란히)
- ASCII 와이어프레임으로 전체 레이아웃 작성:
  ```
  ┌─────────────────────────────────────────────────────┐
  │ Portfolio Summary                                    │
  │ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ │
  │ │ Total USD │ │ Lending  │ │ Yield    │ │ Perp PnL │ │
  │ │ $XX,XXX  │ │ $X,XXX   │ │ $X,XXX   │ │ +$XXX    │ │
  │ └──────────┘ └──────────┘ └──────────┘ └──────────┘ │
  ├─────────────────────────────────────────────────────┤
  │ [All] [Lending] [Yield] [Perp] [Staking] ← 탭 필터 │
  ├─────────────────────────────────────────────────────┤
  │ Position Cards (카테고리별 카드)                     │
  └─────────────────────────────────────────────────────┘
  ```
- 상단 StatCard 4개: Total Portfolio USD, Lending TVL, Yield TVL, Perp Unrealized PnL
- 탭 필터: 카테고리별 포지션 카드 그룹 전환

**섹션 8.2: 카테고리별 카드 구성**
- 각 카테고리의 포지션 카드 레이아웃을 ASCII 와이어프레임으로 작성
- **Lending 카드:**
  ```
  ┌──────────────────────────────────────┐
  │ [Aave V3] SUPPLY ETH               │
  │ Amount: 2.5 ETH ($7,500)           │
  │ APY: 3.2%  │  Health Factor: 1.85  │
  │ Status: ACTIVE  │  Synced: 2m ago  │
  └──────────────────────────────────────┘
  ```
- **Yield 카드:** provider, token_type(PT/YT/LP), amount, maturity date, APY
- **Perp 카드:** provider, direction(LONG/SHORT), leverage, entry price, unrealized PnL, liquidation price
- **Staking 카드:** provider, amount, APY, pending unstake info

**섹션 8.3: USD 환산 + APY + 헬스 팩터 표시**
- USD 환산: fetchDisplayCurrency + formatWithDisplay 기존 패턴 활용
- APY: 소수점 2자리 퍼센트 표시 (예: 3.21%)
- 헬스 팩터: 색상 코딩 (> 2.0: green, 1.5-2.0: yellow, 1.2-1.5: orange, < 1.2: red)
- Perp PnL: 양수는 green, 음수는 red, 소수점 2자리 USD 표시
- 마진 비율: 진행 바(progress bar)로 시각화 (available / total)

**섹션 8.4: 상태 관리 패턴**
- Preact signals 기반 상태:
  - `positions = signal<Position[]>([])`: 전체 포지션 목록
  - `selectedCategory = signal<PositionCategory | 'ALL'>('ALL')`: 선택된 탭 필터
  - `loading = signal<boolean>(true)`: 로딩 상태
- 데이터 페칭: useEffect에서 GET /v1/wallets/:id/positions 호출
- 실시간 갱신 없음 (수동 새로고침 버튼 제공 -- PositionTracker가 서버에서 동기화)

**설계 결정 기록:**
- Wallet Detail 내 탭으로 배치하는 이유 (별도 페이지 vs 탭 -- 탭이 컨텍스트 유지에 유리)
- 실시간 WebSocket 갱신을 제외하는 이유 (PositionTracker가 서버 측 동기화, Admin은 스냅샷 조회)
- 카드 레이아웃을 테이블 대신 선택한 이유 (카테고리별 필드가 달라 테이블 컬럼 통일 불가)
