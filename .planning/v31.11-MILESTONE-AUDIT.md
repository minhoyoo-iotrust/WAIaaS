## 4. E2E Design Flow Verification

### Flow 1: Off-chain Action Execution (signedData)

```
ActionProvider.resolve() -> SignedDataAction
  -> isApiDirectResult() check (NO)
  -> normalizeResolvedActions() (kind already present)
  -> Policy evaluation (VENUE_WHITELIST + ACTION_CATEGORY_LIMIT)
  -> CredentialVault.get(credentialRef, walletId) [if credentialRef]
  -> assembleSigningParams() (scheme-specific)
  -> SignerCapabilityRegistry.resolve(action) -> ISignerCapability
  -> capability.canSign(params) -> capability.sign(params)
  -> recordOffchainAction() -> DB (action_kind='signedData', bridge_status='PENDING'|NULL)
  -> credential memory clear
  -> ActionExecutionResult { kind: 'signedData', ... }
```

**Completeness:** All 5 stages of signedData pipeline are fully designed across Phase 380 (types), 381 (credential resolve), 382 (signing), 383 (pipeline + DB), 384 (policy + tracking). **COMPLETE**

### Flow 2: Backward-Compatible contractCall

```
Existing ActionProvider.resolve() -> ContractCallRequest (no kind)
  -> isApiDirectResult() check (NO)
  -> normalizeResolvedActions() -> adds kind: 'contractCall'
  -> routeByKind('contractCall')
  -> existing 6-stage pipeline (UNCHANGED)
  -> DB record (action_kind='contractCall' via DEFAULT)
```

**Completeness:** Normalization strategy documented in Phase 380. Pipeline routing in Phase 383 explicitly strips kind before passing to 6-stage pipeline. **COMPLETE**

### Flow 3: Credential Lifecycle

```
Admin -> POST /v1/wallets/:id/credentials (masterAuth)
  -> UUID v7 + HKDF subkey + AES-256-GCM encrypt -> DB
  -> credential available for ActionProvider resolve() via credentialRef
Pipeline -> CredentialVault.get(ref, walletId) -> decrypt + expire check
  -> inject into SigningParams -> sign() -> memory clear
Admin -> PUT /v1/wallets/:id/credentials/:ref/rotate
  -> new IV + re-encrypt -> DB UPDATE
Master Password change -> re-encrypt all wallet_credentials
Backup -> include encrypted wallet_credentials
```

**Completeness:** Full lifecycle (create/use/rotate/expire/delete/re-encrypt/backup) documented in Phase 381. **COMPLETE**

### Flow 4: Async Tracking for Off-chain Orders

```
ActionProvider.resolve() -> SignedDataAction { tracking: { trackerName, metadata } }
  -> Pipeline records DB: bridge_status='PENDING', bridge_metadata={trackerName, venue, ...}
  -> AsyncPollingService picks up (PENDING or PARTIALLY_FILLED)
  -> resolveTrackerName() -> trackerName (priority) or tracker (fallback)
  -> tracker.checkStatus() -> PARTIALLY_FILLED/FILLED/CANCELED/SETTLED/EXPIRED
  -> bridge_status + bridge_metadata update
  -> Notification events (external_action_* namespace)
```

**Completeness:** Phase 384 designs the full tracking flow, builds on Phase 383 DB schema, and integrates with existing AsyncPollingService. **COMPLETE**
